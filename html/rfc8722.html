Defining the Role and Function of IETF Protocol Parameter Registry Operators

IETFプロトコルパラメータレジストリオペレータの役割と機能の定義

Abstract

概要

Many Internet Engineering Task Force (IETF) protocols make use of commonly defined values that are passed in messages or packets. To ensure consistent interpretation of these values between independent implementations, there is a need to ensure that the values and associated semantic intent are uniquely defined. The IETF uses registry functions to record assigned protocol parameter values and their associated semantic intentions. For each IETF protocol parameter, it is current practice for the IETF to delegate the role of Protocol Parameter Registry Operator to a nominated entity. This document provides a description of, and the requirements for, these delegated functions. This document obsoletes RFC 6220 to replace all references to the IETF Administrative Support Activity (IASA) and related structures with those defined by the IASA 2.0 Model.

多くのインターネット技術特別調査委員会（IETF）プロトコルは、メッセージまたはパケットで渡される一般的に定義された値を利用します。独立した実装間でこれらの値の一貫した解釈を保証するために、値と関連する意味的意図が一意に定義されることを保証する必要があります。 IETFは、レジストリ関数を使用して、割り当てられたプロトコルパラメータ値とそれらに関連付けられた意味的意図を記録します。各IETFプロトコルパラメータについて、IETFは現在、プロトコルパラメータレジストリオペレータの役割を指定されたエンティティに委任することが慣例です。このドキュメントでは、これらの委任された機能の説明と要件について説明します。このドキュメントはRFC 6220を廃止し、IETF管理サポートアクティビティ（IASA）および関連する構造へのすべての参照をIASA 2.0モデルで定義されたものに置き換えます。

1. Overview
1. 概観

Many IETF protocols make use of commonly defined values that are passed within messages or packets. To ensure consistent interpretation of these values between independent implementations, there is a need to ensure that the values and associated semantic intent are uniquely defined. The IETF uses registries to record each of the possible values of a protocol parameter and their associated semantic intent. These registries, their registration policy, and the layout of their content are defined in the so-called "IANA Considerations" sections of IETF documents.

多くのIETFプロトコルは、メッセージまたはパケット内で渡される一般的に定義された値を利用します。独立した実装間でこれらの値の一貫した解釈を保証するために、値と関連する意味的意図が一意に定義されることを保証する必要があります。 IETFは、レジストリーを使用して、プロトコル・パラメーターの可能な値のそれぞれとそれらに関連付けられたセマンティック・インテントを記録します。これらのレジストリ、その登録ポリシー、およびコンテンツのレイアウトは、IETF文書のいわゆる「IANAに関する考慮事項」セクションで定義されています。

The organizational separation between the IETF and its Protocol Parameter Registry Operators parallels ones that are fairly common among standards development organizations (SDOs) although less common among technology consortia and similar bodies. These functions have been separated into different organizations for several reasons. They include dealing with administrative issues, addressing concerns about maintaining an adequate distance between basic policy and specific allocations, and avoiding any potential conflicts of interest that might arise from commercial or organizational relationships. For example, most ISO and ISO/IEC JTC1 standards that require registration activities specify a Registration Authority (RA) or Maintenance Agency (MA) that, in turn, control the actual registration decisions. The databases of what is registered for each standard may then be maintained by a secretariat or database function associated with the RA or MA or, less frequently, by the secretariat of the body that created and maintains the standard itself.

IETFとそのプロトコルパラメーターレジストリオペレーターの組織的な分離は、標準化組織（SDO）の間ではかなり一般的ですが、テクノロジーコンソーシアムや同様の組織の間ではあまり一般的ではありません。これらの機能は、いくつかの理由で異なる組織に分離されています。これには、管理上の問題への対処、基本ポリシーと特定の割り当ての間の適切な距離の維持に関する懸念への対処、および商業的または組織的な関係から生じる可能性のある潜在的な利益相反の回避が含まれます。たとえば、登録アクティビティを必要とするほとんどのISOおよびISO / IEC JTC1標準では、登録機関（RA）または保守機関（MA）が指定されており、これらの登録機関が実際の登録決定を管理しています。各規格に登録されているデータベースは、RAまたはMAに関連付けられた事務局またはデータベース機能によって、またはそれほど頻繁ではありませんが、規格自体を作成および維持している団体の事務局によって維持されます。

This structural separation of roles exists within several places in the IETF framework (e.g., the RFC Editor function). The Internet Architecture Board (IAB), on behalf of the IETF, has the responsibility to define and manage the relationship with the Protocol Parameter Registry Operator role. This responsibility includes the selection and management of the Protocol Parameter Registry Operator, as well as management of the parameter registration process and the guidelines for parameter allocation.

この役割の構造的な分離は、IETFフレームワークのいくつかの場所に存在します（RFCエディター機能など）。 IETFに代わってインターネットアーキテクチャボード（IAB）は、プロトコルパラメーターレジストリオペレーターの役割との関係を定義および管理する責任があります。この責任には、プロトコルパラメータレジストリオペレータの選択と管理、およびパラメータ登録プロセスの管理とパラメータ割り当てのガイドラインが含まれます。

As with other SDOs, although it may delegate authority for some specific decisions, the IETF asserts authority and responsibility for the management of all of its protocol parameters and their registries, even while it generally remains isolated from the selection of particular values once a registration is approved. This document describes the function of these registries as they apply to individual protocol parameters defined by the IETF Internet Standards Process (see RFC 6410 [BCP9]) to allow for an orderly implementation by the IETF Administration Limited Liability Company (IETF LLC), and others as needed, under guidance from the IAB. This document obsoletes RFC 6220 to replace all references to the IASA and related structures with those defined by the IASA 2.0 Model [RFC8711].

他のSDOと同様に、特定の決定に対する権限を委任する場合がありますが、IETFは、登録が完了すると特定の値の選択から一般的に分離されたままであっても、そのすべてのプロトコルパラメータとそのレジストリの管理に対する権限と責任を主張します承認されました。このドキュメントでは、IETF Internet Standards Process（RFC 6410 [BCP9]を参照）によって定義された個々のプロトコルパラメータに適用され、IETF Administration Limited Liability Company（IETF LLC）などによる秩序だった実装を可能にするこれらのレジストリの機能について説明しますIABの指導の下、必要に応じて。このドキュメントはRFC 6220を廃止し、IASAおよび関連構造へのすべての参照をIASA 2.0モデル[RFC8711]で定義されたものに置き換えます。

Below we provide a description of the requirements for these delegated functions, which the IETF traditionally refers to as the Internet Assigned Numbers Authority (IANA) function.

以下では、これらの委任された機能の要件について説明します。これは、IETFが伝統的にInternet Assigned Numbers Authority（IANA）機能と呼んでいる機能です。

2. Roles and Responsibilities Concerning IETF Protocol Parameter Registries

2. IETFプロトコルパラメータレジストリに関する役割と責任

The IETF's longstanding practice is to outsource the management and implementation of some important functions (e.g., [RFC8728]). The protocol parameter registry function falls into this category of outsourced functions, and what follows here is the description of the roles and responsibilities with respect to the registration of IETF protocol parameters.

IETFの長年の慣行は、いくつかの重要な機能（[RFC8728]など）の管理と実装を外部委託することです。プロトコルパラメータレジストリ機能は、この外部委託機能のカテゴリに分類されます。次に、IETFプロトコルパラメータの登録に関する役割と責任について説明します。

Specifically, this document describes the operation and role of a delegated IETF Protocol Parameter Registry Operator, to be selected and administered by the IETF Administrative Support Activity (IASA) [RFC8711]. While there is generally a single Protocol Parameter Registry Operator, additional Operators may be selected to implement specific registries, and that has been done occasionally. Having a single Protocol Parameter Registry Operator facilitates coordination among registries, even those that are not obviously related, and also makes it easier to have consistency of formats and registry structure, which aids users of the registries and assists with quality control.

具体的には、このドキュメントでは、IETF管理サポートアクティビティ（IASA）[RFC8711]によって選択および管理される、委任されたIETFプロトコルパラメータレジストリオペレータの操作と役割について説明します。通常、単一のプロトコルパラメータレジストリオペレータが存在しますが、特定のレジストリを実装するために追加のオペレータが選択される場合があり、それは時々行われています。単一のプロトコルパラメータレジストリオペレータを使用すると、明らかに関連していないものでも、レジストリ間の調整が容易になり、フォーマットとレジストリ構造の一貫性を維持しやすくなり、レジストリのユーザーを支援し、品質管理を支援します。

Many protocols make use of identifiers consisting of constants and other well-known values. Even after a protocol has been defined and deployment has begun, new values may need to be assigned (e.g., for a new option type in DHCP, or a new encryption or authentication algorithm for IPsec). To ensure that such quantities have consistent values and interpretations in different implementations, their assignment must be administered by a central authority. For IETF protocols, that role is provided by a delegated Protocol Parameter Registry Operator. For any particular protocol parameter there is a single delegated Registry Operator.

多くのプロトコルは、定数やその他の既知の値で構成される識別子を利用しています。プロトコルが定義され、展開が開始された後でも、新しい値を割り当てる必要がある場合があります（たとえば、DHCPの新しいオプションタイプ、またはIPsecの新しい暗号化または認証アルゴリズム）。そのような数量が異なる実装で一貫した値と解釈を持つことを保証するために、それらの割り当ては中央当局によって管理されなければなりません。 IETFプロトコルの場合、その役割は委任されたプロトコルパラメータレジストリオペレータによって提供されます。特定のプロトコルパラメータには、委任された単一のレジストリオペレータがあります。

2.1. Protocol Parameter Registry Operator Role
2.1. プロトコルパラメータレジストリオペレータの役割

The IETF Protocol Parameter Registry function is undertaken under the auspices of the Internet Architecture Board.

IETFプロトコルパラメータレジストリ機能は、インターネットアーキテクチャボードの後援の下で行われます。

The roles of the Protocol Parameter Registry Operator (Registry Operator) are as follows:

プロトコルパラメータレジストリオペレータ（レジストリオペレータ）の役割は次のとおりです。

* Review and Advise

* レビューとアドバイス

- A Registry Operator may be requested to review Internet-Drafts that are being considered by the Internet Engineering Steering Group (IESG), with the objective of offering advice to the IESG regarding the contents of the "IANA Considerations" section, whether such a section, when required, is clear in terms of direction to the Registry Operator, and whether the section is consistent with the current published Registry Operator guidelines.

- レジストリオペレーターは、「IANAの考慮事項」セクションの内容についてIESGにアドバイスを提供することを目的として、インターネットエンジニアリングステアリンググループ（IESG）によって検討されているインターネットドラフトを確認するように要求される場合があります。必要に応じて、レジストリオペレーターへの指示、およびセクションが現在公開されているレジストリオペレーターのガイドラインと一致しているかどうかを明確にします。

* Registry

* レジストリ

- To operate a registry of protocol parameter assignments.

- プロトコルパラメータ割り当てのレジストリを操作します。

- The delegated Registry Operator registers values for Internet protocol parameters only as directed by the criteria and procedures specified in RFCs, including Standards Track documents [BCP9], Best Current Practice documents, and other RFCs that require protocol parameter assignment.

- 委任されたレジストリオペレーターは、Standards Trackドキュメント[BCP9]、Best Current Practiceドキュメント、およびプロトコルパラメーターの割り当てを必要とするその他のRFCを含む、RFCで指定された基準と手順によって指示された場合のみ、インターネットプロトコルパラメーターの値を登録します。

If values for Internet protocol parameters were not specified, or in case of ambiguity, the Registry Operator will continue to assign and register only those protocol parameters that have already been delegated to the Registry Operator, following past and current practice for such assignments, unless otherwise directed in terms of operating practice by the IESG. In the case of ambiguity, the Registry Operator is expected to identify the ambiguity to the IAB or IESG as appropriate and either suggest better text or ask the appropriate parties for clarification.

インターネットプロトコルパラメーターの値が指定されていない場合、またはあいまいな場合は、レジストリオペレーターは、他の場合を除き、そのような割り当ての過去および現在の慣行に従って、レジストリオペレーターに既に委任されているプロトコルパラメーターのみを割り当てて登録し続けます。 IESGによる運用慣行の指示。あいまいな場合、レジストリオペレーターはIABまたはIESGのあいまいさを適切に識別し、より適切なテキストを提案するか、適切な関係者に説明を求めることが求められます。

- For each protocol parameter, the associated registry includes:

- プロトコルパラメータごとに、関連するレジストリには次のものが含まれます。

o a reference to the RFC document that describes the parameter and the associated "IANA Considerations" concerning the parameter, and

o パラメータを説明するRFCドキュメントへの参照と、パラメータに関する関連する「IANAの考慮事項」、および

o for each registration of a protocol parameter value, the source of the registration and the date of the registration, if the date of registration is known, and

o プロトコルパラメータ値の登録ごとに、登録のソースと登録の日付（登録の日付がわかっている場合）、および

o any other information specified as being included in the registration data in the RFC document that describes the parameter.

o パラメータを説明するRFC文書の登録データに含まれるものとして指定されているその他の情報。

o If in doubt or in case of a technical dispute, the Registry Operator will seek and follow technical guidance exclusively from the IESG. Where appropriate, the IESG will appoint an expert to advise the Registry Operator.

o 疑わしい場合、または技術的な論争が発生した場合、レジストリオペレーターはIESGからのみ技術ガイダンスを求め、それに従います。 IESGは必要に応じて、レジストリオペレーターに助言する専門家を任命します。

- The Registry Operator will work with the IETF to develop any missing criteria and procedures over time, which the Registry Operator will adopt when so instructed by the IESG.

- レジストリオペレーターは、IETFと協力して不足している基準と手順を開発し、それをIESGから指示されたときにレジストリオペレーターが採用します。

- Unless special circumstances apply to subsets of the data and specific rules are established by IETF consensus, each protocol parameter registry operates as a public registry, and the contents of the registry are openly available to the public, on-line and free of charge.

- データのサブセットに特別な状況が適用され、IETFの合意により特定のルールが確立されない限り、各プロトコルパラメータレジストリはパブリックレジストリとして動作し、レジストリのコンテンツはオンラインで無料で公開されています。

- The Registry Operator assigns protocol parameter values in accordance with the policy associated with the protocol parameter, such as "First Come First Served" or "Expert Review" [RFC8126].

- レジストリオペレーターは、 "先着順"や "エキスパートレビュー" [RFC8126]など、プロトコルパラメーターに関連付けられたポリシーに従ってプロトコルパラメーター値を割り当てます。

* Mailing Lists

* メーリングリスト

- The Registry Operator maintains public mailing lists as specified in IANA Considerations [RFC8126]. Such lists are designated for the purpose of review of assignment proposals in conjunction with a designated expert review function. In addition, each Registry Operator should maintain a mailing list that enables the registry staff of the Registry Operator to be contacted by email.

- レジストリオペレーターは、IANAの考慮事項[RFC8126]で指定されている公開メーリングリストを維持しています。このようなリストは、指定された専門家レビュー機能と連携して、割り当て提案のレビューを目的として指定されます。さらに、各レジストリオペレーターは、レジストリオペレーターのレジストリスタッフに電子メールで連絡できるようにするメーリングリストを維持する必要があります。

* Liaison Activity

* リエゾン活動

- The Registry Operator will nominate a liaison point of contact. The Registry Operator, through this liaison, may be requested to provide advice to the IESG on IETF protocol parameters as well as the "IANA Considerations" section of each Internet-Draft that is being reviewed for publication as an RFC. Where appropriate the IESG will appoint an expert to advise the Registry Operator.

- レジストリオペレーターは連絡窓口を指名します。レジストリオペレーターは、この連絡窓口を通じて、IETFプロトコルパラメーターに関するIESG、およびRFCとして公開するためにレビューされている各インターネットドラフトの「IANAに関する考慮事項」セクションに関するアドバイスを提供するように要求される場合があります。適切な場合、IESGは、レジストリオペレーターに助言する専門家を任命します。

* Reporting

* 報告

- The Registry Operator will submit periodic reports to the IAB concerning the operational performance of the registry function. As an example of the requirements for such reports, the reader is referred to a supplement [MoU_SUPP2019] to the "Memorandum of Understanding Concerning the Technical Work of the Internet Assigned Numbers Authority" [RFC2860] that provides service level agreement (SLA) guidelines under which ICANN, the current protocol parameter registry, must operate.

- レジストリオペレーターは、レジストリ機能の運用パフォーマンスに関する定期レポートをIABに提出します。そのようなレポートの要件の例として、読者は以下のサービスレベル契約（SLA）ガイドラインを提供する「インターネット割り当て番号機関の技術的作業に関する覚書」[RFC2860]の補足[MoU_SUPP2019]を参照されます。現在のプロトコルパラメータレジストリであるICANNが動作する必要があります。

- At the request of the chair of the IETF or IAB, or the IETF Executive Director [RFC8711], the Registry Operator will undertake periodic reports to IETF Plenary meetings or elsewhere as directed, concerning the status of the registry function.

- IETFまたはIABの議長、またはIETFエグゼクティブディレクター[RFC8711]の要請に応じて、レジストリオペレーターは、IETF総会またはその他の指示に従って、レジストリ機能のステータスについて定期的に報告を行います。

- The Registry Operator will publish an annual report describing the status of the function and a summary of performance indicators.

- レジストリオペレーターは、機能のステータスとパフォーマンスインジケーターの概要を説明する年次レポートを発行します。

* Intellectual Property Rights and the Registry Operator

* 知的財産権とレジストリオペレーター

Unless special circumstances apply (see above):

特別な状況が適用されない限り（上記を参照）：

- All assigned values are to be published and made available free of any charges.

- 割り当てられた値はすべて公開され、無料で利用できるようになります。

- The assignment values may be redistributed without modification.

- 割り当て値は変更せずに再配布できます。

In any case,

とにかく、

- any intellectual property rights of the IETF protocol parameter assignment information, including the IETF protocol parameter registry and its contents, are to be held by the IETF Trust [RFC8711] [RFC8714].

- IETFプロトコルパラメータレジストリとその内容を含む、IETFプロトコルパラメータ割り当て情報の知的財産権は、IETFトラスト[RFC8711] [RFC8714]が保有します。

2.2. IAB Role
2.2. IABの役割

An Operator of an IETF protocol parameter registry undertakes the role as a delegated function under the authority of the IAB.

IETFプロトコルパラメータレジストリのオペレータは、IABの権限の下で委任された機能としての役割を引き受けます。

The IAB has the responsibility to review the current description of the registry function from time to time and direct the Registry Operator to adopt amendments relating to its role and mode of operation according to the best interests of the IETF and the Internet community in general.

IABは、レジストリ機能の現在の説明を時々見直し、レジストリオペレーターに、IETFとインターネットコミュニティ一般の最善の利益に従って、その役割と動作モードに関連する修正を採用するよう指示する責任があります。

The IAB has the responsibility to appoint an organization to undertake the delegated functions of the Registry Operator for each IETF protocol parameter. Specifically, the IAB defines the role and requirements for the desired functions. The IETF LLC is responsible for identifying a potential vendor, and once under agreement, managing the various aspects of the relationships with that vendor. To be clear, the IAB is in the deciding role (e.g., for appointment and termination), but must work in close consultation with the IETF LLC.

IABは、IETFプロトコルパラメータごとにレジストリオペレータの委任された機能を実行する組織を指名する責任があります。具体的には、IABは目的の機能の役割と要件を定義します。 IETF LLCは、潜在的なベンダーを特定し、いったん合意すると、そのベンダーとの関係のさまざまな側面を管理する責任があります。明確にするために、IABは決定的な役割を担っています（たとえば、任命および解任のため）が、IETF LLCと綿密に協議して作業する必要があります。

The IAB has the responsibility to determine the terms and conditions of this delegated role. Such terms and conditions should ensure that the registry operates in a manner that is fully conformant to the functions described in this document. In addition, such terms and conditions must not restrict the rights and interests of the IETF with respect to the registry contents and maintenance.

IABには、この委任された役割の条件を決定する責任があります。このような条件により、レジストリがこのドキュメントで説明されている機能に完全に準拠した方法で動作することが保証されます。さらに、そのような契約条件は、レジストリの内容とメンテナンスに関するIETFの権利と利益を制限してはなりません。

2.3. IESG Role
2.3. IESGの役割

The IESG is responsible for the technical direction regarding entries into IETF protocol parameter registries and maintaining the policies by which such technical directions are given. Technical direction itself is provided through the adoption of directives within the "IANA Considerations" section of IETF Stream RFCs or through stand-alone "IANA Considerations" RFCs.

IESGは、IETFプロトコルパラメータレジストリへのエントリに関する技術的な指示と、そのような技術的な指示を与えるためのポリシーの維持を担当します。技術的な方向性自体は、IETFストリームRFCの「IANAに関する考慮事項」セクション内のディレクティブの採用、またはスタンドアロンの「IANAに関する考慮事項」RFCを通じて提供されます。

The IESG shall verify that Internet-Drafts that are offered for publication as IETF Stream RFCs [RFC8729] include "IANA Considerations" sections when needed, and that "IANA Considerations" sections conform to the current published guidelines.

IESGは、IETFストリームRFC [RFC8729]として公開するために提供されているインターネットドラフトが、必要に応じて「IANAに関する考慮事項」セクションを含み、「IANAに関する考慮事項」セクションが現在公開されているガイドラインに準拠していることを確認します。

Since technical assessment is not generally a responsibility of the Registry Operator, as part of providing the technical direction the IESG is responsible for identifying the technical experts that are required to, where appropriate, review registration requests or resolve open technical questions that relate to the registration of parameters.

通常、技術評価はレジストリオペレーターの責任ではないため、技術的指示を提供する一環として、IESGは、必要に応じて、登録要求を確認したり、登録に関連する未解決の技術的な質問を解決したりするために必要な技術専門家を特定する責任があります。パラメータの。

At its discretion, the IESG will organize the liaison activities with the Registry Operator's liaison point of contact so as to facilitate clear communications and effective operation of the registry function.

IESGは、その裁量により、レジストリ機能の明確なコミュニケーションと効果的な運用を促進するために、レジストリオペレーターの連絡窓口との連絡活動を組織します。

2.4. Role of the IETF Trust
2.4. IETFトラストの役割

The IETF Trust [RFC4371] was formed to act as the administrative custodian of all copyrights and other intellectual property rights relating to the IETF Standards Process, a function that had previously been performed by the Internet Society (ISOC) and the Corporation for National Research Initiatives (CNRI).

IETFトラスト[RFC4371]は、IETF標準プロセスに関連するすべての著作権およびその他の知的財産権の管理責任者として機能するために設立されました。 （CNRI）。

Any intellectual property rights of IETF protocol parameter assignment information, including the registry and its contents, and all registry publications, are to be held by the IETF Trust on behalf of the IETF.

レジストリとその内容、およびすべてのレジストリ出版物を含む、IETFプロトコルパラメータ割り当て情報の知的財産権は、IETFに代わってIETFトラストが保有するものとします。

The IETF Trust may make such regulations as appropriate for the redistribution of assignment values and registry publications.

IETFトラストは、割り当て値の再配布とレジストリの公開に適切なように、そのような規制を作成する場合があります。

2.5. Role of the IETF Administration Limited Liability Company
2.5. IETF Administration Limited Liability Companyの役割

The IETF Administration Limited Liability Company (IETF LLC) [RFC8711] is responsible for identifying a potential vendor in a manner of its choosing, based on IAB consultation, and for managing the various aspects of the relationships with that vendor.

IETF管理有限責任会社（IETF LLC）[RFC8711]は、IABの協議に基づいて、選択した方法で潜在的なベンダーを特定し、そのベンダーとの関係のさまざまな側面を管理する責任があります。

In addition, the IETF LLC has the responsibility to ensure long-term access, stability, and uniqueness across all such registries. This responsibility is of particular significance in the event that a relation with a Protocol Parameter Registry Operator is terminated.

さらに、IETF LLCは、そのようなすべてのレジストリで長期的なアクセス、安定性、および一意性を確保する責任があります。この責任は、プロトコルパラメータレジストリオペレータとの関係が終了した場合に特に重要です。

3. Miscellaneous Considerations
3. その他の考慮事項

While this document has focused on the creation of protocols by the IETF, the requirements provided are generically applicable to the extended IETF community as well (e.g., Internet Research Task Force (IRTF)).

このドキュメントはIETFによるプロトコルの作成に焦点を当てていますが、提供された要件は一般に、拡張IETFコミュニティ（たとえば、インターネットリサーチタスクフォース（IRTF））にも適用できます。

The IESG is responsible for the technical direction of the IETF protocol parameter registries and maintaining the policies by which such technical directions are given. The IESG is responsible, as part of the document approval process associated with the IETF Stream RFCs [RFC8729], for "IANA Considerations" verification. For the other RFC streams, the approval bodies are responsible for verifying that the documents include "IANA Considerations" sections when needed, and that "IANA Considerations" sections conform to the current published guidelines. In the case that IANA considerations in non-IETF document streams lead to a dispute, the IAB makes the final decision.

IESGは、IETFプロトコルパラメータレジストリの技術的な指示と、そのような技術的な指示が与えられるポリシーの維持を担当します。 IESGは、IETFストリームRFC [RFC8729]に関連するドキュメント承認プロセスの一環として、「IANAに関する考慮事項」の検証を担当しています。他のRFCストリームの場合、承認機関は、必要に応じてドキュメントに「IANAの考慮事項」セクションが含まれていること、および「IANAの考慮事項」セクションが現在公開されているガイドラインに準拠していることを確認する責任があります。非IETFドキュメントストリームでのIANAの考慮事項が紛争につながる場合、IABが最終決定を下します。

This document talks about "Registry Operator" (singular), and while there are stability and economy-of-scale advantages for one single Registry Operator, this document does not exclude having different Registry Operators for different protocol registries when justified by the circumstances.

このドキュメントでは「レジストリオペレーター」（単数）について説明しています。1つのレジストリオペレーターには安定性と規模の経済性の利点がありますが、このドキュメントでは、状況に応じて異なるプロトコルレジストリに異なるレジストリオペレーターが存在することを除外していません。

4. Security Considerations
4. セキュリティに関する考慮事項

This document does not propose any new protocols and does not introduce any new security considerations.

このドキュメントでは、新しいプロトコルは提案されておらず、セキュリティに関する新しい考慮事項も紹介されていません。

5. IANA Considerations
5. IANAに関する考慮事項

This document requires no direct IANA actions in terms of the creation or operation of a protocol parameter registry. However, this document does define the roles and responsibilities of various bodies who are responsible for, and associated with, the operation of protocol parameter registration functions for the IETF.

このドキュメントは、プロトコルパラメータレジストリの作成または操作に関して、直接のIANAアクションを必要としません。ただし、このドキュメントでは、IETFのプロトコルパラメータ登録機能の運用を担当し、これに関連するさまざまな機関の役割と責任を定義しています。
