Abstract

概要

This document describes the testing result of a network utilizing a Mapping of Address and Port using Translation (MAP-T) double translation solution; it provides an overview of user applications' behavior with a shared IPv4 address.

このドキュメントでは、変換を使用したアドレスとポートのマッピング（MAP-T）二重変換ソリューションを利用したネットワークのテスト結果について説明します。共有IPv4アドレスでのユーザーアプリケーションの動作の概要を示します。

The MAP-T software is from CERNET Center and the test environment is on the NIC.br network with real and virtualized machines.

1. Introduction
1. はじめに

This testing is based on most common applications utilized by home users. The main purpose is to check if those applications work correctly on a network using MAP-T [RFC7599]. MAP-T is based on [RFC6791].

このテストは、ホームユーザーが利用する最も一般的なアプリケーションに基づいています。主な目的は、これらのアプリケーションがMAP-T [RFC7599]を使用してネットワーク上で正しく動作するかどうかを確認することです。 MAP-Tは[RFC6791]に基づいています。

Based on testing, we know which applications could be used on a network with MAP-T and the impact on a typical Internet user in Brazil. The classification as a working application is based on user experience, not on network measurements.

テストに基づいて、MAP-Tを使用するネットワークで使用できるアプリケーションと、ブラジルの一般的なインターネットユーザーへの影響を把握しています。動作するアプリケーションとしての分類は、ネットワーク測定ではなく、ユーザーエクスペリエンスに基づいています。

The MAP-T Core and MAP-T Customer Premises Equipment (CPE) are virtualized Linux machines running Fedora 11 and MAP-T 1.1 software developed by CERNET (https://github.com/cernet/MAP).

MAP-T CoreおよびMAP-T Customer Premises Equipment（CPE）は、Fedora 11を実行する仮想化Linuxマシンであり、CERNET（https://github.com/cernet/MAP）によって開発されたMAP-T 1.1ソフトウェアです。

The host machine is, in fact, three virtualized machines: one with Linux Kubuntu 12.04, one with Windows 7, and one with Windows XP. The host machine is replaced in the topology to test each operating system (OS).

ホストマシンは、実際には3つの仮想化マシンです。1つはLinux Kubuntu 12.04、1つはWindows 7、1つはWindows XPです。ホストマシンはトポロジ内で置き換えられ、各オペレーティングシステム（OS）をテストします。

The IPv6 router is a Linux machine with static routes to match the MAP-T network topology routes.

IPv6ルーターは、MAP-Tネットワークトポロジールートと一致する静的ルートを持つLinuxマシンです。

The NIC.br IPv4 Network is an IPv4 network with public IPv4 addresses. It is connected to the global IPv4 Internet.

NIC.br IPv4ネットワークは、パブリックIPv4アドレスを持つIPv4ネットワークです。グローバルIPv4インターネットに接続されています。

The NIC.br IPv6 Network is an IPv6 network with global unicast IPv6 addresses. It is connected to the global IPv6 Internet.

NIC.br IPv6ネットワークは、グローバルユニキャストIPv6アドレスを持つIPv6ネットワークです。グローバルIPv6インターネットに接続されています。

The host may have both IPv4 and IPv6 addresses, but to guarantee that the translation was being tested, the host received only an IPv4 address; the IPv4 packet must be translated to pass on the IPv6-only network. If the host is assigned an IPv6 address, this address should have a preference as specified in [RFC6555], and the translation might not be tested in some cases where the application or the content provider is available on IPv6.

ホストはIPv4とIPv6の両方のアドレスを持つことができますが、変換がテストされていることを保証するために、ホストはIPv4アドレスのみを受け取りました。 IPv4パケットは、IPv6のみのネットワークを通過するように変換する必要があります。ホストにIPv6アドレスが割り当てられている場合、このアドレスには[RFC6555]で指定されている優先順位が必要です。また、アプリケーションまたはコンテンツプロバイダーがIPv6で使用できる場合、変換がテストされないことがあります。

5. Effect Analysis
5. 効果分析
5.1. User Experience
5.1. ユーザー体験

User experience can only be evaluated subjectively: there is no quantitative rule to define if the user experience is acceptable. Network delay, streaming experience, and download time are similar to a network without MAP-T using IPv4 addresses and a NAT44 on the CPE.

ユーザーエクスペリエンスは主観的にのみ評価できます。ユーザーエクスペリエンスが許容できるかどうかを定義する定量的なルールはありません。ネットワーク遅延、ストリーミングエクスペリエンス、およびダウンロード時間は、CPEでIPv4アドレスとNAT44を使用するMAP-Tのないネットワークに似ています。

The user experience was reported to be very good. Almost all the software and websites worked correctly, the exceptions were the network traceroute and command-line FTP.

ユーザーエクスペリエンスは非常に良いと報告されています。ほとんどすべてのソフトウェアとWebサイトは正しく機能しましたが、例外はネットワークtracerouteとコマンドラインFTPでした。

The traceroute is only capable of reaching the MAP-T CPE and receiving a return message that the destination net is unreachable.

tracerouteは、MAP-T CPEに到達し、宛先ネットに到達できないという返信メッセージを受信することしかできません。

The command-line FTP, using active mode, is capable of connecting to the host with or without authentication, creating, deleting, and navigating folders, but it is not capable of listing folder contents or sending/receiving files. The differences between active and passive mode are explained in [RFC959].

コマンドラインFTPは、アクティブモードを使用して、認証ありまたはなしでホストに接続し、フォルダーを作成、削除、およびナビゲートできますが、フォルダーの内容を一覧表示したり、ファイルを送受信したりすることはできません。アクティブモードとパッシブモードの違いは[RFC959]で説明されています。

The BitTorrent applications can't seed or share files, since there aren't incoming connections to the host, it may cause some difficulties and low downloading speeds.

ホストへの着信接続がないため、BitTorrentアプリケーションはファイルをシードまたは共有できません。これにより、問題が発生し、ダウンロード速度が低下する場合があります。

5.2. Testing Summary
5.2. テストの要約

The working applications had no need of a special configuration to function. The user experience was reported to be very good. Almost all the software and websites worked correctly, the exceptions were the network traceroute and command-line FTP.

機能するアプリケーションは、機能するために特別な構成を必要としませんでした。ユーザーエクスペリエンスは非常に良いと報告されています。ほとんどすべてのソフトウェアとWebサイトは正しく機能しましたが、例外はネットワークtracerouteとコマンドラインFTPでした。

The command-line FTP doesn't work correctly because active mode requires incoming connections to specific ports without having an outbound connection on those ports. When the test was run on FTP passive mode on Linux, the FTP worked correctly. When the test was run on FTP passive mode on Windows 7 and Windows XP, the FTP didn't work correctly. When using FileZilla FTP in passive mode, FTP worked on all tested operating systems.

コマンドラインFTPは正しく機能しません。アクティブモードでは、特定のポートへの送信接続がなくても、それらのポートへの受信接続が必要になるためです。 LinuxのFTPパッシブモードでテストを実行すると、FTPは正しく機能しました。 Windows 7およびWindows XPのFTPパッシブモードでテストを実行すると、FTPが正しく機能しませんでした。 FileZilla FTPをパッシブモードで使用すると、テストされたすべてのオペレーティングシステムでFTPが機能しました。

The network traceroute doesn't work in inbound or outbound directions because there is no continuity of the IPv4 network, as it is interrupted by an IPv6-only network. The tool is not capable of detecting the hosts in this IPv6-only network and because of that the traceroute doesn't succeed.

IPv6のみのネットワークによって中断されるため、IPv4ネットワークの連続性がないため、ネットワークtracerouteはインバウンド方向またはアウトバウンド方向では機能しません。このツールは、このIPv6のみのネットワークでホストを検出できません。そのため、tracerouteは成功しません。

The testing was run with the version 1.1 of the MAP-T software developed by CERNET. In this version, the MAP-T CPE uses a NAT44, so it is not possible to receive incoming connections even on the ports assigned to the host by the address plus port division. Because of that, it is not possible to configure a server on the host. We provided this feedback to the CERNET team and they developed a new version of the software (2.2c) that is capable of receiving incoming connections on specific ports in each CPE.

テストは、CERNETによって開発されたMAP-Tソフトウェアのバージョン1.1で実行されました。このバージョンでは、MAP-T CPEはNAT44を使用しているため、アドレスとポート分割によってホストに割り当てられたポートでも着信接続を受信できません。そのため、ホスト上にサーバーを構成することはできません。このフィードバックをCERNETチームに提供し、彼らは各CPEの特定のポートで着信接続を受信できるソフトウェアの新しいバージョン（2.2c）を開発しました。

6. Future Testing Considerations
6. 今後のテストに関する考慮事項

Future tests should consider testing Mapping of Address and Port with Encapsulation (MAP-E) [RFC7597] as well (currently supported in the new version of CERNET's MAP software). Other operating systems (Mac OS, Android, IOS, etc.), devices (mobile phones, tablets, video games, etc.), and programs (mail and instant messaging clients and servers) could be tested, too. Another consideration is the port sharing ratio and if it affects performance and functionalities. The results of the testing made in MAP Interoperability Testing Results [MAP-TESTING] are also relevant for future tests. Impact on other protocols, like DNS and DNSSEC, could also be verified.

今後のテストでは、アドレスとポートのマッピングとカプセル化（MAP-E）[RFC7597]のテストも検討する必要があります（現在、CERNETのMAPソフトウェアの新しいバージョンでサポートされています）。その他のオペレーティングシステム（Mac OS、Android、IOSなど）、デバイス（携帯電話、タブレット、ビデオゲームなど）、およびプログラム（メールおよびインスタントメッセージングのクライアントとサーバー）もテストできます。もう1つの考慮事項は、ポート共有率と、それがパフォーマンスと機能に影響を与えるかどうかです。 MAP相互運用性テスト結果[MAP-TESTING]で行われたテストの結果は、将来のテストにも関連しています。 DNSやDNSSECなどの他のプロトコルへの影響も確認できます。

A.10.4. Note about BitTorrent Seeders
A.10.4. BitTorrentシーダーに関する注意

BitTorrent uses distributed queues, each seeder owns the queues for the files they have. The seeder informs the tracker that it has the file and the tracker informs the clients about this seeder. The client sends messages to this announced seeder to try to download the file. With no incoming connection, the BitTorrent fails here on the MAP-T. Despite being a possible seeder, the machine doesn't upload the file; without upload, it will be considered a leech and will be penalized on future downloading speeds. This is the same problem that happens on a network with IPv4 without port forwarding. For details about the influence on shared IPv4 addresses on torrent, see "Behavior of BitTorrent service in PCP-enabled networks with Address Sharing" [PCP-BITTORRENT].

BitTorrentは分散キューを使用し、各シーダーはそれらが持つファイルのキューを所有します。シーダーはファイルがあることをトラッカーに通知し、トラッカーはこのシーダーについてクライアントに通知します。クライアントは、このアナウンスされたシーダーにメッセージを送信して、ファイルをダウンロードしようとします。着信接続がない場合、BitTorrentはMAP-Tで失敗します。シーダーの可能性があるにもかかわらず、マシンはファイルをアップロードしません。アップロードしない場合、リーチと見なされ、将来のダウンロード速度にペナルティが課されます。これは、ポート転送のないIPv4のネットワークで発生するのと同じ問題です。トレント上の共有IPv4アドレスへの影響の詳細については、「アドレス共有を使用したPCP対応ネットワークでのBitTorrentサービスの動作」[PCP-BITTORRENT]を参照してください。
