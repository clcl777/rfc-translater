1. Introduction
1. はじめに

The IETF has specified YANG modules for VPN services, e.g., the Layer 3 VPN Service Model (L3SM) [RFC8299] or the Layer 2 VPN Service Model (L2SM) [RFC8466]. Other relevant YANG data models are the Layer 3 VPN Network Model (L3NM) [RFC9182] and the Layer 2 VPN Network Model (L2NM) [L2NM-YANG]. There are common data nodes and structures that are present in all of these models or at least a subset of them.

IETFは、VPNサービス、例えばレイヤ3 VPNサービスモデル（L3SM）[RFC8299]またはレイヤ2 VPNサービスモデル（L2SM）[RFC8466]のYANGモジュールを指定しています。その他の関連YANGデータモデルは、レイヤ3 VPNネットワークモデル（L3NM）[RFC9182]とレイヤ2 VPNネットワークモデル（L2NM）[L2NM-YANG]です。これらのモデルのすべてまたはそれらの少なくともサブセットに存在する一般的なデータノードと構造体があります。

This document defines a common YANG module that is meant to be reused by various VPN-related modules such as the L3NM [RFC9182] and the L2NM [L2NM-YANG]: "ietf-vpn-common" (Section 4).

この文書は、L3NM [RFC9182]やL2NM [L2NM-YANG]など、さまざまなVPN関連モジュールによって再利用されることを目的とした一般的なYANGモジュールを定義しています（セクション4）。

The "ietf-vpn-common" module includes a set of identities, types, and groupings that are meant to be reused by other VPN-related YANG modules independently of their layer (e.g., Layer 2, Layer 3) and the type of the module (e.g., network model, service model), including possible future revisions of existing models (e.g., the L3SM [RFC8299] or the L2SM [RFC8466]).

「IETF - VPN - Common」モジュールは、それらの層（例えば、レイヤ2、レイヤ3）およびの種類とは無関係に他のVPN関連YANモジュールによって再使用されることを意図した一連の識別、タイプ、およびグループ化を含む。既存のモデルの将来のリビジョンを含むモジュール（例えば、ネットワークモデル、サービスモデル）（例えば、L3SM [RFC8299]またはL2SM [RFC8466]）。

The descriptions of the common groupings are provided below:

一般的なグループの説明は以下のとおりです。

'vpn-description': A YANG grouping that provides common administrative VPN information such as an identifier, a name, a textual description, and a customer name.

'vpn-description'：識別子、名前、テキスト記述、および顧客名などの一般的な管理VPN情報を提供するYANGグループ化。

'vpn-profile-cfg': A YANG grouping that defines a set of valid profiles (encryption, routing, forwarding, etc.) that can be bound to a Layer 2/3 VPN. This document does not make any assumptions about the structure of such profiles but allows "gluing" a VPN service with other parameters that can be required locally to provide value-added features to requesting customers.

'vpn-profile-cfg'：レイヤ2/3 VPNにバインドできる有効なプロファイルのセット（暗号化、ルーティング、転送など）を定義するYANDグループ化。この文書はそのようなプロファイルの構造についての仮定をしませんが、顧客を要求するためにバリューに追加された機能を提供するためにローカルに要求される可能性がある他のパラメータとのVPNサービスを「接着」することを可能にします。

For example, a service provider may provide external connectivity to a VPN customer (e.g., to a private or public cloud, Internet). Such a service may involve tweaking both filtering and NAT rules (e.g., binding a Virtual Routing and Forwarding (VRF) interface with a NAT instance as discussed in Section 2.10 of [RFC8512]). These value-added features may be bound to all, or a subset of, network accesses. Some of these value-added features may be implemented in nodes other than Provider Edges (PEs) (e.g., a P node or even a dedicated node that hosts the NAT function).

例えば、サービスプロバイダは、VPN顧客（例えば、プライベートまたはパブリッククラウド、インターネットに）に外部接続を提供することができる。そのようなサービスは、[RFC8512]のセクション2.10で説明されているように、フィルタリングおよびNATルールの両方を微調整することを含み得る（例えば、2.10節）。これらの付加価値機能は、ネットワークアクセスのすべて、またはサブセットにバインドされてもよい。これらの付加価値機能のいくつかは、プロバイダエッジ（PES）以外のノード（例えば、PノードまたはNAT機能をホストする専用ノード）に実装されてもよい。

Elaborating on the structure of these profiles is beyond the scope of this document.

これらのプロファイルの構造を詳しく説明することはこの文書の範囲を超えています。

'oper-status-timestamp': A YANG grouping that defines the operational status updates of a VPN service or component.

'oper-status-timestamp'：VPNサービスまたはコンポーネントの運用ステータスの更新を定義するYANGグループ化。

'service-status': A YANG grouping that defines the administrative and operational status of a component. The grouping can be applied to the whole service or an endpoint.

'service-status'：コンポーネントの管理ステータスと運用ステータスを定義するYANGグループ化。グループ化は、サービス全体またはエンドポイントに適用できます。

'underlay-transport': A YANG grouping that defines the type of the underlay transport for a VPN service or how that underlay is set.

'underlay-transport'：VPNサービスのアンダーレイトランスポートの種類またはその下方が設定されている方法を定義するYANGグループ化。

5. Security Considerations
5. セキュリティに関する考慮事項

The YANG module specified in this document defines a schema for data that is designed to be accessed via network management protocols such as NETCONF [RFC6241] or RESTCONF [RFC8040]. The lowest NETCONF layer is the secure transport layer, and the mandatory-to-implement secure transport is Secure Shell (SSH) [RFC6242]. The lowest RESTCONF layer is HTTPS, and the mandatory-to-implement secure transport is TLS [RFC8446].

このドキュメントで指定されているYANGモジュールは、NetConf [RFC6241]またはRESTCONF [RFC8040]などのネットワーク管理プロトコルを介してアクセスするように設計されているデータのスキーマを定義しています。最低のNETCONFレイヤーはセキュアトランスポート層であり、必須のセキュアトランスポートはSecure Shell（SSH）[RFC6242]です。最低のRETCONFレイヤーはhttpsで、必須のセキュアトランスポートはTLS [RFC8446]です。

The Network Configuration Access Control Model (NACM) [RFC8341] provides the means to restrict access for particular NETCONF or RESTCONF users to a preconfigured subset of all available NETCONF or RESTCONF protocol operations and content.

ネットワーク構成アクセス制御モデル（NACM）[RFC8341]は、特定のNETCONFまたはRESTCONFユーザへのアクセスを利用可能なすべてのNetConfまたはRESTCONFプロトコル操作およびコンテンツの事前設定されたサブセットに制限する手段を提供します。

The "ietf-vpn-common" module defines a set of identities, types, and groupings. These nodes are intended to be reused by other YANG modules. The module by itself does not expose any data nodes that are writable, data nodes that contain read-only state, or RPCs. As such, there are no additional security issues related to the "ietf-vpn-common" module that need to be considered.

「IETF-VPN-Common」モジュールは、一連のID、型、およびグループ化を定義します。これらのノードは他のYangモジュールによって再利用されることを意図しています。モジュールによるモジュールは、書き込み可能なデータノード、読み取り専用状態を含むデータノード、またはRPCSを公開しません。そのため、考慮する必要がある「IETF-VPN-Common」モジュールに関連する追加のセキュリティ問題はありません。

Modules that use the groupings that are defined in this document should identify the corresponding security considerations. For example, reusing some of these groupings will expose privacy-related information (e.g., 'customer-name'). Disclosing such information may be considered a violation of the customer-provider trust relationship.

この文書で定義されているグループ化を使用するモジュールは、対応するセキュリティ上の考慮事項を識別する必要があります。たとえば、これらのグループのいくつかを再利用すると、プライバシー関連の情報（例えば、「顧客名」）が表示されます。そのような情報を開示することは、顧客プロバイダ信頼関係の違反と見なされ得る。

Appendix A. Example of Common Data Nodes in Early L2NM/L3NM Designs

付録A.初期のL2NM / L3NMデザインにおける一般的なデータノードの例

In order to avoid duplication of data nodes and to ease passing data among layers (i.e., from the service layer to the network layer and vice versa), early versions of the L3NM reused many of the data nodes that are defined in the L3SM. Nevertheless, that approach was abandoned because that design was interpreted as if the deployment of the L3NM depends on the L3SM, while this is not required. For example, a service provider may decide to use the L3NM to build its L3VPN services without exposing the L3SM to customers.

データノードの重複を回避し、層間（サービス層からネットワーク層へ、およびその逆に）データを渡すために、L3NMの初期のバージョンは、L3SMで定義されている多くのデータノードを再利用しました。それにもかかわらず、そのアプローチは、L3NMの展開がL3SMに依存しているかのようにその設計が解釈されたため、そのアプローチは放棄されましたが、これは必要ありません。たとえば、サービスプロバイダは、L3SMを顧客に公開することなく、L3VPNサービスを構築するためにL3NMを使用することを決定することができます。

Likewise, early versions of the L2NM reused many of the data nodes that are defined in both the L2SM and the L3NM. An example of L3NM groupings reused in the L2NM is shown in Figure 5. Such reuse of data nodes was interpreted as if the deployment of the L2NM requires support for the L3NM, which is not required.

同様に、L2NMの初期バージョンは、L2SMとL3NMの両方で定義されているデータノードの多くを再使用しました。L2NMで再使用されたL3NMグループの例を図5に示します。このようなデータノードの再利用は、L2NMの展開がL3NMのサポートを必要とするかのように解釈されています。 

   module ietf-l2vpn-ntw {
    ...
     import ietf-l3vpn-ntw {
       prefix l3vpn-ntw;
       reference
         "RFC 9182: A YANG Network Data Model for Layer 3 VPNs";
     }
     ...
     container l2vpn-ntw {
       ...
       container vpn-services {
         list vpn-service {
           ...
           uses l3vpn-ntw:service-status;
           uses l3vpn-ntw:svc-transport-encapsulation;
           ...
         }
       }
       ...
     }
   }
