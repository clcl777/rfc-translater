1. Introduction
1. はじめに

One of WebRTC's key features is its support of peer-to-peer connections. However, when establishing such a connection, which involves connection attempts from various IP addresses, WebRTC may allow a web application to learn additional information about the user compared to an application that only uses the Hypertext Transfer Protocol (HTTP) [RFC7230]. This may be problematic in certain cases. This document summarizes the concerns and makes recommendations on how WebRTC implementations should best handle the trade-off between privacy and media performance.

WebRTCの重要な機能の1つは、ピアツーピア接続のサポートです。ただし、さまざまなIPアドレスからの接続試行を含むこのような接続を確立する場合、WebRTCは、ハイパーテキスト転送プロトコル（HTTP）[RFC7230]のみを使用するアプリケーションと比較して、Webアプリケーションがユーザーに関する追加情報を学習できるようにする場合があります。これは、特定の場合に問題になる可能性があります。このドキュメントでは、懸念事項を要約し、WebRTC実装がプライバシーとメディアパフォーマンスの間のトレードオフを最適に処理する方法について推奨します。

3. Problem Statement
3. 問題文

In order to establish a peer-to-peer connection, WebRTC implementations use Interactive Connectivity Establishment (ICE) [RFC8445]. ICE attempts to discover multiple IP addresses using techniques such as Session Traversal Utilities for NAT (STUN) [RFC5389] and Traversal Using Relays around NAT (TURN) [RFC5766] and then checks the connectivity of each local-address-remote-address pair in order to select the best one. The addresses that are collected usually consist of an endpoint's private physical or virtual addresses and its public Internet addresses.

ピアツーピア接続を確立するために、WebRTC実装はInteractive Connectivity Establishment（ICE）[RFC8445]を使用します。ICEは、NATのセッショントラバーサルユーティリティ（STUN）[RFC5389]やNAT周辺のリレーを使用したトラバーサル（TURN）[RFC5766]などの手法を使用して、複数のIPアドレスを検出しようとします。次に、の各ローカルアドレス-リモートアドレスペアの接続を確認します。最適なものを選択するために。収集されるアドレスは通常、エンドポイントのプライベート物理アドレスまたは仮想アドレスとそのパブリックインターネットアドレスで構成されます。

These addresses are provided to the web application so that they can be communicated to the remote endpoint for its checks. This allows the application to learn more about the local network configuration than it would from a typical HTTP scenario, in which the web server would only see a single public Internet address, i.e., the address from which the HTTP request was sent.

これらのアドレスは、チェックのためにリモートエンドポイントと通信できるように、Webアプリケーションに提供されます。これにより、アプリケーションは、Webサーバーが単一のパブリックインターネットアドレス、つまりHTTP要求の送信元のアドレスのみを参照する一般的なHTTPシナリオよりも、ローカルネットワーク構成について詳しく知ることができます。

The additional information revealed falls into three categories:

明らかになった追加情報は、次の3つのカテゴリに分類されます。

1. If the client is multihomed, additional public IP addresses for the client can be learned. In particular, if the client tries to hide its physical location through a Virtual Private Network (VPN), and the VPN and local OS support routing over multiple interfaces (a "split-tunnel" VPN), WebRTC can discover not only the public address for the VPN, but also the ISP public address over which the VPN is running.

1. クライアントがマルチホームの場合、クライアントの追加のパブリックIPアドレスを学習できます。特に、クライアントが仮想プライベートネットワーク（VPN）を介して物理的な場所を隠そうとし、VPNとローカルOSが複数のインターフェイス（「スプリットトンネル」VPN）を介したルーティングをサポートしている場合、WebRTCはパブリックアドレスだけでなく検出することもできます。VPNの場合だけでなく、VPNが実行されているISPパブリックアドレスの場合もあります。

2. If the client is behind a Network Address Translator (NAT), the client's private IP addresses, often [RFC1918] addresses, can be learned.

2. クライアントがネットワークアドレス変換（NAT）の背後にある場合、クライアントのプライベートIPアドレス（多くの場合[RFC1918]アドレス）を学習できます。

3. If the client is behind a proxy (a client-configured "classical application proxy", as defined in [RFC1919], Section 3), but direct access to the Internet is permitted, WebRTC's STUN checks will bypass the proxy and reveal the public IP address of the client. This concern also applies to the "enterprise TURN server" scenario described in [RFC7478], Section 2.3.5.1 if, as above, direct Internet access is permitted. However, when the term "proxy" is used in this document, it is always in reference to an [RFC1919] proxy server.

3. クライアントがプロキシ（[RFC1919]のセクション3で定義されているクライアント構成の「クラシックアプリケーションプロキシ」）の背後にあるが、インターネットへの直接アクセスが許可されている場合、WebRTCのSTUNチェックはプロキシをバイパスし、パブリックIPを明らかにしますクライアントのアドレス。この懸念は、上記のように直接インターネットアクセスが許可されている場合、[RFC7478]のセクション2.3.5.1で説明されている「エンタープライズTURNサーバー」シナリオにも当てはまります。ただし、このドキュメントで「プロキシ」という用語が使用されている場合は、常に[RFC1919]プロキシサーバーを指します。

Of these three concerns, the first is the most significant, because for some users, the purpose of using a VPN is for anonymity. However, different VPN users will have different needs, and some VPN users (e.g., corporate VPN users) may in fact prefer WebRTC to send media traffic directly -- i.e., not through the VPN.

一部のユーザーにとって、VPNを使用する目的は匿名性であるため、これら3つの懸念のうち、最初の懸念が最も重要です。ただし、VPNユーザーごとにニーズは異なり、一部のVPNユーザー（企業VPNユーザーなど）は、実際にはWebRTCを使用してメディアトラフィックを直接送信することを好む場合があります。つまり、VPN経由ではありません。

The second concern is less significant but valid nonetheless. The core issue is that web applications can learn about addresses that are not exposed to the Internet; typically, these address are IPv4, but they can also be IPv6, as in the case of NAT64 [RFC6146]. While disclosure of the [RFC4941] IPv6 addresses recommended by [RFC8835] is fairly benign due to their intentionally short lifetimes, IPv4 addresses present some challenges. Although private IPv4 addresses often contain minimal entropy (e.g., 192.168.0.2, a fairly common address), in the worst case, they can contain 24 bits of entropy with an indefinite lifetime. As such, they can be a fairly significant fingerprinting surface. In addition, intranet web sites can be attacked more easily when their IPv4 address range is externally known.

2番目の懸念はそれほど重要ではありませんが、それでも有効です。中心的な問題は、Webアプリケーションがインターネットに公開されていないアドレスについて学習できることです。通常、これらのアドレスはIPv4ですが、NAT64 [RFC6146]の場合のように、IPv6にすることもできます。[RFC8835]が推奨する[RFC4941] IPv6アドレスの開示は、意図的に寿命が短いためかなり害がありませんが、IPv4アドレスにはいくつかの課題があります。プライベートIPv4アドレスには、最小限のエントロピー（たとえば、192.168.0.2、かなり一般的なアドレス）が含まれていることがよくありますが、最悪の場合、24ビットのエントロピーが含まれている可能性があります。そのため、それらはかなり重要な指紋の表面になる可能性があります。さらに、イントラネットWebサイトは、IPv4アドレス範囲が外部からわかっていると、より簡単に攻撃される可能性があります。

Private IP addresses can also act as an identifier that allows web applications running in isolated browsing contexts (e.g., normal and private browsing) to learn that they are running on the same device. This could allow the application sessions to be correlated, defeating some of the privacy protections provided by isolation. It should be noted that private addresses are just one potential mechanism for this correlation and this is an area for further study.

プライベートIPアドレスは、分離されたブラウジングコンテキスト（通常のブラウジングとプライベートブラウジングなど）で実行されているWebアプリケーションが同じデバイスで実行されていることを学習できるようにする識別子としても機能します。これにより、アプリケーションセッションを相互に関連付けることができ、分離によって提供されるプライバシー保護の一部が無効になる可能性があります。プライベートアドレスは、この相関関係の潜在的なメカニズムの1つにすぎず、これは今後の研究領域であることに注意してください。

The third concern is the least common, as proxy administrators can already control this behavior through organizational firewall policy, and generally, forcing WebRTC traffic through a proxy server will have negative effects on both the proxy and media quality.

3番目の懸念は最も一般的ではありません。プロキシ管理者はすでに組織のファイアウォールポリシーを介してこの動作を制御でき、一般に、プロキシサーバーを介してWebRTCトラフィックを強制すると、プロキシとメディアの両方の品質に悪影響を及ぼします。

Note also that these concerns predate WebRTC; Adobe Flash Player has provided similar functionality since the introduction of Real-Time Media Flow Protocol (RTMFP) support [RFC7016] in 2008.

これらの懸念はWebRTCよりも前のものであることに注意してください。Adobe Flash Playerは、2008年にReal-Time Media Flow Protocol（RTMFP）サポート[RFC7016]が導入されて以来、同様の機能を提供してきました。

4. Goals
4. 目標

WebRTC's support of secure peer-to-peer connections facilitates deployment of decentralized systems, which can have privacy benefits. As a result, blunt solutions that disable WebRTC or make it significantly harder to use are undesirable. This document takes a more nuanced approach, with the following goals:

WebRTCによる安全なピアツーピア接続のサポートにより、分散型システムの展開が容易になり、プライバシー上のメリットが得られます。その結果、WebRTCを無効にしたり、使用を大幅に困難にしたりする鈍いソリューションは望ましくありません。このドキュメントは、次の目標を持って、より微妙なアプローチを取ります。

* Provide a framework for understanding the problem so that controls might be provided to make different trade-offs regarding performance and privacy concerns with WebRTC.

* 問題を理解するためのフレームワークを提供して、WebRTCのパフォーマンスとプライバシーの懸念に関してさまざまなトレードオフを行うための制御が提供されるようにします。

* Using that framework, define settings that enable peer-to-peer communications, each with a different balance between performance and privacy.

* そのフレームワークを使用して、パフォーマンスとプライバシーのバランスがそれぞれ異なるピアツーピア通信を可能にする設定を定義します。

* Finally, provide recommendations for default settings that provide reasonable performance without also exposing addressing information in a way that might violate user expectations.

* 最後に、ユーザーの期待に反する可能性のある方法でアドレス情報を公開することなく、妥当なパフォーマンスを提供するデフォルト設定の推奨事項を提供します。

5. Detailed Design
5. きめ細かなデザイン
5.1. Principles
5.1. 原則

The key principles for our framework are stated below:

私たちのフレームワークの主要な原則は以下のとおりです。

1. By default, WebRTC traffic should follow typical IP routing (i.e., WebRTC should use the same interface used for HTTP traffic) and only the system's 'typical' public addresses (or those of an enterprise TURN server, if present) should be visible to the application. However, in the interest of optimal media quality, it should be possible to enable WebRTC to make use of all network interfaces to determine the ideal route.

1. デフォルトでは、WebRTCトラフィックは通常のIPルーティングに従う必要があり（つまり、WebRTCはHTTPトラフィックに使用されるのと同じインターフェイスを使用する必要があります）、システムの「通常の」パブリックアドレス（または存在する場合はエンタープライズTURNサーバーのパブリックアドレス）のみが表示されます。応用。ただし、最適なメディア品質を実現するために、WebRTCがすべてのネットワークインターフェイスを利用して理想的なルートを決定できるようにする必要があります。

2. By default, WebRTC should be able to negotiate direct peer-to-peer connections between endpoints (i.e., without traversing a NAT or relay server) when such connections are possible. This ensures that applications that need true peer-to-peer routing for bandwidth or latency reasons can operate successfully.

2. デフォルトでは、WebRTCは、エンドポイント間で直接ピアツーピア接続をネゴシエートできる必要があります（つまり、NATまたはリレーサーバーを経由せずに）。これにより、帯域幅または遅延の理由で真のピアツーピアルーティングを必要とするアプリケーションが正常に動作できるようになります。

3. It should be possible to configure WebRTC to not disclose private local IP addresses, to avoid the issues associated with web applications learning such addresses. This document does not require this to be the default state, as there is no currently defined mechanism that can satisfy this requirement as well as the aforementioned requirement to allow direct peer-to-peer connections.

3. プライベートローカルIPアドレスを開示しないようにWebRTCを構成して、そのようなアドレスを学習するWebアプリケーションに関連する問題を回避できるようにする必要があります。この要件と、直接ピアツーピア接続を許可する前述の要件を満たすことができるメカニズムが現在定義されていないため、このドキュメントではこれをデフォルトの状態にする必要はありません。

4. By default, WebRTC traffic should not be sent through proxy servers, due to the media-quality problems associated with sending WebRTC traffic over TCP, which is almost always used when communicating with such proxies, as well as proxy performance issues that may result from proxying WebRTC's long-lived, high-bandwidth connections. However, it should be possible to force WebRTC to send its traffic through a configured proxy if desired.

4. デフォルトでは、WebRTCトラフィックはプロキシサーバー経由で送信しないでください。これは、TCPを介したWebRTCトラフィックの送信に関連するメディア品質の問題と、プロキシとの通信時にほとんど常に使用される問題、およびプロキシに起因するプロキシパフォーマンスの問題があるためです。WebRTCの長寿命の高帯域幅接続。ただし、必要に応じて、WebRTCが構成済みのプロキシを介してトラフィックを送信するように強制することは可能です。

5.2. Modes and Recommendations
5.2. モードと推奨事項

Based on these ideas, we define four specific modes of WebRTC behavior, reflecting different media quality/privacy trade-offs:

これらのアイデアに基づいて、さまざまなメディア品質/プライバシーのトレードオフを反映して、WebRTC動作の4つの特定のモードを定義します。

Mode 1 - Enumerate all addresses: WebRTC MUST use all network interfaces to attempt communication with STUN servers, TURN servers, or peers. This will converge on the best media path and is ideal when media performance is the highest priority, but it discloses the most information.

モード1-すべてのアドレスを列挙する：WebRTCは、すべてのネットワークインターフェースを使用して、STUNサーバー、TURNサーバー、またはピアとの通信を試行する必要があります。これは最適なメディアパスに収束し、メディアパフォーマンスが最優先される場合に理想的ですが、ほとんどの情報を開示します。

Mode 2 - Default route + associated local addresses: WebRTC MUST follow the kernel routing table rules, which will typically cause media packets to take the same route as the application's HTTP traffic. If an enterprise TURN server is present, the preferred route MUST be through this TURN server. Once an interface has been chosen, the private IPv4 and IPv6 addresses associated with this interface MUST be discovered and provided to the application as host candidates. This ensures that direct connections can still be established in this mode.

モード2-デフォルトルートに関連付けられたローカルアドレス：WebRTCはカーネルルーティングテーブルのルールに従う必要があります。これにより、通常、メディアパケットはアプリケーションのHTTPトラフィックと同じルートをたどります。エンタープライズTURNサーバーが存在する場合、優先ルートはこのTURNサーバーを経由する必要があります。インターフェイスが選択されると、このインターフェイスに関連付けられているプライベートIPv4およびIPv6アドレスが検出され、ホスト候補としてアプリケーションに提供される必要があります。これにより、このモードでも直接接続を確立できます。

Mode 3 - Default route only: This is the same as Mode 2, except that the associated private addresses MUST NOT be provided; the only IP addresses gathered are those discovered via mechanisms like STUN and TURN (on the default route). This may cause traffic to hairpin through a NAT, fall back to an application TURN server, or fail altogether, with resulting quality implications.

モード3-デフォルトルートのみ：これはモード2と同じですが、関連するプライベートアドレスを指定してはならない点が異なります。収集されるIPアドレスは、STUNやTURN（デフォルトルート上）などのメカニズムを介して検出されたものだけです。これにより、トラフィックがNATを介してヘアピンしたり、アプリケーションのTURNサーバーにフォールバックしたり、完全に失敗したりして、品質に影響が出る可能性があります。

Mode 4 - Force proxy: This is the same as Mode 3, but when the application's HTTP traffic is sent through a proxy, WebRTC media traffic MUST also be proxied. If the proxy does not support UDP (as is the case for all HTTP and most SOCKS [RFC1928] proxies), or the WebRTC implementation does not support UDP proxying, the use of UDP will be disabled, and TCP will be used to send and receive media through the proxy. Use of TCP will result in reduced media quality, in addition to any performance considerations associated with sending all WebRTC media through the proxy server.

モード4-強制プロキシ：これはモード3と同じですが、アプリケーションのHTTPトラフィックがプロキシを介して送信される場合、WebRTCメディアトラフィックもプロキシされる必要があります。プロキシがUDPをサポートしていない場合（すべてのHTTPおよびほとんどのSOCKS [RFC1928]プロキシの場合のように）、またはWebRTC実装がUDPプロキシをサポートしていない場合、UDPの使用は無効になり、TCPを使用して送信およびプロキシを介してメディアを受信します。TCPを使用すると、プロキシサーバーを介したすべてのWebRTCメディアの送信に関連するパフォーマンスの考慮事項に加えて、メディアの品質が低下します。

Mode 1 MUST NOT be used unless user consent has been provided. The details of this consent are left to the implementation; one potential mechanism is to tie this consent to getUserMedia (device permissions) consent, described in [RFC8827], Section 6.2. Alternatively, implementations can provide a specific mechanism to obtain user consent.

ユーザーの同意が得られない限り、モード1を使用してはなりません。この同意の詳細は実装に任されています。考えられるメカニズムの1つは、[RFC8827]のセクション6.2で説明されているように、この同意をgetUserMedia（デバイス権限）の同意に関連付けることです。あるいは、実装は、ユーザーの同意を得るための特定のメカニズムを提供できます。

In cases where user consent has not been obtained, Mode 2 SHOULD be used.

ユーザーの同意が得られていない場合は、モード2を使用する必要があります。

These defaults provide a reasonable trade-off that permits trusted WebRTC applications to achieve optimal network performance but gives applications without consent (e.g., 1-way streaming or data-channel applications) only the minimum information needed to achieve direct connections, as defined in Mode 2. However, implementations MAY choose stricter modes if desired, e.g., if a user indicates they want all WebRTC traffic to follow the default route.

これらのデフォルトは、信頼できるWebRTCアプリケーションが最適なネットワークパフォーマンスを達成できるようにする合理的なトレードオフを提供しますが、同意なしにアプリケーション（たとえば、一方向ストリーミングまたはデータチャネルアプリケーション）に、モードで定義されているように、直接接続を達成するために必要な最小限の情報のみを提供します。2.ただし、実装では、必要に応じてより厳密なモードを選択できます。たとえば、ユーザーがすべてのWebRTCトラフィックをデフォルトルートに従わせたい場合などです。

Future documents may define additional modes and/or update the recommended default modes.

将来のドキュメントでは、追加のモードを定義したり、推奨されるデフォルトモードを更新したりする可能性があります。

Note that the suggested defaults can still be used even for organizations that want all external WebRTC traffic to traverse a proxy or enterprise TURN server, simply by setting an organizational firewall policy that allows WebRTC traffic to only leave through the proxy or TURN server. This provides a way to ensure the proxy or TURN server is used for any external traffic but still allows direct connections (and, in the proxy case, avoids the performance issues associated with forcing media through said proxy) for intra-organization traffic.

提案されたデフォルトは、すべての外部WebRTCトラフィックがプロキシまたはエンタープライズTURNサーバーを通過することを望む組織でも、WebRTCトラフィックがプロキシまたはTURNサーバーを通過することのみを許可する組織ファイアウォールポリシーを設定するだけで使用できることに注意してください。これにより、プロキシまたはTURNサーバーが外部トラフィックに使用されることを保証しながら、組織内トラフィックの直接接続が可能になります（プロキシの場合、メディアをプロキシ経由で強制することに関連するパフォーマンスの問題が回避されます）。

6. Implementation Guidance
6. 実装ガイダンス

This section provides guidance to WebRTC implementations on how to implement the policies described above.

このセクションでは、上記のポリシーを実装する方法に関するWebRTC実装のガイダンスを提供します。

6.1. Ensuring Normal Routing
6.1. 通常のルーティングの確保

When trying to follow typical IP routing, as required by Modes 2 and 3, the simplest approach is to bind() the sockets used for peer-to-peer connections to the wildcard addresses (0.0.0.0 for IPv4, :: for IPv6), which allows the OS to route WebRTC traffic the same way as it would HTTP traffic. STUN and TURN will work as usual, and host candidates can still be determined as mentioned below.

モード2および3で必要とされる、一般的なIPルーティングに従おうとする場合、最も簡単なアプローチは、ワイルドカードアドレス（IPv4の場合は0.0.0.0、IPv6の場合は：:）へのピアツーピア接続に使用されるソケットをbind（）することです。、これにより、OSはHTTPトラフィックと同じ方法でWebRTCトラフィックをルーティングできます。STUNとTURNは通常どおり機能しますが、ホスト候補は以下のように決定できます。

6.2. Determining Associated Local Addresses
6.2. 関連するローカルアドレスの決定

When binding to a wildcard address, some extra work is needed to determine the associated local address required by Mode 2, which we define as the source address that would be used for any packets sent to the web application host (assuming that UDP and TCP get the same routing treatment). Use of the web-application host as a destination ensures the right source address is selected, regardless of where the application resides (e.g., on an intranet).

ワイルドカードアドレスにバインドする場合、モード2で必要な関連ローカルアドレスを決定するために追加の作業が必要です。モード2は、Webアプリケーションホストに送信されるパケットに使用される送信元アドレスとして定義されます（UDPとTCPが取得すると仮定）同じルーティング処理）。Webアプリケーションホストを宛先として使用すると、アプリケーションが存在する場所（イントラネットなど）に関係なく、正しい送信元アドレスが選択されます。

First, the appropriate remote IPv4/IPv6 address is obtained by resolving the host component of the web application URI [RFC3986]. If the client is behind a proxy and cannot resolve these IPs via DNS, the address of the proxy can be used instead. Or, if the web application was loaded from a file:// URI [RFC8089] rather than over the network, the implementation can fall back to a well-known DNS name or IP address.

まず、WebアプリケーションURI [RFC3986]のホストコンポーネントを解決することにより、適切なリモートIPv4 / IPv6アドレスが取得されます。クライアントがプロキシの背後にあり、DNSを介してこれらのIPを解決できない場合は、代わりにプロキシのアドレスを使用できます。または、Webアプリケーションがネットワーク経由ではなくfile：// URI [RFC8089]からロードされた場合、実装は既知のDNS名またはIPアドレスにフォールバックできます。

Once a suitable remote IP has been determined, the implementation can create a UDP socket, bind() it to the appropriate wildcard address, and then connect() to the remote IP. Generally, this results in the socket being assigned a local address based on the kernel routing table, without sending any packets over the network.

適切なリモートIPが決定されると、実装はUDPソケットを作成し、それを適切なワイルドカードアドレスにバインド（）してから、リモートIPにconnect（）することができます。通常、これにより、ネットワーク経由でパケットを送信せずに、カーネルルーティングテーブルに基づいてソケットにローカルアドレスが割り当てられます。

Finally, the socket can be queried using getsockname() or the equivalent to determine the appropriate local address.

最後に、getsockname（）または同等の機能を使用してソケットを照会し、適切なローカルアドレスを決定できます。

7. Application Guidance
7. アプリケーションガイダンス

The recommendations mentioned in this document may cause certain WebRTC applications to malfunction. In order to be robust in all scenarios, the following guidelines are provided for applications:

このドキュメントに記載されている推奨事項により、特定のWebRTCアプリケーションが誤動作する可能性があります。すべてのシナリオで堅牢にするために、アプリケーションに対して次のガイドラインが提供されています。

* Applications SHOULD deploy a TURN server with support for both UDP and TCP connections to the server. This ensures that connectivity can still be established, even when Mode 3 or 4 is in use, assuming the TURN server can be reached.

* アプリケーションは、サーバーへのUDP接続とTCP接続の両方をサポートするTURNサーバーをデプロイする必要があります。これにより、モード3または4が使用されている場合でも、TURNサーバーに到達できると仮定して、接続を確立できます。

* Applications SHOULD detect when they don't have access to the full set of ICE candidates by checking for the presence of host candidates. If no host candidates are present, Mode 3 or 4 is in use; this knowledge can be useful for diagnostic purposes.

* アプリケーションは、ホスト候補の存在をチェックすることにより、ICE候補のフルセットにアクセスできない場合を検出する必要があります。ホスト候補が存在しない場合、モード3または4が使用されています。この知識は、診断目的に役立ちます。

8. Security Considerations
8. セキュリティに関する考慮事項

This document describes several potential privacy and security concerns associated with WebRTC peer-to-peer connections and provides mechanisms and recommendations for WebRTC implementations to address these concerns.

このドキュメントでは、WebRTCピアツーピア接続に関連するいくつかの潜在的なプライバシーとセキュリティの懸念について説明し、これらの懸念に対処するためのWebRTC実装のメカニズムと推奨事項を提供します。
