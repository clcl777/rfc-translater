Abstract

概要

This document updates RFC 6353 ("Transport Layer Security (TLS) Transport Model for the Simple Network Management Protocol (SNMP)") to reflect changes necessary to support Transport Layer Security version 1.3 (TLS 1.3) and Datagram Transport Layer Security version 1.3 (DTLS 1.3), which are jointly known as "(D)TLS 1.3". This document is compatible with (D)TLS 1.2 and is intended to be compatible with future versions of SNMP and (D)TLS.

1. Introduction
1. はじめに

This document updates and clarifies how the rules of [RFC6353] apply when using Transport Layer Security (TLS) or Datagram Transport Layer Security (DTLS) versions later than 1.2. This document jointly refers to these two protocols as "(D)TLS". The update also emphasizes the requirement in [RFC8996] prohibiting the use of TLS versions prior to TLS 1.2 [RFC5246] when using SNMP. Although the text of this document specifically references SNMPv3 and (D)TLS 1.3, this document may be applicable to future versions of these protocols and is backwards compatible with (D)TLS 1.2.

このドキュメントは、輸送層セキュリティ（TLS）またはデータグラムトランスポートレイヤーセキュリティ（DTLS）バージョンを1.2以下に使用する場合の[RFC6353]のルールがどのように適用されるかを更新および明確にします。このドキュメントは、これら2つのプロトコルを「（d）TLS」と共同で言及しています。この更新は、SNMPを使用する場合のTLS 1.2 [RFC5246]の前にTLSバージョンの使用を禁止する[RFC8996]の要件を強調しています。このドキュメントのテキストは、SNMPv3および（d）TLS 1.3を特に参照していますが、このドキュメントはこれらのプロトコルの将来のバージョンに適用できる場合があり、（d）TLS 1.2と互換性があります。

1.2. Conventions
1.2. 規約

Within this document, the terms "TLS", "DTLS", and "(D)TLS" apply to all versions of the indicated protocols. The term "SNMP" means "SNMPv3" unless a specific version number is indicated. Specific version numbers are used when the text needs to emphasize version numbers.

このドキュメント内では、「TLS」、「DTLS」という用語、および「（d）TLS」という用語は、指定されたプロトコルのすべてのバージョンに適用されます。「SNMP」という用語は、特定のバージョン番号が表示されない限り、「SNMPV3」を意味します。テキストがバージョン番号を強調する必要があるときに特定のバージョン番号が使用されます。

For consistency with SNMP-related specifications, this document favors terminology as defined in [STD62], rather than favoring terminology that is consistent with non-SNMP specifications. "Authentication" in this document typically refers to the English meaning of "serving to prove the authenticity of" the message, not data source authentication or peer identity authentication. The terms "manager" and "agent" are not used in this document because, in the architecture defined in RFC 3411 [STD62], all SNMP entities have the capability of acting as manager, agent, or both, depending on the SNMP application types supported in the implementation. Where distinction is necessary, the application names of command generator, command responder, notification originator, notification receiver, and proxy forwarder are used. 2.1. TLSTM Fingerprint
2.1. TLSTM指紋

[RFC6353] defines the SnmpTLSFingerprint textual convention to include the one-octet TLS 1.2 hash algorithm identifier. This one-octet algorithm identifier is only applicable to (D)TLS protocol versions prior to 1.3. The TLS community does not plan to ever add additional values to the "TLS HashAlgorithm" registry [RFC5246], because some might incorrectly infer that using a new hash algorithm with TLS 1.2 would overcome the limitations of TLS 1.2. However, there is still a need within TLSTM to support new values as they are developed.

2.2. Security Level
2.2. セキュリティレベル

The architecture defined in RFC 3411 [STD62] recognizes three levels of security:

RFC 3411 [STD62]で定義されているアーキテクチャは、3つのレベルのセキュリティを認識しています。

* without authentication and without privacy (noAuthNoPriv)

* 認証がなく、プライバシーなし（noauthnopriv）

* with authentication but without privacy (authNoPriv)

* 認証付きが、プライバシーがない（authnopriv）

* with authentication and with privacy (authPriv)

* 認証とプライバシー付き（AuthPRIV）

Cipher suites for (D)TLS 1.3 defined in [RFC8446] provide both authentication and privacy. Cipher suites defined in [RFC9150] for (D)TLS 1.3 provide only authentication, without any privacy protection. Implementations MAY choose to force (D)TLS 1.3 to only allow cipher suites that provide both authentication and privacy.

[RFC8446]で定義されている（d）TLS 1.3の暗号スイートは、認証とプライバシーの両方を提供します。[RFC9150]で（d）TLS 1.3のために定義された暗号スイートは、プライバシー保護なしに認証のみを提供します。実装は、（d）TLS 1.3を強制的に強制することを選択できます。

2.3. (D)TLS Version
2.3. （d）TLSバージョン

[RFC6353] states that TLSTM clients and servers MUST NOT request, offer, or use SSL 2.0. [RFC8996] prohibits the use of (D)TLS versions prior to version 1.2. TLSTM MUST only be used with (D)TLS versions 1.2 and later.

[RFC6353]は、TLSTMクライアントとサーバーがSSL 2.0を要求、提供、または使用してはならないと述べています。[RFC8996]は、バージョン1.2より前に（d）TLSバージョンの使用を禁止しています。TLSTMは、（d）TLSバージョン1.2以降でのみ使用する必要があります。

3. Additional Rules for TLS 1.3
3. TLS 1.3の追加規則

This document specifies additional rules and clarifications for the use of TLS 1.3. These rules may additionally apply to future versions of TLS.

このドキュメントは、TLS 1.3を使用するための追加のルールと説明を指定します。これらのルールは、TLSの将来のバージョンにさらに適用される場合があります。

3.1. Zero Round-Trip Time Resumption (0-RTT)
3.1. ゼロラウンドトリップタイム再開（0-RTT）

TLS 1.3 implementations for SNMP MUST NOT enable the 0-RTT mode of session resumption (either sending or accepting) and MUST NOT automatically resend 0-RTT data if it is rejected by the server. 0-RTT is disallowed because there are no "safe" SNMP messages that, if replayed, will be guaranteed to cause no harm at the server side: all incoming notifications or command responses are meant to be acted upon only once. See Section 5 ("Security Considerations") for further details.

SNMPのTLS 1.3実装は、セッション再開の0-RTTモードを有効にしてはならず（送信または受け入れます）、サーバーによって拒否された場合、0-RTTデータを自動的に再送信してはなりません。0-RTTは、再生された場合、サーバー側に害を及ぼさないように保証される「安全な」SNMPメッセージがないため、禁止されています。詳細については、セクション5（「セキュリティ上の考慮事項」）を参照してください。

TLSTM clients and servers MUST NOT request, offer, or use the 0-RTT mode of TLS 1.3. [RFC8446] removed the renegotiation supported in TLS 1.2 [RFC5246]; for session resumption, it introduced a zero-RTT (0-RTT) mode, saving a round trip at connection setup at the cost of increased risk of replay attacks (it is possible for servers to guard against this attack by keeping track of all the messages received). [RFC8446] requires that a profile be written for any application that wants to use 0-RTT, specifying which messages are "safe to use" with this mode. Within SNMP, there are no messages that are "safe to use" with this mode.

TLSTMクライアントとサーバーは、TLS 1.3の0-RTTモードを要求、提供、または使用してはなりません。[RFC8446]は、TLS 1.2 [RFC5246]でサポートされている再交渉を除去しました。セッションの再開のために、ゼロRTT（0-RTT）モードを導入し、リプレイ攻撃のリスクを高めるコストで接続セットアップで往復を節約しました（すべてのすべてを追跡することにより、サーバーがこの攻撃を防ぐことができます受信したメッセージ）。[RFC8446]は、0-RTTを使用したいアプリケーションに対してプロファイルを記述する必要があり、このモードで「安全に使用できる」メッセージを指定します。SNMP内には、このモードで「安全に使用できる」メッセージはありません。

Renegotiation of sessions is not supported, as it is not supported by TLS 1.3. If a future version of TLS supports renegotiation, this RFC should be updated to indicate whether there are any additional requirements related to its use.

セッションの再交渉はサポートされていません。これは、TLS 1.3でサポートされていないためです。TLSの将来のバージョンが再交渉をサポートする場合、このRFCを更新して、その使用に関連する追加要件があるかどうかを示す必要があります。

3.2. TLS Cipher Suites, Extensions, and Protocol Invariants
3.2. TLS暗号スイート、エクステンション、およびプロトコル不変剤

Section 9 of [RFC8446] requires that, in the absence of application profiles, certain cipher suites, TLS extensions, and TLS protocol invariants be mandatory to implement. This document does not specify an application profile; hence, all the compliance requirements in [RFC8446] apply.

[RFC8446]のセクション9では、アプリケーションプロファイルがない場合、特定の暗号スイート、TLSエクステンション、およびTLSプロトコル不変剤を実装することが必須である必要があります。このドキュメントでは、アプリケーションプロファイルを指定しません。したがって、[RFC8446]のすべてのコンプライアンス要件が適用されます。

5. Security Considerations
5. セキュリティに関する考慮事項

This document updates a transport model that permits SNMP to utilize (D)TLS security services. The security threats and how the TLSTM mitigates these threats are covered throughout this document and in [RFC6353]. Security considerations for TLS are described in Section 10 and Appendix E of TLS 1.3 [RFC8446]. Security considerations for DTLS are described in Section 11 of DTLS 1.3 [RFC9147].

このドキュメントは、SNMPが（d）TLSセキュリティサービスを利用できるようにするトランスポートモデルを更新します。セキュリティの脅威とTLSTMがこれらの脅威をどのように軽減するかは、このドキュメント全体および[RFC6353]でカバーされています。TLSのセキュリティ上の考慮事項は、TLS 1.3 [RFC8446]のセクション10および付録Eで説明されています。DTLのセキュリティ上の考慮事項は、DTLS 1.3 [RFC9147]のセクション11で説明されています。

Implementations should consider the latest recommendations on the use of (DTLS), such as those documented in [RFC9325].

[RFC9325]で文書化されたものなど、実装は（DTL）の使用に関する最新の推奨事項を考慮する必要があります。

SNMP versions prior to SNMPv3 did not include adequate security. Even if the network itself is secure (for example, by using IPsec), there is no control as to who on the secure network is allowed to access and GET/SET (read/change/create/delete) the objects in this MIB module.

SNMPV3以前のSNMPバージョンには、適切なセキュリティは含まれていませんでした。ネットワーク自体が（たとえば、IPSECを使用して）安全である場合でも、このMIBモジュールのオブジェクトにセキュアネットワークのアクセスと取得/セット（読み取り/変更/作成/削除）が誰が許可されているかについての制御はありません。。

It is RECOMMENDED that only SNMPv3 messages using the Transport Security Model (TSM) or another secure-transport-aware security model be sent over the TLSTM transport.

TLSTMトランスポートを介して送信されるトランスポートセキュリティモデル（TSM）または別の安全な輸送対象セキュリティモデルを使用するSNMPV3メッセージのみをお勧めします。

6. IANA Considerations
6. IANAの考慮事項

IANA has created a new registry called "SNMP-TLSTM HashAlgorithms" within the "Structure of Management Information (SMI) Numbers (MIB Module Registrations)" group. The description of this registry is "iso.org.dod.internet.mgmt.mib-2.snmpTlstmMIB.snmpTlstmHashAlgorithms (1.3.6.1.2.1.198.4)".

IANAは、「管理情報の構造（SMI）番号（MIBモジュール登録）」グループ内で「SNMP-TLSTMハスハルゴリズム」と呼ばれる新しいレジストリを作成しました。このレジストリの説明は、「iso.org.dod.internet.mgmt.mib-2.snmptlstmmib.snmptlstmhashalgorithms（1.3.6.1.1.2.1.198.4）」です。

The registry has the following fields: Value, Description, Recommended, and References. The range of values is zero to 255, with initial assignments shown in Section 2.1. The "Recommended" column indicates "Y" for hashing algorithms that are Standards Track and are deemed to be acceptable for widely applicable current use and "N" for hashing algorithms that reflect meanings that are not recommended (e.g., they do not provide sufficient security for modern systems, they are not Standards Track, and they have limited applicability). A blank field indicates that no recommendation is made (e.g., because the value is unassigned or left for private use).

レジストリには、値、説明、推奨、参照の次のフィールドがあります。値の範囲はゼロから255で、セクション2.1に初期割り当てが表示されます。「推奨される」列は、標準の追跡であり、広く適用可能な現在の使用に受け入れられているとみなされるハッシュアルゴリズムの「Y」を示します。最新のシステムの場合、それらは標準の追跡ではなく、適用性が限られています）。空白のフィールドは、推奨が行われていないことを示します（たとえば、値が割り当てられていないか、プライベートな使用のために残されているため）。

This registry is expected to be updated infrequently; as such, its values are limited to one octet.

このレジストリはまれに更新されると予想されます。そのため、その値は1つのオクテットに制限されています。

The policy for updates to the "SNMP-TLSTM HashAlgorithms" registry is Expert Review [RFC8126]. Registry requests should be sent to the <mailto:snmp-tlstm-reg-review@ietf.org> mailing list. Registration requests sent to the mailing list for review SHOULD use an appropriate subject (e.g., 'Request to register value in "SNMP-TLSTM HashAlgorithms" registry'). In addition, designated experts should consult with the <mailto:tls-reg-review@ietf.org> mailing list to make sure that any new hash algorithms are considered for inclusion in this registry.

「SNMP-TLSTMハスハルゴリズム」レジストリの更新に関するポリシーは、専門家レビュー[RFC8126]です。レジストリリクエストは、<mailto：snmp-tlstm-reg-review@ietf.org>メーリングリストに送信する必要があります。レビューのためにメーリングリストに送信された登録リクエストは、適切な科目を使用する必要があります（たとえば、「snmp-tlstm hashalgorithms "レジストリ」に値を登録するリクエスト）。さらに、指定された専門家は、<mailto：tls-reg-review@ietf.org>メーリングリストに相談して、新しいハッシュアルゴリズムがこのレジストリに含めることを考慮していることを確認する必要があります。

Designated experts SHOULD ascertain the existence of suitable documentation that defines a hash algorithm and SHOULD also verify that the request does not conflict with or duplicate other entries in the registry. Only publicly available specifications that represent current industry-accepted practices should receive an assignment of "Y" in the "Recommended" column; all other specific assignments in the registry should receive an assignment of "N".
