1. The Internet-Standard Management Framework
1. インターネット標準の管理フレームワーク

For a detailed overview of the documents that describe the current Internet-Standard Management Framework, please refer to section 7 of RFC 3410 [RFC3410].

現在のインターネット標準管理フレームワークを説明するドキュメントの詳細な概要については、RFC 3410 [RFC3410]のセクション7を参照してください。

Managed objects are accessed via a virtual information store, termed the Management Information Base or MIB. MIB objects are generally accessed through the Simple Network Management Protocol (SNMP). Objects in the MIB are defined using the mechanisms defined in the Structure of Management Information (SMI). This memo specifies a MIB module that is compliant to the SMIv2, which is described in STD 58, RFC 2578 [RFC2578], STD 58, RFC 2579 [RFC2579] and STD 58, RFC 2580 [RFC2580].

管理対象オブジェクトは、管理情報ベースまたはMIBと呼ばれる仮想情報ストアを介してアクセスされます。 MIBオブジェクトには、通常、簡易ネットワーク管理プロトコル（SNMP）を介してアクセスします。 MIB内のオブジェクトは、管理情報の構造（SMI）で定義されたメカニズムを使用して定義されます。このメモは、SMIv2に準拠するMIBモジュールを指定します。これは、STD 58、RFC 2578 [RFC2578]、STD 58、RFC 2579 [RFC2579]およびSTD 58、RFC 2580 [RFC2580]で説明されています。

3. Relationship to Other MIB Modules
3. 他のMIBモジュールとの関係

The iSCSI MIB module is normally layered between the SCSI MIB module [RFC4455] and the TCP MIB module [RFC4022], and it makes use of the IP Storage (IPS) Identity Authentication MIB module [RFC4545]. Here is how these modules are related:

iSCSI MIBモジュールは通常、SCSI MIBモジュール[RFC4455]とTCP MIBモジュール[RFC4022]の間にあり、IPストレージ（IPS）ID認証MIBモジュール[RFC4545]を利用しています。これらのモジュールの関係は次のとおりです。

SCSI MIB Within systems where a SCSI layer is present, each iscsiNode, whether it has an initiator role, target role, or both, is related to one SCSI device within the SCSI MIB module. In this case, the iscsiNodeTransportType attribute points to the SCSI transport object within the SCSI MIB module, which in turn contains an attribute that points back to the iscsiNode. In this way, a management station can navigate between the two MIB modules. In systems where a SCSI layer is not present, such as within an iSCSI proxy device, the iscsiNodeTransportType attribute points to the appropriate corresponding object within the appropriate MIB or is left blank.

SCSI MIB SCSIレイヤーが存在するシステム内では、各iscsiNodeは、イニシエーターの役割、ターゲットの役割、またはその両方を持つかどうかに関係なく、SCSI MIBモジュール内の1つのSCSIデバイスに関連付けられています。この場合、iscsiNodeTransportType属性は、SCSI MIBモジュール内のSCSIトランスポートオブジェクトを指します。これには、iscsiNodeを指す属性が含まれています。このようにして、管理ステーションは2つのMIBモジュール間を移動できます。 iSCSIプロキシデバイス内など、SCSIレイヤーが存在しないシステムでは、iscsiNodeTransportType属性は、適切なMIB内の適切な対応するオブジェクトを指すか、空白のままです。

TCP MIB Each iSCSI connection is related to one transport-level connection. Currently, iSCSI uses only TCP; the iSCSI connection is related to a TCP connection using its normal (protocol, source address, source port, destination address, destination port) 5-tuple.

TCP MIB各iSCSI接続は、1つのトランスポートレベル接続に関連しています。現在、iSCSIはTCPのみを使用しています。 iSCSI接続は、通常の（プロトコル、送信元アドレス、送信元ポート、宛先アドレス、宛先ポート）5タプルを使用するTCP接続に関連しています。

AUTH MIB Each iSCSI node that serves a target role can have a list of authorized initiators. Each of the entries in this list points to an identity within the IPS Identity Authentication MIB module that will be allowed to access the target. iSCSI nodes that serve in an initiator role can also have a list of authorized targets. Each of the entries in this list points to an identity within the IPS-AUTH MIB module to which the initiator should attempt to establish sessions. The IPS-AUTH MIB module includes information used to identify initiators and targets by their iSCSI name, IP address, and/or credentials.

AUTH MIBターゲットの役割を担う各iSCSIノードは、許可されたイニシエーターのリストを持つことができます。このリストの各エントリは、ターゲットへのアクセスを許可されるIPS ID認証MIBモジュール内のIDを指します。イニシエーターの役割を担うiSCSIノードは、許可されたターゲットのリストを持つこともできます。このリストの各エントリは、イニシエーターがセッションを確立しようとするIPS-AUTH MIBモジュール内のIDを指します。 IPS-AUTH MIBモジュールには、iSCSI名、IPアドレス、資格情報によってイニシエーターとターゲットを識別するために使用される情報が含まれています。

This MIB module imports objects from RFCs 2578 [RFC2578], 2579 [RFC2579], 2580 [RFC2580], and 3411 [RFC3411]. It also imports textual conventions from the INET-ADDRESS-MIB [RFC4001].

このMIBモジュールは、RFC 2578 [RFC2578]、2579 [RFC2579]、2580 [RFC2580]、および3411 [RFC3411]からオブジェクトをインポートします。また、INET-ADDRESS-MIB [RFC4001]からテキスト表記法をインポートします。

4. Relationship to SNMP Contexts
4. SNMPコンテキストとの関係

Each non-scalar object in the iSCSI MIB module is indexed first by an iSCSI instance. Each instance is a collection of nodes, portals, sessions, etc., that can define a physical or virtual partitioning of an iSCSI-capable device. The use of an instance works well with partitionable or hierarchical storage devices and fits in logically with other management schemes. Instances do not replace SNMP contexts; however, they do provide a very simple way to assign a virtual or physical partition of a device to one or more SNMP contexts, without having to do so for each individual node, portal, and session row.

iSCSI MIBモジュール内の各非スカラーオブジェクトには、最初にiSCSIインスタンスによってインデックスが付けられます。各インスタンスは、ノード、ポータル、セッションなどのコレクションであり、iSCSI対応デバイスの物理または仮想パーティションを定義できます。インスタンスの使用は、パーティション化または階層化されたストレージデバイスでうまく機能し、他の管理スキーマと論理的に適合します。インスタンスはSNMPコンテキストを置き換えません。ただし、デバイスの仮想パーティションまたは物理パーティションを1つ以上のSNMPコンテキストに割り当てる非常に簡単な方法を提供します。個々のノード、ポータル、およびセッション行ごとに割り当てる必要はありません。

6. Discussion
6. 討論

This MIB module structure supplies configuration, fault, and statistics information for iSCSI devices [RFC7143]. It is structured around the well-known iSCSI objects, such as targets, initiators, sessions, connections, and the like.

このMIBモジュール構造は、iSCSIデバイスの構成、障害、および統計情報を提供します[RFC7143]。これは、ターゲット、イニシエーター、セッション、接続など、よく知られたiSCSIオブジェクトを中心に構成されています。

This MIB module may also be used to configure access to iSCSI targets, by creating iSCSI portals and authorization list entries.

このMIBモジュールは、iSCSIポータルと承認リストエントリを作成することにより、iSCSIターゲットへのアクセスを構成するためにも使用できます。

It is worthwhile to note that this is an iSCSI MIB module and as such reflects only iSCSI objects. This module does not contain information about the SCSI-layer attributes of a device. If a SCSI layer is present, the SCSI MIB module [RFC4455] may be used to manage SCSI information for a device.

これはiSCSI MIBモジュールであるため、iSCSIオブジェクトのみを反映していることに注意してください。このモジュールには、デバイスのSCSIレイヤー属性に関する情報は含まれていません。 SCSI層が存在する場合、SCSI MIBモジュール[RFC4455]を使用して、デバイスのSCSI情報を管理できます。

The iSCSI MIB module consists of several "objects", each of which is represented by one or more tables. This section contains a brief description of the object hierarchy and a description of each object, followed by a discussion of the actual table structure within the objects.

iSCSI MIBモジュールは複数の「オブジェクト」で構成され、それぞれが1つ以上のテーブルで表されます。このセクションでは、オブジェクト階層の簡単な説明と各オブジェクトの説明に続いて、オブジェクト内の実際のテーブル構造について説明します。

6.1. iSCSI MIB Object Model
6.1. iSCSI MIBオブジェクトモデル

The top-level object in this structure is the iSCSI instance, which "contains" all of the other objects.

この構造の最上位オブジェクトは、他のすべてのオブジェクトを「含む」iSCSIインスタンスです。 

      iscsiInstance
         -- A distinct iSCSI entity within the managed system.
         iscsiPortal
            -- An IP address used by this instance.
            iscsiTargetPortal
               -- Contains portal information relevant when the portal
               -- is used to listen for connections to its targets.
            iscsiInitiatorPortal
               -- Contains portal information relevant when the portal
               -- is used to initiate connections to other targets.
         iscsiNode
            -- An iSCSI node can act as an initiator, a target, or both.
            -- Contains generic (non-role-specific) information.
            iscsiTarget
               -- Target-specific iSCSI node information.
               iscsiTgtAuth
                  -- A list of initiator identities that are allowed
                  -- access to this target.
            iscsiInitiator
               -- Initiator-specific iSCSI node information.
               iscsiIntrAuth
                  -- A list of target identities to which this initiator
                  -- is configured to establish sessions.
            iscsiSession
               -- An active iSCSI session between an initiator and
               -- target.  The session's direction may be Inbound
               -- (an outside initiator to the target represented by
               -- this node) or Outbound (the initiator represented by
               -- this node to an outside target).
               iscsiConnection
                  -- An active TCP connection within an iSCSI session.

An iSCSI node can be an initiator, a target, or both. The iSCSI node's portals may be used to initiate connections (initiator) or listen for connections (target), depending on whether the iSCSI node is acting as an initiator or target. The iSCSI MIB module assumes that any target may be accessed via any portal that can take on a target role, although other access controls not reflected in the module might limit this.

iSCSIノードは、イニシエーター、ターゲット、またはその両方にすることができます。 iSCSIノードのポータルは、iSCSIノードがイニシエーターとターゲットのどちらとして機能しているかに応じて、接続の開始（イニシエーター）または接続のリッスン（ターゲット）に使用できます。 iSCSI MIBモジュールは、モジュールに反映されていない他のアクセス制御がこれを制限する可能性がありますが、ターゲットの役割を引き受けることができる任意のポータルを介して任意のターゲットにアクセスできると想定しています。

6.2. iSCSI MIB Table Structure
6.2. iSCSI MIBテーブルの構造

Each iSCSI object exports one or more tables: an attributes table, and zero or more statistics tables, which augment the attributes table. Since iSCSI is an evolving standard, it is much cleaner to provide statistics and attributes as separate tables, allowing attributes and statistics to be added independently. In a few cases, there are multiple categories of statistics that will likely grow; in this case, an object will contain multiple statistics tables.

各iSCSIオブジェクトは、1つまたは複数のテーブルをエクスポートします。属性テーブルと、属性テーブルを補強する0個以上の統計テーブルです。 iSCSIは進化する標準であるため、統計と属性を別々のテーブルとして提供する方がはるかにクリーンであり、属性と統計を個別に追加できます。まれに、増加する可能性のある統計のカテゴリが複数ある場合があります。この場合、オブジェクトには複数の統計テーブルが含まれます。

        iscsiInitiatorPortal

Note that this module does not attempt to count everything that could be counted; it is designed to include only those counters that would be useful for identifying performance, security, and fault problems from a management station.

このモジュールは、数えることができるすべてのものを数えることを試みないことに注意してください。管理ステーションからパフォーマンス、セキュリティ、および障害の問題を特定するのに役立つカウンタのみが含まれるように設計されています。

6.3. iscsiInstance
6.3. iscsiInstance

The iscsiInstanceAttributesTable is the primary table of the iSCSI MIB module. Every table entry in this module is "owned" by exactly one iSCSI instance; all other table entries in the module include this table's index as their primary index.

iscsiInstanceAttributesTableは、iSCSI MIBモジュールのプライマリテーブルです。このモジュールのすべてのテーブルエントリは、1つのiSCSIインスタンスによって「所有」されています。モジュール内の他のすべてのテーブルエントリには、このテーブルのインデックスがプライマリインデックスとして含まれています。

Most implementations will include just one iSCSI instance row in this table. However, this table exists to allow for multiple virtual instances. For example, many IP routing products now allow multiple virtual routers. The iSCSI MIB module has the same premise; a large system could be "partitioned" into multiple, distinct virtual systems.

ほとんどの実装では、このテーブルにiSCSIインスタンス行が1つだけ含まれます。ただし、このテーブルは複数の仮想インスタンスを可能にするために存在しています。たとえば、多くのIPルーティング製品で、複数の仮想ルーターを使用できるようになりました。 iSCSI MIBモジュールも同じ前提を持っています。大規模なシステムは、複数の個別の仮想システムに「分割」できます。

This also allows a single SNMP agent to proxy for multiple subsystems, perhaps a set of stackable devices, each of which has one or even more instances.

これにより、単一のSNMPエージェントが複数のサブシステム（おそらく、それぞれが1つ以上のインスタンスを持つスタック可能なデバイスのセット）をプロキシすることもできます。

The instance attributes include the iSCSI vendor and version, as well as information on the last target or initiator at the other end of a session that caused a session failure.

インスタンスの属性には、iSCSIベンダーとバージョン、およびセッション障害の原因となったセッションの反対側の最後のターゲットまたはイニシエーターに関する情報が含まれます。

The iscsiInstanceSsnErrorStatsTable augments the attributes table and provides statistics on session failures due to digest, connection, or iSCSI format errors.

iscsiInstanceSsnErrorStatsTableは、属性テーブルを拡張し、ダイジェスト、接続、またはiSCSIフォーマットエラーによるセッション障害の統計を提供します。

6.4. iscsiPortal
6.4. iscsiPortal

The iscsiPortalAttributesTable lists iSCSI portals that can be used to listen for connections to targets, to initiate connections to other targets, or to do both.

iscsiPortalAttributesTableは、ターゲットへの接続をリッスンする、他のターゲットへの接続を開始する、またはその両方を行うために使用できるiSCSIポータルをリストします。

Each row in the table includes an IP address (either v4 or v6), and a transport protocol (currently only TCP is defined). Each portal may have additional attributes, depending on whether it is an initiator portal, a target portal, or both. Initiator portals also have portal tags; these are placed in corresponding rows in the iscsiIntrPortalAttributesTable. Target portals have both portal tags and ports (e.g., TCP listen ports if the transport protocol is TCP); these are placed in rows in the iscsiTgtPortalAttributesTable.

テーブルの各行には、IPアドレス（v4またはv6のいずれか）、およびトランスポートプロトコル（現在はTCPのみが定義されています）が含まれています。イニシエーターポータル、ターゲットポータル、またはその両方に応じて、各ポータルには追加の属性があります。イニシエーターポータルにもポータルタグがあります。これらは、iscsiIntrPortalAttributesTableの対応する行に配置されます。ターゲットポータルには、ポータルタグとポートの両方（トランスポートプロトコルがTCPの場合はTCPリスンポートなど）があります。これらは、iscsiTgtPortalAttributesTableの行に配置されます。

Portal rows, along with their initiator and target portal counterparts, may be created and destroyed through this MIB module by a management station. Rows in the initiator and target portal tables are created and destroyed automatically by the agent when a row is created or destroyed in the iscsiPortalAttributesTable or when the value of iscsiPortalRoles changes. Attributes in these tables may then be modified by the management station if the agent implementation allows.

ポータル行は、イニシエーターとターゲットポータルの対応物とともに、管理ステーションによってこのMIBモジュールを介して作成および破棄できます。イニシエーターおよびターゲットポータルテーブルの行は、iscsiPortalAttributesTableで行が作成または破棄されたとき、またはiscsiPortalRolesの値が変更されたときに、エージェントによって自動的に作成および破棄されます。これらのテーブルの属性は、エージェントの実装で許可されている場合、管理ステーションによって変更される可能性があります。

When created by a management station, the iscsiPortalRoles attribute is used to control row creation in the initiator and target portal tables. Creating a row with the targetTypePortal bit set in iscsiPortalRoles will cause the implementation to start listening for iSCSI connections on the portal. Creating a row with the initiatorTypePortal bit set in iscsiPortalRoles will not necessarily cause connections to be established; it is left to the implementation whether and when to make use of the portal. Both bits may be set if the portal is to be used by both initiator and target nodes.

iscsiPortalRoles属性は、管理ステーションによって作成されると、イニシエーターおよびターゲットポータルテーブルでの行の作成を制御するために使用されます。 iscsiPortalRolesでtargetTypePortalビットが設定された行を作成すると、実装はポータルでiSCSI接続の待機を開始します。 iscsiPortalRolesで発信元タイプポータルビットを設定して行を作成しても、必ずしも接続が確立されるとは限りません。ポータルを利用するかどうか、いつ利用するかは実装に任されています。ポータルがイニシエーターノードとターゲットノードの両方で使用される場合は、両方のビットを設定できます。

When deleting a row in the iscsiPortalAttibutesTable, all connections associated with that row are terminated. The implementation may either terminate the connection immediately or request a clean shutdown as specified in [RFC7143]. An outbound connection (when an iscsiInitiatorPortal is deleted) matches the portal if its iscsiCxnLocalAddr matches the iscsiPortalAddr. An inbound connection (when an iscsiTargetPortal is deleted) matches the portal if its iscsiCxnLocalAddr matches the iscsiPortalAddr and if its iscsiCxnLocalPort matches the iscsiTargetPortalPort.

iscsiPortalAttibutesTableの行を削除すると、その行に関連付けられているすべての接続が終了します。 [RFC7143]で指定されているように、実装は接続をすぐに終了するか、クリーンシャットダウンを要求することができます。 iscsiCxnLocalAddrがiscsiPortalAddrと一致する場合、発信接続（iscsiInitiatorPortalが削除されたとき）はポータルと一致します。インバウンド接続（iscsiTargetPortalが削除された場合）は、そのiscsiCxnLocalAddrがiscsiPortalAddrと一致し、iscsiCxnLocalPortがiscsiTargetPortalPortと一致する場合、ポータルと一致します。

Individual objects within a row in this table may not be modified while the row is active. For instance, changing the IP address of a portal requires that the rows associated with the old IP address be deleted and that new rows be created (in either order).

このテーブルの行内の個々のオブジェクトは、行がアクティブな間は変更できません。たとえば、ポータルのIPアドレスを変更するには、古いIPアドレスに関連付けられている行を削除し、新しい行を（いずれかの順序で）作成する必要があります。

6.5. iscsiTargetPortal
6.5. iscsiTargetPortal

The iscsiTgtPortalAttributesTable contains target-specific attributes for iSCSI portals. Rows in this table use the same indices as their corresponding rows in the iscsiPortalAttributesTable, with the addition of iscsiNodeIndex.

iscsiTgtPortalAttributesTableには、iSCSIポータルのターゲット固有の属性が含まれています。このテーブルの行は、iscsiPortalAttributesTableの対応する行と同じインデックスを使用し、iscsiNodeIndexを追加しています。

Rows in this table are created when the targetTypePortal bit is set in the iscsiPortalRoles attribute of the corresponding iscsiPortalAttributesEntry; they are destroyed when this bit is cleared.

この表の行は、対応するiscsiPortalAttributesEntryのiscsiPortalRoles属性でtargetTypePortalビットが設定されている場合に作成されます。このビットがクリアされると、それらは破棄されます。

This table contains the TCP (or other protocol) port on which the socket is listening for incoming connections. It also includes a portal group aggregation tag; iSCSI target portals that are within this instance and share the same tag can contain connections within the same session.

このテーブルには、ソケットが着信接続をリッスンしているTCP（または他のプロトコル）ポートが含まれています。また、ポータルグループ集約タグも含まれています。このインスタンス内にあり、同じタグを共有するiSCSIターゲットポータルには、同じセッション内の接続を含めることができます。

This table will be empty for iSCSI instances that contain only initiators (such as iSCSI host driver implementations).

イニシエーターのみを含むiSCSIインスタンス（iSCSIホストドライバーの実装など）の場合、このテーブルは空になります。

Many implementations use the same Target Portal Group Tag and protocol port for all nodes accessed via a portal. These implementations will create a single row in the iscsiTgtPortalAttributeTable, with an iscsiNodeIndex of zero.

多くの実装では、ポータルを介してアクセスされるすべてのノードに同じターゲットポータルグループタグとプロトコルポートを使用します。これらの実装では、iscsiNodeIndexがゼロのiscsiTgtPortalAttributeTableに単一の行が作成されます。

Other implementations do not use the same tag and/or port for all nodes; these implementations will create a row in this table for each (portal, node) tuple, using iscsiNodeIndex to designate the node for this portal tag and port.

他の実装では、すべてのノードに同じタグやポートを使用していません。これらの実装は、iscsiNodeIndexを使用してこのポータルタグとポートのノードを指定し、（テーブル、ノード）タプルごとにこのテーブルに行を作成します。

6.6. iscsiInitiatorPortal
6.6. iscsiイニシエーターポータル

The iscsiIntrPortalAttributesTable contains initiator-specific objects for iSCSI portals. Rows in this table use the same indices as their corresponding entries in the iscsiPortalAttributesTable. A row in this table is created when the initiatorTypePortal bit is set in the iscsiPortalRoles attribute; it is destroyed when this bit is cleared.

iscsiIntrPortalAttributesTableには、iSCSIポータルのイニシエーター固有のオブジェクトが含まれています。このテーブルの行は、iscsiPortalAttributesTableの対応するエントリと同じインデックスを使用します。このテーブルの行は、iscsiPortalRoles属性でinitiatorTypePortalビットが設定されている場合に作成されます。このビットがクリアされると破棄されます。

Each row in this table contains a portal group aggregation tag, indicating which portals an initiator may use together within a multiple-connection session.

この表の各行にはポータルグループ集約タグが含まれており、イニシエーターが複数接続セッション内で一緒に使用できるポータルを示します。

This table will be empty for iSCSI instances that contain only targets (such as most iSCSI devices).

ターゲットのみを含むiSCSIインスタンス（ほとんどのiSCSIデバイスなど）の場合、このテーブルは空になります。

Many implementations use the same initiator tag for all nodes accessing targets via a given portal. These implementations will create a single row in iscsiIntrPortalAttributeTable, with an iscsiNodeIndex of zero.

多くの実装では、特定のポータルを介してターゲットにアクセスするすべてのノードに同じイニシエータータグを使用します。これらの実装はiscsiIntrPortalAttributeTableに単一の行を作成し、iscsiNodeIndexはゼロです。

Other implementations do not use the same tag and/or port for all nodes; these implementations will create a row in this table for each (portal, node) tuple, using iscsiNodeIndex to designate the node for this portal tag and port.

他の実装では、すべてのノードに同じタグやポートを使用していません。これらの実装は、iscsiNodeIndexを使用してこのポータルタグとポートのノードを指定し、（テーブル、ノード）タプルごとにこのテーブルに行を作成します。

6.7. iscsiNode
6.7. iscsiNode

The iscsiNodeAttributesTable contains a list of iSCSI nodes, each of which may have an initiator role, a target role, or both.

iscsiNodeAttributesTableには、iSCSIノードのリストが含まれています。各iSCSIノードには、イニシエーターの役割、ターゲットの役割、またはその両方が含まれる場合があります。

This table contains the node's attributes that are common to both roles, such as its iSCSI name and alias string. Attributes specific to initiators or targets are available in the iscsiTarget and iscsiInitiator objects. Each row in this table that can fulfill a target role has a corresponding row in the iscsiTarget table; each entry that fulfills an initiator role has a row in the iscsiInitiator table. Nodes such as copy managers that can take on both roles have a corresponding row in each table.

このテーブルには、iSCSI名やエイリアス文字列など、両方の役割に共通のノードの属性が含まれています。イニシエーターまたはターゲットに固有の属性は、iscsiTargetおよびiscsiInitiatorオブジェクトで使用できます。ターゲットの役割を果たし得るこのテーブルの各行には、iscsiTargetテーブルに対応する行があります。イニシエーターの役割を満たす各エントリーは、iscsiInitiatorテーブルに行があります。両方の役割を担うことができるコピーマネージャなどのノードには、各テーブルに対応する行があります。

This table also contains the login negotiations preferences for this node. These objects indicate the values this node will offer or prefer in the operational negotiation phase of the login process.

このテーブルには、このノードのログインネゴシエーション設定も含まれています。これらのオブジェクトは、ログインプロセスの操作ネゴシエーションフェーズでこのノードが提供または優先する値を示します。

For most implementations, each entry in the table also contains a RowPointer to the transport table entry in the SCSI MIB module that this iSCSI node represents. For implementations without a standard SCSI layer above iSCSI, such as an iSCSI proxy or gateway, this RowPointer can point to a row in an implementation-specific table that this iSCSI node represents.

ほとんどの実装では、テーブルの各エントリには、このiSCSIノードが表すSCSI MIBモジュールのトランスポートテーブルエントリへのRowPointerも含まれています。 iSCSIプロキシやゲートウェイなど、iSCSIの上に標準のSCSIレイヤーがない実装の場合、このRowPointerは、このiSCSIノードが表す実装固有のテーブルの行を指すことができます。

6.8. iscsiTarget
6.8. iscsiTarget

The iscsiTargetAttributesTable contains target-specific attributes for iSCSI nodes. Each entry in this table uses the same index values as its corresponding iscsiNode entry.

iscsiTargetAttributesTableには、iSCSIノードのターゲット固有の属性が含まれています。このテーブルの各エントリは、対応するiscsiNodeエントリと同じインデックス値を使用します。

This table contains attributes used to indicate the last failure that was (or should have been) sent as a notification.

このテーブルには、通知として送信された（または送信されるべきだった）最後の障害を示すために使用される属性が含まれています。

This table is augmented by the iscsiTargetLoginStatsTable and the iscsiTargetLogoutStatsTable, which count the numbers of normal and abnormal logins and logouts to this target.

このテーブルは、iscsiTargetLoginStatsTableおよびiscsiTargetLogoutStatsTableによって拡張されます。iscsiTargetLogoutStatsTableは、このターゲットへの正常および異常なログインとログアウトの数をカウントします。

6.9. iscsiTgtAuthorization
6.9. iscsi Tgt Authorization

The iscsiTgtAuthAttributesTable contains an entry for each initiator identifier that will be allowed to access the target under which it appears. Each entry contains a RowPointer to a user identity in the IPS Authorization MIB module, which contains the name, address, and credential information necessary to authenticate the initiator.

iscsiTgtAuthAttributesTableには、それが表示されているターゲットへのアクセスが許可される各イニシエーターIDのエントリーが含まれています。各エントリには、IPS認証MIBモジュールのユーザーIDへのRowPointerが含まれています。これには、イニシエーターの認証に必要な名前、アドレス、および資格情報が含まれています。

6.10. iscsiInitiator
6.10. iscsiInitiator

The iscsiInitiatorAttributesTable contains a list of initiator-specific attributes for iSCSI nodes. Each entry in this table uses the same index values as its corresponding iscsiNode entry.

iscsiInitiatorAttributesTableには、iSCSIノードのイニシエーター固有の属性のリストが含まれています。このテーブルの各エントリは、対応するiscsiNodeエントリと同じインデックス値を使用します。

Most implementations will include a single entry in this table, regardless of the number of physical interfaces the initiator may use.

イニシエーターが使用できる物理インターフェイスの数に関係なく、ほとんどの実装では、このテーブルに1つのエントリが含まれます。

This table is augmented by the iscsiInitiatorLoginStatsTable and the iscsiInitiatorLogoutStatsTable, which count the numbers of normal and abnormal logins and logouts from this initiator.

このテーブルは、iscsiInitiatorLoginStatsTableおよびiscsiInitiatorLogoutStatsTableによって拡張されます。iscsiInitiatorLogoutStatsTableは、このイニシエーターからの正常および異常なログインとログアウトの数をカウントします。

6.11. iscsiIntrAuthorization
6.11. iscsiIntrAuthorization

The iscsiIntrAuthAttributesTable contains an entry for each target identifier to which the initiator is configured to establish a session.

iscsiIntrAuthAttributesTableには、イニシエーターがセッションを確立するように構成されている各ターゲットIDのエントリーが含まれています。

Each entry contains a RowPointer to a user identity in the IPS Authorization MIB module, which contains the name, address, and credential information necessary to identify (for discovery purposes) and authenticate the target.

各エントリには、IPS認可MIBモジュールのユーザーIDへのRowPointerが含まれています。これには、ターゲットの識別（検出目的）および認証に必要な名前、アドレス、および資格情報が含まれています。

6.12. iscsiSession
6.12. iscsiSession

The iscsiSessionAttributesTable contains a set of rows that list the sessions known to exist locally for each node in each iSCSI instance.

iscsiSessionAttributesTableには、各iSCSIインスタンスの各ノードのローカルに存在することがわかっているセッションをリストする一連の行が含まれています。

The session type for each session indicates whether the session is used for normal SCSI commands or for discovery using the SendTargets text command. Discovery sessions that do not belong to any particular node have a node index attribute of zero.

各セッションのセッションタイプは、セッションが通常のSCSIコマンドに使用されるか、SendTargetsテキストコマンドを使用した検出に使用されるかを示します。特定のノードに属していないディスカバリセッションのノードインデックス属性はゼロです。

The session direction for each session indicates whether it is an Inbound session or an Outbound session. Inbound sessions are from some other initiator to the target node under which the session appears. Outbound sessions are from the initiator node under which the session appears to a target outside this iSCSI instance.

各セッションのセッション方向は、セッションがインバウンドセッションかアウトバウンドセッションかを示します。インバウンドセッションは、他のイニシエーターから、セッションが表示されるターゲットノードへのセッションです。発信セッションは、セッションがこのiSCSIインスタンスの外部のターゲットに表示されるイニシエーターノードからのものです。

Many attributes may be negotiated when starting an iSCSI session. Most of these attributes are included in the session object.

iSCSIセッションの開始時に、多くの属性がネゴシエートされる場合があります。これらの属性のほとんどは、セッションオブジェクトに含まれています。

Some attributes, such as the integrity and authentication schemes, have some standard values that can be extended by vendors to include their own schemes. These contain an object identifier, rather than the expected enumerated type, to allow these values to be extended by other MIB modules, such as an enterprise MIB module.

整合性や認証スキームなどの一部の属性には、ベンダーが独自のスキームを含めるように拡張できるいくつかの標準値があります。これらには、予期される列挙型ではなくオブジェクト識別子が含まれ、これらの値をエンタープライズMIBモジュールなどの他のMIBモジュールで拡張できるようにします。

The iscsiSessionStatsTable includes statistics related to performance; it counts iSCSI data bytes and PDUs.

iscsiSessionStatsTableには、パフォーマンスに関連する統計が含まれています。 iSCSIデータバイトとPDUをカウントします。

For implementations that support error recovery without terminating a session, the iscsiSessionCxnErrorStatsTable contains counters for the numbers of digest and connection errors that have occurred within the session.

セッションを終了せずにエラー回復をサポートする実装の場合、iscsiSessionCxnErrorStatsTableには、セッション内で発生したダイジェストおよび接続エラーの数のカウンターが含まれます。

6.13. iscsiConnection
6.13. iscsiConnection

The iscsiConnectionAttributesTable contains a list of active connections within each session. It contains the IP addresses and TCP (or other protocol) ports of both the local and remote sides of the connection. These may be used to locate other connection-related information and statistics in the TCP MIB module [RFC4022].

iscsiConnectionAttributesTableには、各セッション内のアクティブな接続のリストが含まれています。これには、接続のローカル側とリモート側の両方のIPアドレスとTCP（または他のプロトコル）ポートが含まれています。これらは、TCP MIBモジュール[RFC4022]で他の接続関連の情報と統計を見つけるために使用できます。

The attributes table also contains a connection state. This state is not meant to directly map to the state tables included within the iSCSI specification; they are meant to be simplified, higher-level definitions of connection state that provide information more useful to a user or network manager.

属性テーブルには、接続状態も含まれています。この状態は、iSCSI仕様に含まれる状態テーブルに直接マップすることを意図したものではありません。これらは、ユーザーまたはネットワーク管理者にとってより有用な情報を提供する、接続状態の簡略化された高レベルの定義を意図しています。

No statistics are kept for connections.

接続に関する統計は保持されません。

6.14. IP Addresses and TCP Port Numbers
6.14. IPアドレスとTCPポート番号

The IP addresses in this module are represented by two attributes, one of type InetAddressType, and the other of type InetAddress. These are taken from [RFC4001], which specifies how to support addresses that may be either IPv4 or IPv6.

このモジュールのIPアドレスは、2つの属性で表されます。1つはタイプInetAddressType、もう1つはタイプInetAddressです。これらは[RFC4001]から取得され、IPv4またはIPv6のいずれかであるアドレスをサポートする方法を指定します。

The TCP port numbers that appear in a few of the structures are described as simply port numbers, with a protocol attribute indicating whether they are TCP ports or something else. This will allow the module to be compatible with iSCSI over transports other than TCP in the future.

いくつかの構造に表示されるTCPポート番号は、単にポート番号として記述されており、プロトコル属性は、それらがTCPポートであるか他のものであるかを示します。これにより、将来的には、TCP以外のトランスポートを介してモジュールをiSCSIと互換性を持つことができます。

6.15. Descriptors: Using OIDs in Place of Enumerated Types
6.15. 記述子：列挙型の代わりにOIDを使用する

The iSCSI MIB module has a few attributes, namely, the digest method attributes, where an enumerated type would work well, except that an implementation may need to extend the attribute and add types of its own. To make this work, this MIB module defines a set of object identities within the iscsiDescriptors subtree. Each of these object identities is basically an enumerated type.

iSCSI MIBモジュールにはいくつかの属性、つまりダイジェストメソッド属性があります。実装では、属性を拡張して独自のタイプを追加する必要がある場合を除いて、列挙型が適切に機能します。これを機能させるために、このMIBモジュールはiscsiDescriptorsサブツリー内のオブジェクトIDのセットを定義します。これらのオブジェクトIDはそれぞれ、基本的に列挙型です。

Attributes that make use of these object identities have a value that is an Object Identifier (OID) instead of an enumerated type. These OIDs can indicate either the object identities defined in this module or object identities defined elsewhere, such as in an enterprise MIB module. Those implementations that add their own digest methods should also define a corresponding object identity for each of these methods within their own enterprise MIB module, and return its OID whenever one of these attributes is using that method.

6.16. Notifications
6.16. お知らせ

Three notifications are provided. One is sent by an initiator detecting a critical login failure, another is sent by a target detecting a critical login failure, and the third is sent upon a session being terminated due to an abnormal connection or digest failure. Critical failures are defined as those that may expose security-related problems that may require immediate action, such as failures due to authentication, authorization, or negotiation problems. Attributes in the initiator, target, and instance objects provide the information necessary to send in the notification, such as the initiator or target name and IP address at the other end that may have caused the failure.

3つの通知が提供されます。 1つは重大なログインの失敗を検出したイニシエーターによって送信され、もう1つは重大なログインの失敗を検出したターゲットによって送信され、3番目は異常な接続またはダイジェストの失敗によりセッションが終了したときに送信されます。重大な障害とは、認証、承認、またはネゴシエーションの問題による障害など、すぐに対処する必要があるセキュリティ関連の問題を公開する可能性がある障害と定義されています。イニシエータ、ターゲット、およびインスタンスオブジェクトの属性は、通知を送信するために必要な情報を提供します。たとえば、失敗の原因となった可能性のある、イニシエータまたはターゲットの名前と相手側のIPアドレスです。

To avoid sending an excessive number of notifications due to multiple errors counted, an SNMP agent implementing the iSCSI MIB module SHOULD NOT send more than three iSCSI notifications in any 10-second period.

複数のエラーがカウントされることによる過剰な数の通知の送信を回避するために、iSCSI MIBモジュールを実装するSNMPエージェントは、任意の10秒間に3つを超えるiSCSI通知を送信しないでください。

The 3-in-10 rule was chosen because one notification every three seconds was deemed often enough, but should two or three different notifications happen at the same time, it would not be desirable to suppress them. Three notifications in 10 seconds is a happy medium, where a short burst of notifications is allowed, without inundating the network and/or notification host with a large number of notifications.

3-in-10ルールが選択されたのは、3秒ごとに1つの通知で十分であると見なされたためですが、2つまたは3つの異なる通知が同時に発生する場合、それらを抑制することは望ましくありません。 10秒に3回の通知は、メディアや通知ホストに大量の通知を殺到させることなく、短いバーストの通知が許可される幸せな媒体です。

8. Security Considerations
8. セキュリティに関する考慮事項

There are a number of management objects defined in this MIB module with a MAX-ACCESS clause of read-write and/or read-create. Such objects may be considered sensitive or vulnerable in some network environments. The support for SET operations in a non-secure environment without proper protection can have a negative effect on network operations. These are the tables and objects and their sensitivity/vulnerability:

このMIBモジュールには、MAX-ACCESS句が読み取り/書き込みまたは読み取り/作成のいずれかで定義された管理オブジェクトがいくつかあります。このようなオブジェクトは、一部のネットワーク環境では機密または脆弱であると見なされる場合があります。適切に保護されていない非セキュア環境でのSET操作のサポートは、ネットワーク操作に悪影響を及ぼす可能性があります。これらは、テーブルとオブジェクト、およびそれらの機密性/脆弱性です。

iscsiPortalAttributesTable, iscsiTgtPortalAttributesTable, and iscsiIntrPortalAttributesTable can be used to add or remove IP addresses to be used by iSCSI.

iscsiPortalAttributesTable、iscsiTgtPortalAttributesTable、iscsiIntrPortalAttributesTableを使用して、iSCSIで使用するIPアドレスを追加または削除できます。

iscsiTgtAuthAttributesTable entries can be added or removed, to allow or disallow access to a target by an initiator.

iscsiTgtAuthAttributesTableエントリを追加または削除して、イニシエーターによるターゲットへのアクセスを許可または禁止できます。

Some of the readable objects in this MIB module (i.e., objects with a MAX-ACCESS other than not-accessible) may be considered sensitive or vulnerable in some network environments. It is thus important to control even GET and/or NOTIFY access to these objects and possibly to even encrypt the values of these objects when sending them over the network via SNMP. These are the tables and objects and their sensitivity/vulnerability:

このMIBモジュールの一部の読み取り可能なオブジェクト（つまり、アクセス不可以外のMAX-ACCESSを持つオブジェクト）は、一部のネットワーク環境では機密または脆弱であると見なされる場合があります。したがって、これらのオブジェクトへのGETおよび/またはNOTIFYアクセスを制御し、SNMP経由でネットワーク経由で送信するときにこれらのオブジェクトの値を暗号化することも重要です。これらは、テーブルとオブジェクト、およびそれらの機密性/脆弱性です。

iscsiNodeAttributesTable, iscsiTargetAttributesTable, and iscsiTgtAuthorization can be used to glean information needed to make connections to the iSCSI targets this module represents. However, it is the responsibility of the initiators and targets involved to authenticate each other to ensure that an inappropriately advertised or discovered initiator or target does not compromise their security. These issues are discussed in [RFC7143].

iscsiNodeAttributesTable、iscsiTargetAttributesTable、iscsiTgtAuthorizationを使用して、このモジュールが表すiSCSIターゲットへの接続に必要な情報を収集できます。ただし、不適切にアドバタイズまたは検出されたイニシエーターまたはターゲットがそれらのセキュリティを危険にさらさないようにするために、お互いを認証することは、関係するイニシエーターとターゲットの責任です。これらの問題は[RFC7143]で議論されています。

SNMP versions prior to SNMPv3 did not include adequate security. Even if the network itself is secure (for example by using IPsec), even then, there is no control as to who on the secure network is allowed to access and GET/SET (read/change/create/delete) the objects in this MIB module.
