1. Introduction
1. はじめに

This document describes the Security Evaluated Standardized Password-Authenticated Key Exchange (SESPAKE) protocol. The SESPAKE protocol provides password-authenticated key exchange for usage in systems for protection of sensitive information. The protocol is intended to be used to establish keys that are then used to organize a secure channel for protection of sensitive information. The security proofs of the protocol were made for situations involving an active adversary in the channel, including man-in-the-middle (MitM) attacks and attacks based on the impersonation of one of the subjects.

このドキュメントでは、Security Evaluated Standardized Password-Authenticated Key Exchange（SESPAKE）プロトコルについて説明します。 SESPAKEプロトコルは、機密情報の保護のためにシステムで使用するためのパスワード認証された鍵交換を提供します。このプロトコルは、機密情報を保護するための安全なチャネルを編成するために使用されるキーを確立するために使用することを目的としています。プロトコルのセキュリティ証明は、中間者（MitM）攻撃や被験者の1人のなりすましに基づく攻撃など、チャネル内のアクティブな敵が関与する状況向けに作成されました。

4. Protocol Description
4. プロトコルの説明

The main point of the SESPAKE protocol is that parties sharing a weak key (a password) generate a strong common key. An active adversary who has access to a channel is not able to obtain any information that can be used to find a key in offline mode, i.e., without interaction with legitimate participants.

SESPAKEプロトコルの主なポイントは、弱い鍵（パスワード）を共有する当事者が強い共通鍵を生成することです。チャネルにアクセスできるアクティブな敵は、正当な参加者との対話なしでは、オフラインモードでキーを見つけるために使用できる情報を取得できません。

The protocol is used by subjects A (client) and B (server) that share some secret parameter that was established in an out-of-band mechanism: a client is a participant who stores a password as a secret parameter, and a server is a participant who stores a password-based computed point of the elliptic curve.

プロトコルは、帯域外メカニズムで確立されたいくつかの秘密パラメーターを共有するサブジェクトA（クライアント）とB（サーバー）によって使用されます。クライアントは、パスワードを秘密パラメーターとして保存する参加者であり、サーバーは楕円曲線のパスワードに基づいて計算された点を保存する参加者。

The SESPAKE protocol consists of two steps: the key-agreement step and the key-confirmation step. During the first step (the key-agreement step), the parties exchange keys using Diffie-Hellman with public components masked by an element that depends on the password -- one of the predefined elliptic curve points multiplied by the password-based coefficient. This approach provides an implicit key authentication, which means that after this step, one party is assured that no other party, aside from a specifically identified second party, may gain access to the generated secret key. During the second step (the key-confirmation step), the parties exchange strings that strongly depend on the generated key. After this step, the parties are assured that a legitimate party, and no one else, actually has possession of the secret key.

SESPAKEプロトコルは、鍵の合意ステップと鍵の確認ステップの2つのステップで構成されています。最初のステップ（鍵合意ステップ）では、パスワードに依存する要素によってマスクされたパブリックコンポーネントを使用して、Diffie-Hellmanを使用して鍵を交換します。事前定義された楕円曲線ポイントの1つにパスワードベースの係数を掛けたものです。このアプローチは、暗黙的なキー認証を提供します。つまり、このステップの後、特定の第2パーティ以外のパーティは、生成された秘密キーにアクセスできないことが保証されます。 2番目のステップ（キー確認ステップ）では、生成されたキーに強く依存する文字列を当事者が交換します。このステップの後、当事者は、正当な当事者だけが実際に秘密鍵を所有していることが保証されます。

To protect against online guessing attacks, counters that indicate the number of failed connections were introduced in the SESPAKE protocol. There is also a special technique for small-order point processing and a mechanism that provides protection against reflection attacks by using different operations for different sides.

オンライン推測攻撃から保護するために、失敗した接続の数を示すカウンターがSESPAKEプロトコルで導入されました。また、小次数のポイントを処理するための特別な手法と、さまざまなサイドでさまざまな操作を使用してリフレクション攻撃から保護するメカニズムもあります。

The full description of the protocol consists of the following steps:

プロトコルの完全な説明は、次の手順で構成されています。

1. If any of the counters C_1^A, C_2^A, or C_3^A is equal to 0, A finishes the protocol with an informational error regarding exceeding the number of trials that is controlled by the corresponding counter.

1. カウンターC_1 ^ A、C_2 ^ A、またはC_3 ^ Aのいずれかが0に等しい場合、Aは、対応するカウンターによって制御される試行回数の超過に関する情報エラーでプロトコルを終了します。

2. A decrements each of the counters C_1^A, C_2^A, and C_3^A by 1, requests open authentication information from B, and sends the ID_A identifier.

2. カウンターC_1 ^ A、C_2 ^ A、およびC_3 ^ Aのそれぞれを1ずつ減分して、Bにオープン認証情報を要求し、ID_A識別子を送信します。

3. If any of the counters C_1^B, C_2^B, or C_3^B is equal to 0, B finishes the protocol with an informational error regarding exceeding the number of trials that is controlled by the corresponding counter.

3. カウンターC_1 ^ B、C_2 ^ B、またはC_3 ^ Bのいずれかが0に等しい場合、Bは対応するカウンターによって制御される試行回数の超過に関する情報エラーでプロトコルを終了します。

4. B decrements each of the counters C_1^B, C_2^B, and C_3^B by 1.

4. Bは、各カウンターC_1 ^ B、C_2 ^ B、およびC_3 ^ Bを1ずつデクリメントします。

5. B sends the values of ind, salt, and the ID_ALG identifier to A. B also can OPTIONALLY send the ID_B identifier to A. All subsequent calculations are done by B in the elliptic curve group defined by the ID_ALG identifier.

5. Bは、ind、salt、およびID_ALG識別子の値をAに送信します。Bは、オプションでID_B識別子をAに送信することもできます。以降のすべての計算は、ID_ALG識別子で定義された楕円曲線グループのBによって実行されます。

6. A sets the curve defined by the received ID_ALG identifier as the used elliptic curve. All subsequent calculations are done by A in this elliptic curve group.

6. Aは、受信したID_ALG識別子で定義された曲線を、使用される楕円曲線として設定します。以降の計算はすべて、この楕円曲線グループのAによって行われます。

7. A calculates the point Q_PW^A = int(F(PW, salt, 2000))*Q_ind.

7. Aは点Q_PW ^ A = int（F（PW、salt、2000））* Q_indを計算します。

8. A chooses randomly (according to the uniform distribution) the value alpha; alpha is in {1,...,q-1}; then A assigns z_A = 0.

8. Aはランダムに（均一分布に従って）値alphaを選択します。 alphaは{1、...、q-1}にあります。次に、Aはz_A = 0を割り当てます。

9. A sends the value u_1 = alpha*P - Q_PW^A to B.

10. After receiving u_1, B checks to see if u_1 is in E. If it is not, B finishes with an error and considers the authentication process unsuccessful.

10. u_1を受け取った後、Bはu_1がEにあるかどうかを確認します。そうでない場合、Bはエラーで終了し、認証プロセスが失敗したと見なします。

11. B calculates Q_B = u_1 + Q_PW, assigns z_B = 0, and chooses randomly (according to the uniform distribution) the value beta; beta is in {1,...,q-1}.

11. BはQ_B = u_1 + Q_PWを計算し、z_B = 0を割り当て、ランダムに（一様分布に従って）ベータ値を選択します。ベータ版は{1、...、q-1}にあります。

12. If m/q*Q_B = O, B assigns Q_B = beta*P and z_B = 1.

12. m / q * Q_B = Oの場合、BはQ_B = beta * Pおよびz_B = 1を割り当てます。

13. B calculates K_B = HASH(BYTES(( m/q*beta*(mod q))*Q_B )).

13. BはK_B = HASH（BYTES（（m / q * beta *（mod q））* Q_B））を計算します。

14. B sends the value u_2 = beta*P + Q_PW to A.

14. Bは値u_2 = beta * P + Q_PWをAに送信します。

15. u_2を受け取った後、Aはu_2がEにあるかどうかを確認します。そうでない場合、Aはエラーで終了し、認証プロセスが失敗したと見なします。

18. AはK_A = HASH（BYTES（（m / q * alpha（mod q））* Q_A））を計算します。

22. Bは、U_1 = BYTES（u_1）、U_2 = BYTES（u_2）を計算します。

23. Bは、MAC_AとHMAC（K_B、0x01 || ID_A || ind || salt || U_1 || U_2 || ID_ALG（OPTIONAL）|| DATA_A）が等しいかどうかを確認します。そうでない場合、エラーで終了し、認証プロセスが失敗したと見なします。

24. z_B = 1の場合、Bはエラーで終了し、認証プロセスが失敗したと見なします。

26. BはMAC_B = HMAC（K_B、0x02 || ID_B || ind || salt || U_1 || U_2 || ID_ALG（OPTIONAL）|| DATA_A || DATA_B）を計算します。DATA_BはMAC_Bで認証されるオプションの文字列です（使用しない場合、DATA_Bは長さゼロと見なされます）。

28. Aは、MAC_BとHMAC（K_A、0x02 || ID_B || ind || salt || U_1 || U_2 || ID_ALG（OPTIONAL）|| DATA_A || DATA_B）が等しいかどうかを確認します。そうでない場合、エラーで終了し、認証プロセスが失敗したと見なします。

29. z_A = 1の場合、Aはエラーで終了し、認証プロセスが失敗したと見なします。

30. Aは、C_1 ^ Aの値をCLim_1に設定し、C_2 ^ Aを1増やします。

5. Construction of Points {Q_1,...,Q_N}
5. ポイントの構築{Q_1、...、Q_N}

This section provides an example of a possible algorithm for the generation of each point Q_i in the set {Q_1,...,Q_N} that corresponds to the given elliptic curve E.

このセクションでは、与えられた楕円曲線Eに対応するセット{Q_1、...、Q_N}の各点Q_iを生成するための可能なアルゴリズムの例を示します。

The algorithm is based on choosing points with coordinates with known preimages of a cryptographic hash function H, which is the GOST R 34.11-2012 hash function (see [RFC6986]) with 256-bit output if 2^254 < q < 2^256, and the GOST R 34.11-2012 hash function (see [RFC6986]) with 512-bit output if 2^508 < q < 2^512.

アルゴリズムは、暗号化ハッシュ関数Hの既知のプリイメージを持つ座標を持つポイントの選択に基づいています。これは、2 ^ 254 <q <2 ^ 256の場合、256ビット出力のGOST R 34.11-2012ハッシュ関数（[RFC6986]を参照）です。 、および2 ^ 508 <q <2 ^ 512の場合、512ビット出力のGOST R 34.11-2012ハッシュ関数（[RFC6986]を参照）。

The algorithm consists of the following steps:

アルゴリズムは次のステップで構成されています。

1. Set i = 1, SEED = 0, s = 4.

1. i = 1、SEED = 0、s = 4に設定します。

2. Calculate X = int(HASH(BYTES(P) || bytes_s(SEED))) mod p.

2. X = int（HASH（BYTES（P）|| bytes_s（SEED）））mod pを計算します。

3. Check to see if the value of X^3 + aX + b is a quadratic residue in the field F_p. If it is not, set SEED = SEED + 1 and return to Step 2.

3. X ^ 3 + aX + bの値がフィールドF_pの2次残差であるかどうかを確認します。そうでない場合は、SEED = SEED + 1を設定し、ステップ2に戻ります。

4. Choose the value of Y = min{r1, r2}, where r1, r2 from {0,1,...,p-1} are such that r1 != r2 and r1^2 = r2^2 = R mod p for R = X^3 + aX + b.

4. Y = min {r1、r2}の値を選択します。{0,1、...、p-1}からのr1、r2は、r1！= r2およびr1 ^ 2 = r2 ^ 2 = R mod p R = X ^ 3 + aX + bの場合。

5. Check to see if the following relations hold for the point Q = (X, Y): Q != O and q*Q = O. If they do, go to Step 6; if not, set SEED = SEED + 1 and return to Step 2.

5. 点Q =（X、Y）について次の関係が成り立つかどうかを確認します。Q！= Oおよびq * Q =O。成り立つ場合は、手順6に進みます。そうでない場合は、SEED = SEED + 1を設定し、ステップ2に戻ります。

6. Set Q_i = Q. If i < N, then set i = i + 1 and go to Step 2; otherwise, finish.

6. Q_i = Qを設定します。i<Nの場合、i = i + 1を設定してステップ2に進みます。それ以外の場合は、終了します。

With the defined algorithm for any elliptic curve E, point sets {Q_1,...,Q_N} are constructed. Constructed points in one set MUST have distinct X-coordinates.

楕円曲線Eに対して定義されたアルゴリズムを使用して、ポイントセット{Q_1、...、Q_N}が作成されます。 1つのセットで構築されたポイントは、異なるX座標を持つ必要があります。

Note: The knowledge of a hash function preimage prevents knowledge of the multiplicity of any point related to generator point P. It is of primary importance, because such knowledge could be used to implement an attack against the protocol with an exhaustive search for the password.

注：ハッシュ関数のプリイメージの知識は、ジェネレーターポイントPに関連するポイントの多重度の知識を妨げます。このような知識は、パスワードを徹底的に検索するプロトコルに対する攻撃を実装するために使用できるため、最も重要です。

6. Security Considerations
6. セキュリティに関する考慮事項

Any cryptographic algorithms -- particularly HASH functions and HMAC functions -- that are used in the SESPAKE protocol MUST be carefully designed and MUST be able to withstand all known types of cryptanalytic attacks.

SESPAKEプロトコルで使用される暗号化アルゴリズム（特にHASH関数とHMAC関数）は、注意深く設計し、既知のすべてのタイプの暗号解読攻撃に耐えられる必要があります。

It is RECOMMENDED that the HASH function satisfy the following condition:

HASH関数は次の条件を満たすことをお勧めします。

o hashlen <= log_2(q) + 4, where hashlen is the length of the HASH function output.

o hashlen <= log_2（q）+4。ここで、hashlenはHASH関数出力の長さです。

It is RECOMMENDED that the output length of hash functions used in the SESPAKE protocol be greater than or equal to 256 bits.

SESPAKEプロトコルで使用されるハッシュ関数の出力長は256ビット以上にすることをお勧めします。

The points {Q_1,Q_2,...,Q_N} and P MUST be chosen in such a way that they are provably pseudorandom. As a practical matter, this means that the algorithm for generation of each point Q_i in the set {Q_1,...,Q_N} (see Section 5) ensures that the multiplicity of any point under any other point is unknown.

ポイント{Q_1、Q_2、...、Q_N}とPは、疑似ランダムであることが証明されるように選択する必要があります。実際問題として、これは、セット{Q_1、...、Q_N}（セクション5を参照）内の各ポイントQ_iの生成アルゴリズムにより、他のポイントの下のポイントの多重度が確実に未知になることを意味します。

Using N = 1 is RECOMMENDED.

N = 1を使用することをお勧めします。

Note: The specific adversary models for the protocol discussed in this document can be found in [SESPAKE-SECURITY], which contains the security proofs.

注：このドキュメントで説明されているプロトコルの特定の敵対モデルは、セキュリティ証明を含む[SESPAKE-SECURITY]にあります。

def XGCD(a,b):
  """XGCD(a,b) returns a list of form [g,x,y], where g is GCD(a,b) and
  x,y satisfy the equation g = ax + by."""
  a1=1; b1=0; a2=0; b2=1; aneg=1; bneg=1; swap = False
  if(a < 0):
    a = -a; aneg=-1
  if(b < 0):
    b = -b; bneg=-1
  if(b > a):
    swap = True
    [a,b] = [b,a]
  while (1):
    quot = -(a / b)
    a = a % b
    a1 = a1 + quot*a2; b1 = b1 + quot*b2
    if(a == 0):
      if(swap):
        return [b, b2*bneg, a2*aneg]
      else:
        return [b, a2*aneg, b2*bneg]
    quot = -(b / a)
    b = b % a
    a2 = a2 + quot*a1; b2 = b2 + quot*b1
    if(b == 0):
      if(swap):
        return [a, b1*bneg, a1*aneg]
      else:
        return [a, a1*aneg, b1*bneg]
