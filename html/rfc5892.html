1. Introduction
1. はじめに

RFC 4690 [RFC4690] suggests an inclusion-based approach for selecting the code points from The Unicode Standard [Unicode52] that should be included in the list of code points that may be used in Internationalized Domain Names.

RFC 4690 [RFC4690]は、国際化ドメイン名で使用できるコードポイントのリストに含める必要があるUnicode標準[Unicode52]からコードポイントを選択するための包含ベースのアプローチを提案しています。

Specifically, RFC 4690 [RFC4690] says the following:

具体的には、RFC 4690 [RFC4690]は次のように述べています。

The IAB has concluded that there is a consensus within the broader community that lists of code points should be specified by the use of an inclusion-based mechanism (i.e., identifying the characters that are permitted), rather than by excluding a small number of characters from the total Unicode set as Stringprep [RFC3454] and Nameprep [RFC3491] do today. That conclusion should be reviewed by the IETF community and action taken as appropriate.

IABは、少数の文字を除外するのではなく、インクルージョンベースのメカニズムを使用する（つまり、許可されている文字を識別する）（つまり、コードポイントのリストが指定されるべきであるという、より広範なコミュニティ内にコンセンサスがあると結論付けました。stringprep [rfc3454]およびnameprep [rfc3491]としての合計ユニコードセットから、今日はそうしています。その結論は、IETFコミュニティと必要に応じて取られた行動によってレビューされるべきです。

This document reviews and classifies the collections of code points in the Unicode character set by examining various properties of the code points. It then defines an algorithm for determining a derived property value. It specifies a procedure, and not a table, of code points so that the algorithm can be used to determine code point sets independent of the version of Unicode that is in use.

このドキュメントでは、コードポイントのさまざまなプロパティを調べることにより、Unicode文字設定のコードポイントのコレクションをレビューおよび分類します。次に、派生したプロパティ値を決定するためのアルゴリズムを定義します。アルゴリズムを使用して、使用中のUnicodeのバージョンに依存しないコードポイントセットを決定できるように、コードポイントのテーブルではなく手順を指定します。

This document is not intended to specify precisely how these property values are to be applied in IDN labels. That information appears in the Protocol document [RFC5891], but it is important to understand that the assignment of a value of this property to a particular character is not sufficient to determine whether it can be used in a given label. In particular, some combinations of allowed code points are not advisable for use in IDNs due to rules specific to a script or class of characters. The requirement for such rules is linked to the operations in the Protocol document and especially to the characters designated as requiring contextual rules.

このドキュメントは、これらのプロパティ値をIDNラベルで適用する方法を正確に指定することを意図したものではありません。その情報はプロトコルドキュメント[RFC5891]に表示されますが、特定のキャラクターへのこのプロパティの値の割り当ては、特定のラベルで使用できるかどうかを判断するのに十分ではないことを理解することが重要です。特に、許可されたコードポイントのいくつかの組み合わせは、スクリプトまたは文字のクラスに固有のルールのため、IDNで使用することをお勧めしません。そのようなルールの要件は、プロトコルドキュメントの操作、特にコンテキストルールを必要とするものとして指定された文字にリンクされています。

The value of the property is to be interpreted as follows.

プロパティの価値は次のように解釈されます。

o PROTOCOL VALID: Those that are allowed to be used in IDNs. Code points with this property value are permitted for general use in IDNs. However, that a label consists only of code points that have this property value does not imply that the label can be used in DNS. See the Protocol document for algorithms to make decisions about labels in domain names. The abbreviated term PVALID is used to refer to this value in the rest of this document.

o 有効なプロトコル：IDNで使用できるもの。このプロパティ値を持つコードポイントは、IDNで一般的に使用するために許可されています。ただし、ラベルは、このプロパティ値を持つコードポイントのみで構成されていることは、ラベルがDNSで使用できることを意味するものではありません。ドメイン名のラベルに関する決定を下すには、アルゴリズムのプロトコルドキュメントを参照してください。略語された用語PVALIDは、このドキュメントの残りの部分でこの値を参照するために使用されます。

o CONTEXTUAL RULE REQUIRED: Some characteristics of the character, such as it being invisible in certain contexts or problematic in others, require that it not be used in labels unless specific other characters or properties are present. The abbreviated term CONTEXT is used to refer to this value in the rest of this document. There are two subdivisions of CONTEXTUAL RULE REQUIRED, one for Join_controls (called CONTEXTJ) and for other characters (called CONTEXTO). These are discussed in more detail below and in the Protocol document.

o 必要なコンテキストルール：特定のコンテキストでは見えない、または他のものでは問題があるなど、キャラクターのいくつかの特性は、特定の他のキャラクターまたはプロパティが存在しない限り、ラベルで使用しないことを要求します。略語された用語のコンテキストは、このドキュメントの残りの部分でこの値を参照するために使用されます。必要なコンテキストルールには2つの下位区分があります。1つはJoin_Controls（ContextJと呼ばれる）と他の文字（Contextoと呼ばれる）です。これらについては、以下とプロトコルドキュメントで詳細に説明します。

o DISALLOWED: Those that should clearly not be included in IDNs. Code points with this property value are not permitted in IDNs.

o 許可されていない：明らかにIDNに含まれるべきではないもの。このプロパティ値を持つコードポイントは、IDNSでは許可されていません。

o UNASSIGNED: Those code points that are not designated (i.e., are unassigned) in the Unicode Standard.

o 未割り当て：Unicode標準で指定されていない（つまり、割り当てられていない）コードポイント。

The mechanisms described here allow determination of the value of the property for future versions of Unicode (including characters added after Unicode 5.2). Changes in Unicode properties that do not affect the outcome of this process do not affect IDN. For example, a character can have its Unicode General_Category value (see [Unicode52]) change from So to Sm or from Lo to Ll, without affecting the algorithm results. Moreover, even if such changes were the result, the BackwardCompatible list (Section 2.7) can be adjusted to ensure the stability of the results.

ここで説明するメカニズムは、Unicodeの将来のバージョンのプロパティの値を決定することを可能にします（Unicode 5.2の後に追加された文字を含む）。このプロセスの結果に影響を与えないユニコードプロパティの変化は、IDNに影響しません。たとえば、文字は、アルゴリズムの結果に影響を与えることなく、unicode general_category値（[unicode52]を参照[unicode52]を参照）またはloからLLに変更できます。さらに、そのような変更が結果であったとしても、結果の安定性を確保するために、後方競争の可能性のあるリスト（セクション2.7）を調整できます。

Some code points need to be allowed in exceptional circumstances but should be excluded in all other cases; these rules are also described in other documents. The most notable of these are the Join Control characters, U+200D ZERO WIDTH JOINER and U+200C ZERO WIDTH NON-JOINER. Both of them have the derived property value CONTEXTJ. A character with the derived property value CONTEXTJ or CONTEXTO (CONTEXTUAL RULE REQUIRED) is not to be used unless an appropriate rule has been established and the context of the character is consistent with that rule. It is invalid to either register a string containing these characters or even to look one up unless such a contextual rule is found and satisfied. Please see Appendix A, "The Contextual Rules Registry", for more information.

いくつかのコードポイントは、例外的な状況で許可する必要がありますが、他のすべてのケースでは除外する必要があります。これらのルールは、他のドキュメントでも説明されています。これらの中で最も注目すべきは、結合制御文字、u 200dゼロ幅のジョイナー、u 200cゼロ幅以外のジョイナーです。どちらも派生したプロパティ値コンテキストを持っています。派生したプロパティ値コンテキストJまたはコンテキスト（コンテキストルールが必要）を備えた文字は、適切なルールが確立され、文字のコンテキストがそのルールと一致していない限り、使用する必要はありません。これらの文字を含む文字列を登録するか、そのような文脈的ルールが見つかり、満たされない限り、1つを調べることさえ無効です。詳細については、付録A「コンテキストルールレジストリ」を参照してください。

This document is part of a series that, together, constitute a proposal for updating the IDNA standards to resolve issues uncovered in recent years, cover a broader range of scripts, and provide for migration to newer versions of Unicode. See the Rationale document [RFC5894] for a broader discussion.

このドキュメントは、IDNA標準を更新して、近年明らかになった問題を解決し、より広範なスクリプトをカバーし、Unicodeの新しいバージョンへの移行を提供するシリーズの一部です。より広範な議論については、Rationale Document [RFC5894]を参照してください。

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 [RFC2119].

この文書のキーワード "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", および "OPTIONAL" はRFC 2119 [RFC2119]で説明されているように解釈されます。

2. Category Definitions Used to Calculate Derived Property Value
2. 派生したプロパティ値を計算するために使用されるカテゴリ定義

The derived property obtains its value based on a two-step procedure. First, characters are placed in one or more character categories based on either core properties defined by the Unicode Standard or by treating the code point as an exception and addressing the code point by its code point value. These categories are not mutually exclusive.

派生プロパティは、2段階の手順に基づいてその値を取得します。まず、文字は、Unicode標準で定義されたコアプロパティに基づいて、またはコードポイントを例外として扱い、コードポイント値でコードポイントに対処することによって、1つ以上の文字カテゴリに配置されます。これらのカテゴリは相互に排他的ではありません。

In the second step, set operations are used with these categories to determine the values for an IDN-specific property. Those operations are specified in Section 3.

2番目のステップでは、これらのカテゴリでセット操作を使用して、IDN固有のプロパティの値を決定します。これらの操作はセクション3で指定されています。

Unicode property names and property value names may have short abbreviations, such as gc for the General_Category property, and Ll for the Lowercase_Letter property value of the gc property.

Unicodeプロパティ名とプロパティ値名には、General_CategoryプロパティのGCなど、GCプロパティのLowercase_Letterプロパティ値のLLなどの短い略語があります。

In the following specification of categories, the operation that returns the value of a particular Unicode character property for a code point is designated by using the formal name of that property (from PropertyAliases.txt) followed by '(cp)'. For example, the value of the General_Category property for a code point is indicated by General_Category(cp).

次のカテゴリの仕様では、コードポイントの特定のUnicode文字プロパティの値を返す操作は、そのプロパティ（PropertyAliase.txtから）の正式な名前を使用して「（CP）」を使用して指定されます。たとえば、コードポイントのgeneral_categoryプロパティの値は、general_category（cp）で示されています。

2.1. LetterDigits (A)
2.1. LetterDigits（a） 
   A: General_Category(cp) is in {Ll, Lu, Lo, Nd, Lm, Mn, Mc}

These rules identify characters commonly used in mnemonics and often informally described as "language characters". In general, only code points assigned to this category are suitable for use in IDN.

これらのルールは、ニーモニックで一般的に使用されており、しばしば「言語文字」と非公式に説明される文字を識別します。一般に、このカテゴリに割り当てられたコードポイントのみがIDNでの使用に適しています。

For more information, see Section 4.5 of The Unicode Standard [Unicode].

詳細については、Unicode標準[Unicode]のセクション4.5を参照してください。

The categories used in this rule are:

このルールで使用されるカテゴリは次のとおりです。

o Ll - Lowercase_Letter

o LL -Lowercase_letter

o Lu - Uppercase_Letter

o lu -appercase_letter

o Lo - Other_Letter

o lo -other_letter

o Nd - Decimal_Number

o nd -decimal_number

o Lm - Modifier_Letter o Mn - Nonspacing_Mark

o lm -modifier_letter o mn -nonspacing_mark

o Mc - Spacing_Mark

o MC -spacing_mark

2.2. Unstable (B)
2.2. 不安定（b） 
   B: toNFKC(toCaseFold(toNFKC(cp))) != cp

This category is used to group the characters that are not stable under Normalization Form K (NFKC) and case folding. In general, these code points are not suitable for use for IDN.

このカテゴリは、正規化フォームK（NFKC）およびケース折りたたみ式で安定していない文字をグループ化するために使用されます。一般に、これらのコードポイントはIDNの使用には適していません。

The toCaseFold() operation is defined in Section 3.13 of The Unicode Standard [Unicode].

ToCaseFold（）操作は、Unicode標準[Unicode]のセクション3.13で定義されています。

The toNFKC() operation returns the code point in normalization form KC. For more information, see Section 5 of Unicode Standard Annex #15 [TR15].

tonfkc（）操作は、コードポイントを正規化フォームKCに戻します。詳細については、Unicode Standard Annex＃15 [TR15]のセクション5を参照してください。

It should be noted that NFKC is used, although Normalization Form C (NFC) is used in the "IDNA Protocol" document [RFC5891].

NFKCが使用されることに注意する必要がありますが、正規化フォームC（NFC）は「IDNAプロトコル」ドキュメント[RFC5891]で使用されています。

2.3. IgnorableProperties (C)
2.3. 無知なプロパティ（c） 
   C: Default_Ignorable_Code_Point(cp) = True or
      White_Space(cp) = True or
      Noncharacter_Code_Point(cp) = True

This category is used to group code points that are not recommended for use in identifiers. In general, these code points are not suitable for use in an IDN.

このカテゴリは、識別子で使用することをお勧めしないコードポイントをグループ化するために使用されます。一般に、これらのコードポイントはIDNでの使用には適していません。

The definition for Default_Ignorable_Code_Point can be found in DerivedCoreProperties.txt [DerivedCoreProperties] and is at the time of Unicode 5.2:

default_ignorable_code_pointの定義は、derivedcoreproperties.txt [derivedcoreproperties]にあり、Unicode 5.2の時点です。

Other_Default_Ignorable_Code_Point + Cf (Format characters) + Variation_Selector - White_Space - FFF9..FFFB (Annotation Characters) - 0600..0603, 06DD, 070F (exceptional Cf characters that should be visible)

other_default_ignorable_code_point cf（フォーマット文字）variation_selector -white_space -fff9..fffb（annotation characteres）-0600..0603、06dd、070f（見えるはずの例外的なCF文字）

2.4. IgnorableBlocks (D)
2.4. 無知なブロック（d） 
   D: Block(cp) is in {Combining Diacritical Marks for Symbols,
                       Musical Symbols, Ancient Greek Musical Notation}

This category is used to identify code points that are not useful in mnemonics or that are otherwise impractical for IDN use. In general, these code points are not suitable for use for IDN.

このカテゴリは、ニーモニックでは役に立たない、またはIDNの使用には非現実的なコードポイントを識別するために使用されます。一般に、これらのコードポイントはIDNの使用には適していません。

The definition of blocks can be found in Blocks.txt [BlockNames].

ブロックの定義は、blocks.txt [ブロック名]にあります。

2.5. LDH (E)
2.5. LDH（E） 
   E: cp is in {002D, 0030..0039, 0061..007A}

This category is used in the second step to preserve the traditional "hostname" (LDH -- as described in the Definitions document [RFC5890]) characters ('-', 0-9, and a-z).
