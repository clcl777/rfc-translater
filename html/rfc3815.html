1. Introduction
1. はじめに

This document defines 4 MIB Modules which together support the configuration and monitoring of the Label Distribution Protocol (LDP). The Label Distribution Protocol (LDP) [RFC3036] is one type of Multiprotocol Label Switching (MPLS) protocols described in [RFC3031] and [RFC3032]. Utilizing all 4 MIB Modules allows an operator to configure LDP sessions using 3 different Layer 2 media. The Layer 2 media supported by the MIB Modules are Ethernet, ATM and Frame Relay as described in [RFC3036], [RFC3034] and [RFC3035].

このドキュメントでは、4つのMIBモジュールを定義し、ラベル分布プロトコル（LDP）の構成と監視をサポートします。ラベル分布プロトコル（LDP）[RFC3036]は、[RFC3031]および[RFC3032]で説明されているマルチプロトコルラベルスイッチング（MPLS）プロトコルの1つのタイプです。4つのMIBすべてのモジュールを使用すると、オペレーターは3つの異なるレイヤー2メディアを使用してLDPセッションを構成できます。MIBモジュールでサポートされるレイヤー2メディアは、[RFC3036]、[RFC3034]、および[RFC3035]で説明されているように、イーサネット、ATM、およびフレームリレーです。

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 [RFC2119].

この文書のキーワード "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", および "OPTIONAL" はRFC 2119 [RFC2119]で説明されているように解釈されます。

For an introduction to the concepts of MPLS, see [RFC3031]. For further on LDP refer to [RFC3037] and [RFC3215].

MPLSの概念の紹介については、[RFC3031]を参照してください。LDPの詳細については、[RFC3037]および[RFC3215]を参照してください。

2. The Internet-Standard Management Framework
2. インターネット標準の管理フレームワーク

For a detailed overview of the documents that describe the current Internet-Standard Management Framework, please refer to section 7 of RFC 3410 [RFC3410].

現在のインターネット標準管理フレームワークを説明するドキュメントの詳細な概要については、RFC 3410 [RFC3410]のセクション7を参照してください。

Managed objects are accessed via a virtual information store, termed the Management Information Base or MIB. MIB objects are generally accessed through the Simple Network Management Protocol (SNMP). Objects in the MIB are defined using the mechanisms defined in the Structure of Management Information (SMI). This memo specifies a MIB module that is compliant to the SMIv2, which is described in STD 58, RFC 2578 [RFC2578], STD 58, RFC 2579 [RFC2579] and STD 58, RFC 2580 [RFC2580].

管理されたオブジェクトは、管理情報ベースまたはMIBと呼ばれる仮想情報ストアからアクセスされます。MIBオブジェクトは通常、単純なネットワーク管理プロトコル（SNMP）からアクセスされます。MIBのオブジェクトは、管理情報の構造（SMI）で定義されたメカニズムを使用して定義されます。このメモは、STD 58、RFC 2578 [RFC2578]、STD 58、RFC 2579 [RFC2579]およびSTD 58、RFC 2580 [RFC2580]に記載されているSMIV2に準拠したMIBモジュールを指定します。

3. Structure of the MIB Modules
3. MIBモジュールの構造

This section describes the structure of the LDP MIB Modules.

このセクションでは、LDP MIBモジュールの構造について説明します。

3.1. Overview
3.1. 概要

There are 4 MIB Modules in this document. These MIB Modules are the MPLS-LDP-STD-MIB, the MPLS-LDP-GENERIC-STD-MIB, the MPLS-LDP-ATM-STD-MIB and the MPLS-LDP-FRAME-RELAY-STD-MIB. The MPLS-LDP-STD-MIB defines objects which are common to all LDP implementations. The MPLS-LDP-GENERIC-STD-MIB defines Layer 2 Per Platform Label Space objects for use with the MPLS-LDP-STD-MIB. The MPLS-LDP-ATM-STD-MIB defines Layer 2 Asynchronous Transfer Mode (ATM) objects for use with the MPLS-LDP-STD-MIB. The MPLS-LDP-FRAME-RELAY-STD-MIB defines Layer 2 FRAME-RELAY objects for use with the MPLS-LDP-STD-MIB.

このドキュメントには4つのMIBモジュールがあります。これらのMIBモジュールは、MPLS-LDP-STD-MIB、MPLS-LDP-GENERIC-STD-MIB、MPLS-LDP-ATM-STD-MIB、およびMPLS-LDP-FRAME-RELAY-STD-MIBです。MPLS-LDP-STD-MIBは、すべてのLDP実装に共通のオブジェクトを定義します。MPLS-LDP-GENERIC-STD-MIBは、MPLS-LDP-STD-MIBで使用するためのプラットフォームラベルスペースオブジェクトごとのレイヤー2を定義します。MPLS-LDP-ATM-STD-MIBは、MPLS-LDP-STD-MIBで使用するレイヤー2非同期転送モード（ATM）オブジェクトを定義します。MPLS-LDP-FRAME-RELAY-STD-MIBは、MPLS-LDP-STD-MIBで使用するレイヤー2フレームレレイオブジェクトを定義します。

The MPLS-LDP-STD-MIB Module MUST be implemented and at least one of the Layer 2 MIB Modules MUST be implemented by an Agent developer on an Label Switching Router (LSR) or Label Edge Router (LER). As an example, if a Label Switching Router (LSR) or Label Edge Router (LER) implementation intends to support LDP utilizing a Layer 2 of Ethernet, then the MPLS-LDP-STD-MIB and the MPLS-LDP-GENERIC-STD-MIB Modules MUST implemented. If an LSR/LER implementation intends to support LDP utilizing a Layer 2 of ATM, then the MPLS-LDP-STD-MIB Module and the MPLS-LDP-ATM-MIB Module MUST be implemented. If an LSR/LER implementation intends to support LDP utilizing a Layer 2 of FRAME-RELAY, then the MPLS-LDP-STD-MIB Module and the MPLS-LDP-FRAME-RELAY-STD-MIB Module MUST be implemented. An LDP implementation that utilizes all three Layer 2 media (Ethernet, Frame-Relay, ATM) MUST support all 4 MIB Modules. Each of the Modules will be discussed in detail in the following sections.

MPLS-LDP-STD-MIBモジュールを実装する必要があり、レイヤー2 MIBモジュールの少なくとも1つは、ラベルスイッチングルーター（LSR）またはラベルエッジルーター（LER）にエージェント開発者によって実装する必要があります。例として、ラベルスイッチングルーター（LSR）またはラベルエッジルーター（LER）の実装が、イーサネットのレイヤー2を使用してLDPをサポートする場合、MPLS-LDP-STD-MIBおよびMPLS-LDP-GENERIC-STD-をサポートする場合MIBモジュールを実装する必要があります。LSR/LER実装がATMのレイヤー2を使用してLDPをサポートする場合、MPLS-LDP-STD-MIBモジュールとMPLS-LDP-ATM-MIBモジュールを実装する必要があります。LSR/LER実装がFrame-Relayのレイヤー2を使用してLDPをサポートする場合、MPLS-LDP-STD-MIBモジュールとMPLS-LDP-FRAME-RELAY-STD-MIBモジュールを実装する必要があります。3つのレイヤー2メディア（イーサネット、フレームレレー、ATM）すべてを使用するLDP実装は、4つのMIBすべてのモジュールをすべてサポートする必要があります。各モジュールについては、次のセクションで詳しく説明します。

There are 2 compliance statements for each MIB Module. One compliance statement is for full compliance which allows both configuration and monitoring via SNMP. The other compliance statement is for read-only compliance which allows only monitoring via SNMP.

MIBモジュールごとに2つのコンプライアンスステートメントがあります。コンプライアンスステートメントの1つは、SNMPを介した構成と監視の両方を可能にする完全なコンプライアンスに関するものです。もう1つのコンプライアンスステートメントは、SNMPを介した監視のみを可能にする読み取り専用コンプライアンスに関するものです。

[RFC2684]はVPマージ機能を説明していないため、この機能は含まれていません。

These areas need to be specified in the LDP Specification or other specifications prior to being added in this or any other MIB document.

これらの領域は、この領域またはその他のMIBドキュメントに追加される前に、LDP仕様またはその他の仕様で指定する必要があります。

3.4. Differences from the LDP Specification
3.4. LDP仕様との違い

Currently, there are 3 differences between this specification and the LDP Specification. As described in the Introduction, this document is almost entirely based on the LDP specification. The differences are documented here.

現在、この仕様とLDP仕様の間には3つの違いがあります。導入部で説明されているように、このドキュメントはほぼ完全にLDP仕様に基づいています。違いはここで文書化されています。

The first difference is that the LDP Entity Table contains some DEFVAL clauses which are not specified explicitly in the LDP Specification. These values, although not documented in the LDP Specification, are widely used by existing LDP MIB implementations and thus, have been adopted within this MPLS-LDP-STD-MIB module. Please note, they can certainly be changed during row creation or a subsequent SET request.

最初の違いは、LDPエンティティテーブルに、LDP仕様で明示的に指定されていないいくつかのdefval句が含まれていることです。これらの値は、LDP仕様には文書化されていませんが、既存のLDP MIB実装で広く使用されているため、このMPLS-LDP-STD-MIBモジュール内で採用されています。注意してください、それらは確かに列の作成中またはその後のセットリクエスト中に変更することができます。

A second difference is the mplsLdpEntityGenericLRTable in the MPLS-LDP-GENERIC-STD-MIB Module. This table, although provided as a way to reserve a range of generic labels, does not exist in the LDP Specification. It was added to the MIB due to a request from the working group and because this table was considered useful for reserving a range of generic labels.

2番目の違いは、MPLS-LDP-GENERIC-STD-MIBモジュールのMPLSLDPENTITYGENERICLRTABLEです。このテーブルは、さまざまな一般的なラベルを予約する方法として提供されていますが、LDP仕様には存在しません。ワーキンググループからのリクエストにより、そしてこのテーブルはさまざまな一般的なラベルを予約するのに役立つと考えられていたため、MIBに追加されました。

The third difference is documented by the TEXTUAL-CONVENTION, MplsAtmVcIdentifier which is in the MPLS-TC-STD-MIB [RFC3811]. This TC was added to restrict vci values to be greater than 31 as described in RFC 3035 [RFC3035].

3番目の違いは、MPLS-TC-STD-MIB [RFC3811]にあるTextual ConventionであるMPLSATMVCIDENTIFIERによって文書化されています。このTCは、RFC 3035 [RFC3035]に記載されているように、VCI値を31を超えるように制限するために追加されました。

3.5. The MPLS-LDP-STD-MIB Module
3.5. MPLS-LDP-STD-MIBモジュール

This MIB Module contains objects which are common to all LDP implementations. This MIB Module MUST always be implemented along with one or more of the Layer 2 MIB Modules. This MIB Module IMPORTS IndexInteger and IndexIntegerNextFree TEXTUAL-CONVENTIONs from [RFC3289], and IMPORTS InetAddressPrefixLength, InetAddressType, InetAddressInetPortNumber TEXTUAL-CONVENTIONs from [RFC3291].

このMIBモジュールには、すべてのLDP実装に共通のオブジェクトが含まれています。このMIBモジュールは、常に1つ以上のレイヤー2 MIBモジュールとともに実装する必要があります。このMIBモジュールは、[RFC3289]からIndexIntegerおよびIndexIntegernextFreeのテキストコンベニオンをインポートし、[RFC3291]からのInetAddressPrefixLength、InetAddressInetPortNumber TextNumber TextNumberのインポートをインポートします。

The mplsLdpEntityTable table allows the Label Edge Router (LER) or the Label Switching Router (LSR) to initiate and/or receive requests to establish LDP sessions. As the LDP protocol distributes labels and establishes sessions with Peers most of the tables in this module are populated by the agent as instructed by the LDP protocol. The exception is the mplsFecTable and the mplsLdpLspFecTable which can be configured by the operator to specify Forwarding Equivalence Class information for an LSP.

MPLSLDDPENTITYTABLEテーブルを使用すると、ラベルエッジルーター（LER）またはラベルスイッチングルーター（LSR）がLDPセッションを確立するためのリクエストを開始および/または受信できます。LDPプロトコルはラベルを配布し、ピアとセッションを確立するため、このモジュールのテーブルのほとんどは、LDPプロトコルの指示に従ってエージェントによって入力されます。例外は、LSPの転送等価クラス情報を指定するために演算子によって構成できるMPLSFECTABLEおよびMPLSLDPLSPFECTABLEです。

Some scalars and each table in the MPLS-LDP-STD-MIB Module are described in the following subsections.

MPLS-LDP-STD-MIBモジュールのいくつかのスカラーと各テーブルについては、以下のサブセクションで説明します。

3.5.1. LDP Scalar Objects
3.5.1. LDPスカラーオブジェクト

There are several scalar objects in the LDP MIB module. The mplsLdpLsrId is a read-only scalar object which reports Label Switching Router's (LSR's) Identifier. This MUST be a globally unique value, such as the 32-bit router ID assigned to the LSR.

LDP MIBモジュールにはいくつかのスカラーオブジェクトがあります。MPLSLDPLSRIDは、ラベルスイッチングルーター（LSRの）識別子を報告する読み取り専用スカラーオブジェクトです。これは、LSRに割り当てられた32ビットルーターIDなど、グローバルに一意の値である必要があります。

The mplsLdpLsrLoopDetectionCapable scalar object denotes whether the LSR is capable of supporting loop detection and if so, which form of loop detection.

mplsldplsrloopdetectioncapable scalarオブジェクトは、LSRがループ検出をサポートできるかどうかを示します。

There are two LastChange scalar objects, mplsLdpEntityLastChange and mplsLdpPeerLastChange. These objects give an indication of there was a change in the number of entries in the table, or if any of the values in the respective tables changed. Please see the object's description for more details.

2つのLastChangeスカラーオブジェクト、mplslddpentitylastchangeとmplsldppeerlastchangeがあります。これらのオブジェクトは、テーブル内のエントリの数に変更があったこと、またはそれぞれのテーブル内の値のいずれかが変更されたかどうかを示しています。詳細については、オブジェクトの説明をご覧ください。

The mplsLdpEntityIndexNext scalar object is described in the next section.

mplslddpentityindexnextスカラーオブジェクトについては、次のセクションで説明します。

3.5.2. The LDP Entity Table
3.5.2. LDPエンティティテーブル

The MPLS-LDP-STD-MIB provides objects to configure/set-up potential LDP sessions on a specific LSR/LER. The mplsLdpEntityTable is used to configure information which is used by the LDP protocol to setup potential LDP Sessions.

MPLS-LDP-STD-MIBは、特定のLSR/LERで潜在的なLDPセッションを構成/セットアップするオブジェクトを提供します。MPLSLDPENTITYTABLEは、LDPプロトコルによって使用される情報を構成するために使用され、潜在的なLDPセッションをセットアップします。

Each entry/row in this table represents a single LDP Entity. There is no maximum number of LDP Entities specified. However, there is an mplsLdpEntityIndexNext object which should be retrieved by the command generator prior to creating an LDP Entity. If the mplsLdpEntityIndexNext object is zero, this indicates that the LSR/LER is not able to create another LDP Entity at that time.

この表の各エントリ/行は、単一のLDPエンティティを表します。指定されたLDPエンティティの最大数はありません。ただし、LDPエンティティを作成する前にコマンドジェネレーターによって取得する必要があるmplslddpentityIndexNextオブジェクトがあります。mplslddpentityIndexNextオブジェクトがゼロの場合、これはLSR/LERがその時点で別のLDPエンティティを作成できないことを示しています。

3.5.2.1. Changing Values After Session Establishment
3.5.2.1. セッションの確立後に値を変更します

One way to manually modify a session's parameters is by using SNMP to change the MIB objects related to that session. Please note, special care should be taken if MIB objects which are used in the MPLS LDP Session Initialization need to be modified. If the modification of any of these MIB variables takes place anytime after the start of session intialization, then the entire session must be halted. Any information learned by that session must be discarded. The objects should then be modified, and session initialization started. Assuming that the configuration was done correctly, then a new session will be created.

セッションのパラメーターを手動で変更する1つの方法は、SNMPを使用してそのセッションに関連するMIBオブジェクトを変更することです。MPLS LDPセッションの初期化で使用されるMIBオブジェクトを変更する必要がある場合は、特別な注意を払う必要があります。これらのMIB変数のいずれかの変更がセッションの開始後いつでも行われる場合、セッション全体を停止する必要があります。そのセッションで学んだ情報はすべて破棄する必要があります。その後、オブジェクトを変更し、セッションの初期化を開始します。構成が正しく行われたと仮定すると、新しいセッションが作成されます。

For example, assume that an operator wishes to change the configuration of a Label Range which is used by a Session that has already been established. The operator should change the mplsLdpEntityAdminStatus to "disable(2)". Setting the mplsLdpEntityAdminStatus to "disable(2)" will cause the session to be torn down (i.e., this will signal to LDP that it should send out tear down messages for that session). Also, all information related to that session should be removed from this MIB by the Agent. This includes Peer information (i.e., relevant row in the mplsPeerTable) and Session statistics (i.e., relevant row in the mplsLdpSessionTable). Also, if the MPLS-LSR-STD-MIB module [RFC3813] is implemented and the optional Mapping Table objects are implemented, then all information related to the LSPs in this session should be removed from these MIB modules. [For more information please see the section on "The Mapping Tables".] At this point, the operator could modify the Label Range. Lastly, the operator should set the mplsLdpEntityAdminStatus to "enable(1)". At this point session initialization should occur. The LDP Entity goes through the Session Initialization in order to communicate the new Label Ranges to the Peer and establish new LSPs.

たとえば、オペレーターが、すでに確立されているセッションで使用されているラベル範囲の構成を変更したいと仮定します。オペレーターは、mplslddpentityadmintatusを「（2）」を無効にするように変更する必要があります。mplslddpentityadmintatusを「無効にする（2）」に設定すると、セッションが取り壊されます（つまり、これはLDPに、そのセッションのために断片化されたメッセージを送信する必要があることを示します）。また、そのセッションに関連するすべての情報は、エージェントがこのMIBから削除する必要があります。これには、ピア情報（つまり、mplspeertableの関連する行）とセッション統計（つまり、mplsldpssessiontableの関連する行）が含まれます。また、MPLS-LSR-STD-MIBモジュール[RFC3813]が実装され、オプションのマッピングテーブルオブジェクトが実装されている場合、このセッションのLSPに関連するすべての情報をこれらのMIBモジュールから削除する必要があります。[詳細については、「マッピングテーブル」のセクションを参照してください。]この時点で、オペレーターはラベル範囲を変更できます。最後に、オペレーターはmplslddpentityadmintatusを「（1）」を有効にするように設定する必要があります。この時点で、セッションの初期化が発生するはずです。LDPエンティティは、新しいラベル範囲をピアに伝え、新しいLSPを確立するために、セッションの初期化を通過します。

3.5.3. The LDP Entity Statistics Table
3.5.3. LDPエンティティ統計テーブル

The mplsLpdEntityStatsTable is a read-only table which contains statistical information related to failed attempts to establish sessions. Each row in this table AUGMENTS an mplsLdpEntityEntry. This table could be used to give insight into how to reconfigure values so that a session could be successfully established. For example, if the mplsLdpEntityStatsSessionRejectedLRErrors Counter object was increasing, then this would indicate that the Label Range (LR) may need to be adjusted.

mplslpdentityStatstableは、セッションを確立するための失敗した試みに関連する統計情報を含む読み取り専用のテーブルです。このテーブルの各行は、mplsldldpentityentryを強化します。このテーブルは、セッションを正常に確立できるように、値を再構成する方法についての洞察を与えるために使用できます。たとえば、mplslddpentitystatsSessionRejectedlrerrorsカウンターオブジェクトが増加している場合、これはラベル範囲（LR）を調整する必要があることを示します。

3.5.4. The LDP Peer Table
3.5.4. LDPピアテーブル

The mplsLdpPeerTable is a read-only table which contains information about LDP Peers known to LDP Entities. In other words, the Peer information is learned by LDP through initialization or discovery. This table should be populated by the agent as directed by the LDP protocol.

MPLSLDPPEERTABLEは、LDPエンティティに知られているLDPピアに関する情報を含む読み取り専用テーブルです。言い換えれば、ピア情報は、初期化または発見を通じてLDPによって学習されます。このテーブルは、LDPプロトコルの指示に従ってエージェントが入力する必要があります。

A row in this table is related to one or more rows in the Hello Adjacency Table and related to a single row in the Session Table. The values in the Peer table are specific to a Peer and may or may not be the same values used in the session. The reason is that the Peer and Entity negotiate certain values. The Entity's values are configured in the mplsLdpEntityTable and the Peer's values are learned (and placed into the mplsLdpPeerTable). The mplsLdpSessionTable shows the values used in establishing the session.

このテーブルの行は、Hello隣接テーブルの1つ以上の行に関連しており、セッションテーブルの1つの行に関連しています。ピアテーブルの値はピアに固有であり、セッションで使用されている値と同じ値である場合とそうでない場合があります。その理由は、ピアとエンティティが特定の価値を交渉するためです。エンティティの値はmplsldpentitytableで構成され、ピアの値は学習されます（およびmplsldppeertableに配置されます）。MPLSLDPSESSIONTABLEには、セッションの確立に使用される値が表示されます。

One example, of when the Peer's values and the Session's values may differ is with the Peer's Path Limit information. The Peer's Path Limit information is learned from the session initialization phase. The actual value for the Path Vector Limit is the Peer's value and may not be the same value that appears in the session. There could be a mismatch in this value between the Entity and the Peer. In the event of a mismatch, then the session will use the Path Limit set by the Entity (and not the Peer).

ピアの値とセッションの値が異なる場合の1つの例は、ピアのパス制限情報との場合です。ピアのパス制限情報は、セッションの初期化フェーズから学習されます。パスベクトル制限の実際の値はピアの値であり、セッションに表示される同じ値ではない場合があります。エンティティとピアの間にこの値に不一致がある可能性があります。ミスマッチが発生した場合、セッションはエンティティによって設定されたパス制限（ピアではなく）を使用します。

The Peer Table information was placed in a separate table from the Session information to allow for a more comprehensive and coherent MIB model.

ピアテーブル情報は、より包括的で一貫したMIBモデルを可能にするために、セッション情報から別のテーブルに配置されました。

3.5.5. The LDP Session Table
3.5.5. LDPセッションテーブル

The mplsLdpSessionTable is a read-only table. Each entry in this table represents a single session between an LDP Entity and a Peer. The mplsLdpSessionEntry AUGMENTS the mplsLdpPeerEntry.

mplsldpssessionTableは読み取り専用のテーブルです。この表の各エントリは、LDPエンティティとピアの間の単一のセッションを表します。mplsldpsssionEntryは、mplsldppeerentryを増強します。

The information in this table is learned during session establishment. NOTE: rows in this table will appear during session intialization.

この表の情報は、セッションの設立中に学習されます。注：この表の行は、セッションのintialization中に表示されます。

3.5.6. The LDP Session Statistics Table
3.5.6. LDPセッション統計テーブル

The mplsLdpSessionStatsTable is a read-only table which contains statistical information on sessions. This table AUGMENTS the mplsLdpPeerTable.

mplsldpssessionStatStableは、セッションに関する統計情報を含む読み取り専用テーブルです。このテーブルは、mplsldppeertableを強化します。

3.5.7. The LDP Hello Adjacency Table
3.5.7. LDPハロー隣接テーブル

This is a table of all adjacencies between all LDP Entities and all LDP Peers. A Session may have one or more adjacencies. A session should not have zero adjacencies, because this indicates that the session has lost contact with the Peer. A session which has zero Hello Adjacencies should be removed.

これは、すべてのLDPエンティティとすべてのLDPピア間のすべての隣接の表です。セッションには、1つ以上の隣接がある場合があります。セッションには隣接がゼロであってはなりません。これは、セッションがピアとの接触を失ったことを示しているためです。Helloの隣接がゼロのセッションを削除する必要があります。

3.5.8. The LDP LSP Tables
3.5.8. LDP LSPテーブル

The Label Information Base (LIB) contains information about labels learned by the LSR. The LIB for LDP, CR-LDP and MPLS-RSVP (i.e., all currently defined MPLS protocols) is represented in the LSR MIB [RFC3813]. The LIB is represented by the LSR MIB's mplsXCTable (mpls Cross Connect Table), mplsInSegmentTable (mpls In Segment Table) and the mplsOutSegmentTable (mpls Out Segment Table). The mplsXCTable models the cross-connection of the incoming label with a specific outgoing label. The mplsInSegmentTable stores the incoming label's information, and the mplsOutSegmentTable stores the outgoing label's information.

ラベル情報ベース（LIB）には、LSRが学んだラベルに関する情報が含まれています。LDP、CR-LDP、およびMPLS-RSVPのLIB（つまり、現在定義されているすべてのMPLSプロトコル）は、LSR MIB [RFC3813]で表されています。LIBは、LSR MIBのMPLSXCTABLE（MPLS Cross Connect Table）、MPLSINSEGEMTABLE（セグメントテーブルのMPLS）、およびMPLSOUTSEGEMTABLE（MPLS Out Segment Table）で表されます。MPLSXCTABLEモデル特定の発信ラベルと着信ラベルの相互接続をモデル化します。mplsinsegmentTableは、着信ラベルの情報を保存し、MplsoutSegmentTableは発信ラベルの情報を保存します。

The LDP Session that created the LSP and the LSP's (incoming label, outgoing label) pair along with other information is contained in the MPLS-LSR-STD-MIB module's mplsXCTable, the mplsInSegmentTable and the mplsOutSegmentTable.

LSPとLSP（着信ラベル、発信ラベル）ペアを作成したLDPセッションと他の情報は、MPLS-LSR-STD-MIBモジュールのMPLSINSEGMENTTABLE、MPLSOUTSEGEMTABLEに含まれています。

In order to utilize the MPLS-LSR-STD-MIB module's mplsXCTable, mplsInSegmentTable and mplsOutSegmentTable for LDP LSPs, there needs to be a mechanism to associate LDP sessions with LDP LSPs created as a result of those LDP sessions. The mplsInSegmentLdpLspTable and mplsOutSegmentLdpLspTable in this MIB contain information to find the LDP LSP entries in the mplsInSegmentTable, mplsOutSegmentTable and the mplsXCTable.

LDP LSPのMPLS-LSR-STD-MIBモジュールのMPLSXCTABLE、MPLSINSEGMENTTABLE、MPLSOUTSEGEMTABLEを利用するには、LDPセッションをLDPセッションの結果として作成したLDP LSPと関連付けるメカニズムが必要です。このMIBでmplsinSegmentldplsptableおよびmplsutsegmentldplsptableは、mplsinsegmenttable、mplsoutsegmenttable、およびmplsxctableのLDP LSPエントリを見つけるための情報を含めています。

These two tables, the mplsInSegmentLdpLspTable and mplsOutSegmentLdpLspTable, have been made optional in the conformance section of the MIB. They only need to be supported if the LSR MIBs mplsInSegmentTable, mplsOutSegmentTable and mplsXCTable are implemented.

これらの2つのテーブル、mplsinsegmentldplsptableおよびmplsoutsegmentldplsptableは、MIBの適合セクションでオプションになっています。LSR MIBS MPLSINSEGEMTABLE、MPLSOUTSEGEMTABLE、MPLSXCTABLEが実装されている場合にのみサポートする必要があります。

As discussed in the section, "Changing Values after Session Establishment", if a session is torn down, then all the information related to this session, must be removed from the both LDP MIB and, if implemented, from the LSR MIB.

セクションで説明したように、セッションが取り壊された場合、「セッションの確立後の値を変更する」、その後、このセッションに関連するすべての情報は、両方のLDP MIBから、および実装された場合、LSR MIBから削除する必要があります。

3.5.9. The FEC Tables
3.5.9. FECテーブル

The mplsLdpFecTable is a table which contains FEC (Forwarding Equivalence Class) information. Each entry/row represents a single FEC Element. There is also an LDP LSP FEC Table, mplsLdpLspFecTable, which associates FECs with the LSPs.

mplsldpectableは、FEC（等価クラスの転送）情報を含むテーブルです。各エントリ/行は、単一のFEC要素を表します。また、FECをLSPに関連付けるLDP LSP FECテーブル、MPLSLDPLSPFECTABLEもあります。

3.5.10. The LDP Session Peer Address Table
3.5.10. LDPセッションピアアドレステーブル

The mplsLdpSessionPeerAddrTable is a table which extends the mplsLdpSessionTable. This table is a read-only table which stores Addresses learned after session initialization via Address Message advertisement.

mplsldpssessionpeeraddrtableは、mplsldpssessionTableを拡張するテーブルです。このテーブルは、アドレスメッセージ広告を介してセッションの初期化後に学習したアドレスを保存する読み取り専用テーブルです。

3.6. LDP Notifications
3.6. LDP通知

Currently, there are several notifications which are specific for LDP. These are described in this section. There are no objects which enable or disable notifications from being generated. RFC 3413 [RFC3413] contains MIB modules which can be implemented that will enable or disable these notifications from being generated.

現在、LDPに固有のいくつかの通知があります。これらについては、このセクションで説明します。生成された通知を有効または無効にするオブジェクトはありません。RFC 3413 [RFC3413]には、実装できるMIBモジュールが含まれており、これらの通知が生成されないようになります。

The mplsLdpInitSessionThresholdExceeded notification indicates to the operator that there may be a misconfigured mplsLdpEntityEntry because the session associated with this Entity is not being established, and the Entity keeps trying to establish the session. A side effect of this situation is that a row in the mplsLdpSessionTable may not be reaching the operational state as indicated by the mplsLdpSessionState object. If the value of mplsLdpEntityInitSessionThreshold is 0 (zero) then this is equal to specifying the value of infinity for the threshold, and the mplsLdpInitSessionThresholdExceeded notification will never be sent.

MPLSLDPINITSESTIONSTHRESHOLDEXの通知は、このエンティティに関連するセッションが確立されておらず、エンティティがセッションを確立しようとし続けているため、誤って構成されたMPLSLDPENTITYENTRYがある可能性があることをオペレーターに示しています。この状況の副作用は、MPLSLDPSSESTIONSTATEオブジェクトで示されているように、MPLSLDPSESTIONTABLE内の行が運用状態に到達していない可能性があることです。mplsldldpentityInitsessionThresholdの値が0（ゼロ）の場合、これはしきい値の無限の値を指定することに等しく、mplsldpinitsessionssessionthresholdexの通知は送信されません。

The mplsLdpPathVectorLimitMismatch notification is generated when there is a mismatch in the Path Vector Limits between the Entity and Peer during session initialization. The session uses the value which is configured as the Entity's Path Vector Limit. However, a notification should be generated to indicate that a mismatch occurred. For further details, please see Section 3.5.3 of the LDP Specification [RFC3036].

MPLSLDPPATHVECTORLIMITMITMATCH通知は、セッションの初期化中にエンティティとピアの間にパスベクトル制限に不一致がある場合に生成されます。セッションでは、エンティティのパスベクトル制限として構成されている値を使用します。ただし、ミスマッチが発生したことを示すために、通知を生成する必要があります。詳細については、LDP仕様[RFC3036]のセクション3.5.3を参照してください。

The mplsLdpSessionUp and mplsLdpSessionDown notifications are generated when there is an appropriate change in the mplsLdpSessionState object, e.g., when sessions change state (Up to Down for the mplsLdpSessionDown notification, or Down to Up for the mplsLdpSessionUp notification). There was discussion about combining these two notifications into a single notification, however, some NMS applications can utilize two different notifications, rather than having to parse the varbind list of a single notification. For example, the SessionDown is matched to a SessionUp notification more easily by some NMS applications, than having to parse a Varbind list in a SessionChange type of notification.

MPLSLDPSPSSESTIONDOWN通知は、MPLSLDPSPSSESTIONSTATEオブジェクトに適切な変更がある場合、たとえばセッションが状態を変更する場合（MPLSLDPSSESSIONDOWN通知の場合、またはMPLSLDSPSSESSIONUP通知の上昇まで）に生成されます。これら2つの通知を単一の通知に組み合わせることについての議論がありましたが、一部のNMSアプリケーションは、単一の通知のVarbindリストを解析するのではなく、2つの異なる通知を利用できます。たとえば、セッションダウンは、SessionChangeタイプの通知でVarbindリストを解析するよりも、一部のNMSアプリケーションによってセッションアップ通知に簡単に一致します。

3.7. LDP Notification Frequency
3.7. LDP通知頻度

LDP Notifications are expected to be few in number when LDP is ubiquitously deployed in a relatively stable network. A notification receiver, e.g., an NMS, that receives these notifications should not be overwhelmed by the frequency of LDP notifications. If this assertion proves to be inaccurate, then a throttling object to throttle these notifications may be added to future versions of the MPLS-LDP-STD-MIB.

LDPの通知は、LDPが比較的安定したネットワークに遍在的に展開されている場合、数が少ないと予想されます。これらの通知を受信するNMSなどの通知レシーバーは、LDP通知の頻度に圧倒されるべきではありません。このアサーションが不正確であることが判明した場合、これらの通知をスロットルするためのスロットリングオブジェクトが、MPLS-LDP-STD-MIBの将来のバージョンに追加される場合があります。

4.1. The MPLS-LDP-ATM-STD-MIB Module
4.1. MPLS-LDP-ATM-STD-MIBモジュール

This MIB Module MUST be supported if LDP uses ATM as the Layer 2 medium. There are three tables in this MIB Module. Two tables are for configuring LDP to use ATM. These tables are the mplsLdpEntityAtmTable and the mplsLdpEntityAtmLRTable. The third table is the mplsLdpAtmSessionTable which is a read-only table. This MIB Module IMPORTS the AtmVpIdentifier TEXTUAL-CONVENTION from [RFC2514].

LDPがレイヤー2培地としてATMを使用する場合、このMIBモジュールはサポートする必要があります。このMIBモジュールには3つのテーブルがあります。2つのテーブルは、ATMを使用するようにLDPを構成するためのものです。これらのテーブルは、mplslddpentityatmtableとmplsldldpentityatmlrtableです。3番目のテーブルは、読み取り専用のテーブルであるmplsldpatmsessionTableです。このMIBモジュールは、[RFC2514]からATMVPIDENTIFIERテキストコンベンションをインポートします。

4.1.1. The LDP Entity ATM Table
4.1.1. LDPエンティティATMテーブル

The mplsLdpEntityAtmTable provides a way to configure information which would be contained in the "Optional Parameter" portion of an LDP PDU Initialization Message.

mplsldpentityAtmtableは、LDP PDU初期化メッセージの「オプションパラメーター」部分に含まれる情報を構成する方法を提供します。

4.1.2. The LDP Entity ATM Label Range Table
4.1.2. LDPエンティティATMラベルレンジテーブル

The mplsLdpEntityAtmLRTable provides a way to configure information which would be contained in the "ATM Label Range Components" portion of an LDP PDU Intialization Message, see [RFC3035] and [RFC3036].

mplsldpentityatmlrtableは、LDP PDU intializationメッセージの「ATMラベル範囲コンポーネント」部分に含まれる情報を構成する方法を提供します。[RFC3035]および[RFC3036]を参照してください。

4.1.3. The LDP ATM Session Table
4.1.3. LDP ATMセッションテーブル

The MPLS LDP ATM Session Table is a read-only table which contains session information specific to ATM.

MPLS LDP ATMセッションテーブルは、ATMに固有のセッション情報を含む読み取り専用テーブルです。

o the mplsLdpEntityTable contains objects to provision potential LDP sessions on the Label Switching Router (LSR) or Label Edge Router (LER). The mplsLdpLspFecTable contains objects which associate an LSP with a FEC. Unauthorized access to objects in these tables, could result in disruption of traffic on the network. This is especially true if an LDP session has been established. The use of stronger mechanisms such as SNMPv3 security should be considered where possible. Specifically, SNMPv3 VACM and USM MUST be used with any v3 agent which implements this MIB. Administrators should consider whether read access to these objects should be allowed, since read access may be undesirable under certain circumstances.

o the mplsLdpEntityTable, mplsLdpPeerTable, mplsLdpSessionTable and mplsLdpSessionStatsTable collectively show the LDP LSP network topology. If an Administrator does not want to reveal the LDP LSP topology of the network, then these tables should be considered sensitive/vulnerable.
