Abstract

概要

This document defines a collection of common data types and groupings in YANG data modeling language. These derived common types and groupings are intended to be imported by modules that model Traffic Engineering (TE) configuration and state capabilities.

このドキュメントは、YANGデータモデリング言語の一般的なデータタイプとグループのコレクションを定義しています。これらの派生した共通のタイプとグループは、Traffic Engineering（TE）構成と状態機能をモデル化するモジュールによってインポートされることを目的としています。

1. Introduction
1. はじめに

YANG [RFC6020] [RFC7950] is a data modeling language used to model configuration data, state data, Remote Procedure Calls, and notifications for network management protocols such as the Network Configuration Protocol (NETCONF) [RFC6241]. The YANG language supports a small set of built-in data types and provides mechanisms to derive other types from the built-in types.

YANG [RFC6020] [RFC7950]は、構成データ、状態データ、リモートプロシージャコール、ネットワーク構成プロトコル（NETCONF）[RFC6241]などのネットワーク管理プロトコルの通知をモデル化するために使用されるデータモデリング言語です。 YANG言語は、組み込みデータ型の小さなセットをサポートし、組み込み型から他の型を派生させるメカニズムを提供します。

This document introduces a collection of common data types derived from the built-in YANG data types. The derived types and groupings are designed to be the common types applicable for modeling Traffic Engineering (TE) features in model(s) defined outside of this document.

このドキュメントでは、組み込みのYANGデータ型から派生した一般的なデータ型のコレクションを紹介します。派生タイプとグループ化は、このドキュメントの外部で定義されたモデルのトラフィックエンジニアリング（TE）機能のモデリングに適用できる一般的なタイプになるように設計されています。

3. Overview
3. 概観

This document defines two YANG modules for common TE types: "ietf-te-types" for TE generic types and "ietf-te-packet-types" for packet-specific types. Other technology-specific TE types are outside the scope of this document.

このドキュメントでは、一般的なTEタイプの2つのYANGモジュールを定義します。TEジェネリックタイプの「ietf-te-types」とパケット固有タイプの「ietf-te-packet-types」です。他のテクノロジー固有のTEタイプは、このドキュメントの範囲外です。

3.1. TE Types Module Contents
3.1. TEタイプモジュールの内容

The "ietf-te-types" module (Section 4) contains common TE types that are independent and agnostic of any specific technology or control-plane instance.

「ietf-te-types」モジュール（セクション4）には、特定のテクノロジーやコントロールプレーンインスタンスに依存せずに独立した一般的なTEタイプが含まれています。

The "ietf-te-types" module contains the following YANG reusable types and groupings:

「ietf-te-types」モジュールには、次のYANG再利用可能なタイプとグループが含まれています。

te-bandwidth: A YANG grouping that defines the generic TE bandwidth. The modeling structure allows augmentation for each technology. For unspecified technologies, the string-encoded "te-bandwidth" type is used.

te-bandwidth：一般的なTE帯域幅を定義するYANGグループ。モデリング構造は、各テクノロジーの拡張を可能にします。未指定のテクノロジーでは、文字列エンコードされた「te-bandwidth」タイプが使用されます。

te-label: A YANG grouping that defines the generic TE label. The modeling structure allows augmentation for each technology. For unspecified technologies, "rt-types:generalized-label" is used.

te-label：汎用TEラベルを定義するYANGグループ。モデリング構造は、各テクノロジーの拡張を可能にします。不特定の技術については、「rt-types：generalized-label」が使用されます。

performance-metrics-attributes: A YANG grouping that defines one-way and two-way measured Performance Metrics (PM) and indications of anomalies on link(s) or the path as defined in [RFC7471], [RFC8570], and [RFC7823].

performance-metrics-attributes：[RFC7471]、[RFC8570]、および[RFC7823]で定義された、一方向および双方向の測定されたパフォーマンスメトリック（PM）とリンクまたはパスの異常の兆候を定義するYANGグループ]。

performance-metrics-throttle-container: A YANG grouping that defines configurable thresholds for advertisement suppression and measurement intervals.

performance-metrics-throttle-container：アドバタイズメント抑制と測定間隔の構成可能なしきい値を定義するYANGグループ。

te-ds-class: A type representing the Differentiated Services (DS) Class-Type of traffic as defined in [RFC4124].

te-ds-class：[RFC4124]で定義されている、トラフィックのDifferentiated Services（DS）Class-Typeを表すタイプ。

te-label-direction: An enumerated type for specifying the forward or reverse direction of a label.

te-label-direction：ラベルの順方向または逆方向を指定するための列挙型。

te-hop-type: An enumerated type for specifying that a hop is loose or strict.

te-hop-type：ホップがルーズまたはストリクトであることを指定する列挙型。

te-global-id: A type representing the identifier that uniquely identifies an operator, which can be either a provider or a client. The definition of this type is taken from [RFC6370] and [RFC5003]. This attribute type is used solely to provide a globally unique context for TE topologies.

te-global-id：プロバイダーを一意に識別する識別子を表すタイプ。プロバイダーまたはクライアントのいずれかになります。このタイプの定義は、[RFC6370]および[RFC5003]から取得されます。この属性タイプは、TEトポロジにグローバルに一意のコンテキストを提供するためにのみ使用されます。

te-node-id: A type representing the identifier for a node in a TE topology. The identifier is represented as 4 octets in dotted-quad notation. This attribute MAY be mapped to the Router Address TLV described in Section 2.4.1 of [RFC3630], the TE Router ID described in Section 3 of [RFC6827], the Traffic Engineering Router ID TLV described in Section 4.3 of [RFC5305], or the TE Router ID TLV described in Section 3.2.1 of [RFC6119]. The reachability of such a TE node MAY be achieved by a mechanism such as that described in Section 6.2 of [RFC6827].

te-node-id：TEトポロジ内のノードの識別子を表すタイプ。識別子は、ドット付きクワッド表記で4オクテットとして表されます。この属性は、[RFC3630]のセクション2.4.1で説明されているルーターアドレスTLV、[RFC6827]のセクション3で説明されているTEルーターID、[RFC5305]のセクション4.3で説明されているトラフィックエンジニアリングルーターID TLVにマッピングされる場合があります。 [RFC6119]のセクション3.2.1で説明されているTEルーターID TLVこのようなTEノードの到達可能性は、[RFC6827]のセクション6.2で説明されているようなメカニズムによって実現される場合があります。

te-topology-id: A type representing the identifier for a topology. It is optional to have one or more prefixes at the beginning, separated by colons. The prefixes can be "network-types" as defined in the "ietf-network" module in [RFC8345], to help the user better understand the topology before further inquiry is made.

te-topology-id：トポロジの識別子を表すタイプ。コロンで区切って、先頭に1つ以上の接頭辞を付けることはオプションです。接頭辞は、[RFC8345]の「ietf-network」モジュールで定義されている「network-types」にすることができます。これにより、ユーザーはさらに問い合わせる前にトポロジーをよりよく理解できます。

te-tp-id: A type representing the identifier of a TE interface Link Termination Point (LTP) on a specific TE node where the TE link connects. This attribute is mapped to a local or remote link identifier [RFC3630] [RFC5305].

te-tp-id：TEリンクが接続する特定のTEノード上のTEインターフェイスリンクターミネーションポイント（LTP）の識別子を表すタイプ。この属性は、ローカルまたはリモートのリンク識別子[RFC3630] [RFC5305]にマッピングされます。

te-path-disjointness：[RFC4872]で定義されている、TEトンネルパスのさまざまなリソースの素性オプションを表すタイプ。

admin-groups: A union type for a TE link's classic or extended administrative groups as defined in [RFC3630], [RFC5305], and [RFC7308].

admin-groups：[RFC3630]、[RFC5305]、および[RFC7308]で定義されているTEリンクのクラシックまたは拡張管理グループのユニオンタイプ。

srlg：[RFC4203]および[RFC5307]で定義されている共有リスクリンクグループ（SRLG）を表すタイプ。

te-metric：[RFC3785]で定義されているTEメトリックを表すタイプ。

te-recovery-status: An enumerated type for the different statuses of a recovery action as defined in [RFC4427] and [RFC6378].

     typedef te-bandwidth {
       type string {
         pattern '0[xX](0((\.0?)?[pP](\+)?0?|(\.0?))|'
               + '1(\.([\da-fA-F]{0,5}[02468aAcCeE]?)?)?'
               + '[pP](\+)?(12[0-7]|'
               + '1[01]\d|0?\d?\d)?)|0[xX][\da-fA-F]{1,8}|\d+'
               + '(,(0[xX](0((\.0?)?[pP](\+)?0?|(\.0?))|'
               + '1(\.([\da-fA-F]{0,5}[02468aAcCeE]?)?)?'
               + '[pP](\+)?(12[0-7]|'
               + '1[01]\d|0?\d?\d)?)|0[xX][\da-fA-F]{1,8}|\d+))*';
       }
       description
         "This is the generic bandwidth type.  It is a string containing
          a list of numbers separated by commas, where each of these
          numbers can be non-negative decimal, hex integer, or
          hex float:

(dec | hex | float)[*(','(dec | hex | float))]

（dec | hex | float）[*（ '、'（dec | hex | float））]

For the packet-switching type, the string encoding follows the type 'bandwidth-ieee-float32' as defined in RFC 8294 (e.g., 0x1p10), where the units are in bytes per second.

パケット交換タイプの場合、文字列エンコーディングは、RFC 8294で定義されているタイプ「bandwidth-ieee-float32」に従います（例：0x1p10）。単位はバイト/秒です。 

          For the Optical Transport Network (OTN) switching type,
          a list of integers can be used, such as '0,2,3,1', indicating
          two ODU0s and one ODU3.  ('ODU' stands for 'Optical Data
          Unit'.)  For Dense Wavelength Division Multiplexing (DWDM),
          a list of pairs of slot numbers and widths can be used,
          such as '0,2,3,3', indicating a frequency slot 0 with
          slot width 2 and a frequency slot 3 with slot width 3.
          Canonically, the string is represented as all lowercase and in
          hex, where the prefix '0x' precedes the hex number.";
       reference
         "RFC 8294: Common YANG Data Types for the Routing Area
          ITU-T Recommendation G.709: Interfaces for the
          optical transport network";

7. Security Considerations
7. セキュリティに関する考慮事項

The YANG module specified in this document defines a schema for data that is designed to be accessed via network management protocols such as NETCONF [RFC6241] or RESTCONF [RFC8040]. The lowest NETCONF layer is the secure transport layer, and the mandatory-to-implement secure transport is Secure Shell (SSH) [RFC6242]. The lowest RESTCONF layer is HTTPS, and the mandatory-to-implement secure transport is TLS [RFC8446].

このドキュメントで指定されているYANGモジュールは、NETCONF [RFC6241]やRESTCONF [RFC8040]などのネットワーク管理プロトコルを介してアクセスするように設計されたデータのスキーマを定義します。最下位のNETCONFレイヤーはセキュアなトランスポートレイヤーであり、実装に必須のセキュアなトランスポートはセキュアシェル（SSH）です[RFC6242]。最下位のRESTCONFレイヤーはHTTPSであり、実装に必須のセキュアなトランスポートはTLS [RFC8446]です。

The Network Configuration Access Control Model (NACM) [RFC8341] provides the means to restrict access for particular NETCONF or RESTCONF users to a preconfigured subset of all available NETCONF or RESTCONF protocol operations and content.

The Network Configuration Access Control Model (NACM) [RFC8341] provides the means to restrict access for particular NETCONF or RESTCONF users to a preconfigured subset of all available NETCONF or RESTCONF protocol operations and content.

The YANG module in this document defines common TE type definitions (e.g., typedef, identity, and grouping statements) in YANG data modeling language to be imported and used by other TE modules. When imported and used, the resultant schema will have data nodes that can be writable or readable. Access to such data nodes may be considered sensitive or vulnerable in some network environments. Write operations (e.g., edit-config) to these data nodes without proper protection can have a negative effect on network operations.
