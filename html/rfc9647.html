1. Introduction
1. はじめに

This document defines a data model for the Babel routing protocol [RFC8966]. The data model is defined using YANG 1.1 [RFC7950] and is compatible with Network Management Datastore Architecture (NMDA) [RFC8342]. It is based on the Babel information model [RFC9046]. The data model only includes data nodes that are useful for managing Babel over IPv6.

このドキュメントは、BABELルーティングプロトコル[RFC8966]のデータモデルを定義しています。データモデルは、Yang 1.1 [RFC7950]を使用して定義され、ネットワーク管理データストアアーキテクチャ（NMDA）[RFC8342]と互換性があります。これは、BABEL情報モデル[RFC9046]に基づいています。データモデルには、IPv6を介してBabelを管理するのに役立つデータノードのみが含まれています。

2. Babel Module
2. バベルモジュール

This document defines a YANG 1.1 [RFC7950] data model for the configuration and management of Babel. The YANG module is based on the Babel information model [RFC9046].

このドキュメントでは、BABELの構成と管理のためのYang 1.1 [RFC7950]データモデルを定義しています。Yangモジュールは、BABEL情報モデル[RFC9046]に基づいています。

2.1. Information Model
2.1. 情報モデル

It's worth noting a few differences between the Babel information model and this data module. The information model mandates the definition of some of the attributes, e.g., "babel-implementation-version" or the "babel-self-router-id". These attributes are marked as read-only objects in the information module as well as in this data module. However, there is no way in the data module to mandate that a read-only attribute be present. It is up to the implementation of this data module to make sure that the attributes that are marked "read only" and are mandatory are indeed present.

Babel情報モデルとこのデータモジュールの間にいくつかの違いがあることに注意する価値があります。情報モデルは、属性のいくつかの定義、例えば「Babel-Implementation-version」または「babel-self-router-id」を義務付けています。これらの属性は、情報モジュールおよびこのデータモジュールの読み取り専用オブジェクトとしてマークされています。ただし、データモジュールには、読み取り専用属性が存在することを義務付ける方法はありません。「読み取り専用」と必須である属性が実際に存在することを確認するのは、このデータモジュールの実装次第です。

The "interfaces" subtree describes attributes such as the "interface" object that is being referenced; the type of link, e.g., wired, wireless, or tunnel, as enumerated by "metric-algorithm" and "split-horizon"; and whether the interface is enabled or not.

「インターフェイス」サブツリーは、参照されている「インターフェイス」オブジェクトなどの属性を記述します。「メトリックアルゴリズム」および「スプリットホリゾン」によって列挙されているリンクのタイプ、有線、ワイヤレス、またはトンネル。インターフェイスが有効になっているかどうか。

The "constants" subtree describes the UDP port used for sending and receiving Babel messages and the multicast group used to send and receive announcements on IPv6.

「定数」サブツリーは、Babelメッセージの送信と受信に使用されるUDPポートと、IPv6で発表を送信および受信するために使用されるマルチキャストグループについて説明しています。

The "routes" subtree describes objects such as the prefix for which the route is advertised, a reference to the neighboring route, and the "next-hop" address.

「ルート」サブツリーは、ルートが宣伝されているプレフィックス、隣接するルートへの参照、「ネクストホップ」アドレスなどのオブジェクトを説明します。

Finally, for security, two subtrees are defined to contain Message Authentication Code (MAC) keys and DTLS certificates. The "mac-key-set" subtree contains keys used with the MAC security mechanism. The boolean flag "default-apply" indicates whether the set of MAC keys is automatically applied to new interfaces. The "dtls" subtree contains certificates used with the DTLS security mechanism. Similar to the MAC mechanism, the boolean flag "default-apply" indicates whether the set of DTLS certificates is automatically applied to new interfaces.

最後に、セキュリティの場合、メッセージ認証コード（MAC）キーとDTLS証明書を含むように2つのサブツリーが定義されています。「Mac-Key-Set」サブツリーには、Macセキュリティメカニズムとともに使用されるキーが含まれています。ブールフラグ「デフォルトアプリ」は、Macキーのセットが新しいインターフェイスに自動的に適用されるかどうかを示します。「DTLS」サブツリーには、DTLSセキュリティメカニズムで使用される証明書が含まれています。MACメカニズムと同様に、ブールフラグ「デフォルトアプリ」は、DTLS証明書のセットが新しいインターフェイスに自動的に適用されるかどうかを示します。

2.3. YANG Module
2.3. ヤンモジュール

This YANG module augments the YANG routing management module [RFC8349] to provide a common framework for all routing subsystems. By augmenting the module, it provides a common building block for routes and Routing Information Bases (RIBs). It also has a reference to an interface defined by "A YANG Data Model for Interface Management" [RFC8343].

このYangモジュールは、Yangルーティング管理モジュール[RFC8349]を拡張して、すべてのルーティングサブシステムに共通のフレームワークを提供します。モジュールを拡張することにより、ルートとルーティング情報ベース（リブ）に共通のビルディングブロックを提供します。また、「インターフェイス管理のためのヤンデータモデル」[RFC8343]で定義されたインターフェイスへの参照もあります。

A router running the Babel routing protocol can sometimes determine the parameters it needs to use for an interface based on the interface name. For example, it can detect that eth0 is a wired interface and that wlan0 is a wireless interface. This is not true for a tunnel interface, where the link parameters need to be configured explicitly.

バベルルーティングプロトコルを実行するルーターは、インターフェイス名に基づいてインターフェイスに使用する必要があるパラメーターを決定することがあります。たとえば、ETH0は有線インターフェイスであり、WLAN0がワイヤレスインターフェイスであることを検出できます。これは、リンクパラメーターを明示的に構成する必要があるトンネルインターフェイスには当てはまりません。

For a wired interface, it will assume "two-out-of-three" is set for "metric-algorithm" and "split-horizon" is set to true. On the other hand, for a wireless interface, it will assume "etx" is set for "metric-algorithm" and "split-horizon" is set to false. However, if the wired link is connected to a wireless radio, the values can be overridden by setting "metric-algorithm" to "etx" and "split-horizon" to false. Similarly, an interface that is a metered 3G link and is used for fallback connectivity needs much higher default time constants, e.g., "mcast-hello-interval" and "update-interval", in order to avoid carrying control traffic as much as possible.

有線インターフェイスの場合、「メトリックアルゴリズム」と「スプリットホリゾン」がTRUEに設定されている「2回の3」が設定されていると仮定します。一方、ワイヤレスインターフェイスの場合、「Metric-Algorithm」に「ETX」が設定され、「Split-Horizon」がfalsに設定されていると仮定します。ただし、有線リンクがワイヤレス無線に接続されている場合、「メトリックアルゴリズム」を「ETX」に、「スプリットホリゾン」にfalseに設定することにより、値をオーバーライドできます。同様に、メーター付きの3Gリンクであり、フォールバック接続に使用されるインターフェイスは、制御トラフィックを可能な限り運ぶのを避けるために、「Mcast-Hello-interval」や「Update-Interval」など、はるかに高いデフォルトの時間定数を必要とします。。

This model defines two actions. The "test" action has been designed to not reveal such information directly.
